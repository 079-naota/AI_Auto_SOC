# 4月20日までの作業報告

## 成果

### 1.サーバー環境の変更とCowrieの導入
本番の環境を保護しつつ、攻撃者を安全に観察するためのサンドボックスを構築した。

&emsp;まず、VPS本来の管理用SSHポート（２２番）を別のポートへ退避させた。これにより、デフォルトの２２番ポートを狙う無差別攻撃から本番環境を保護する体制を整えた。


&emsp;次に、開けた２２番ポートに対して、対話型のハニーポットであるCowrieを導入した。
攻撃者からは通常のLinuxサーバーに見えるが、実際には入力されたコマンドやダウンロードされたファイルが全て記録される偽の環境である。


&emsp;最後に、手元の端末からハニーポットへ意図的にSSHへ接続し、`ls -al`や`pwd`などのコマンドを実行した。
Cowrieが指定のディレクトリに、攻撃者の行動をJSON形式で正確に出力していることを確認した。

### 2.WazuhManagerへのログ取り込み
Cowrieが生成した生のJSONログを、Wazuhの分析エンジンに通す設定を行った。
&emsp;まず、対象のVPSがWazuh`Agent`ではなく`Manager`として動作していることをコマンドの実行結果から特定した。


&emsp;次にManager本体の設定ファイル`ossec.conf`を編集した。
設定ファイルにはWazuhに対して「CowrieのJsonログをリアルタイム読み込みせよ」という指示を実装した。


### 3.Wazuh-logtestを用いたテスト
Wazuhがログを読み込んでいるにもかかわらずダッシュボードにアラートが表示されない問題に対しアナログな手法で原因を調べた。
&emsp;Wazuhの内部解析ツールである。`sudo /var/ossec/bin/wazuh-logtest`を起動し、Cowrieの実際のログ{"eventid":"cowrie.command.input",...}を直接入力し挙動テストを行った。


ログテストの出力結果（Phase1:読み込み、Phase2:デコード、Phase3:アラート判定）を分析した結果、以下２つの致命的な問題を特定した。
１つ目は、ルールの結合について`Rule ID '100001' is duplicated.`の警告から、デフォルトのルールIDと自作ルールが重複し、Wazuhが自作ルールを無視していることを発見した。

2つ目は、デコーダーの指定漏れについてJson形式のログとして解釈させるための`<decoed_as>json</decoded_as>タグが欠落しており、Wazuhがログの中身を正しく抽出できていなかった。

### 4.カスタム検知ルールの設定とダッシュボードでの可視化
特定した問題を修正し、攻撃ログを「レベル５の脅威」としてSIEM上で可視化する独自のルールエンジンを完成させた。
&emsp;まず、`local_rules.xml`を編集し、不要な重複ルールを削除し、新たに`Rule ID:100002`として、JSON解読機能をも耐えたCowrie専用の検知アルゴリズムを実装した。

&emsp;次に、`sudo /.../wazuh-control restart`にてWazuhManagerを再起動し、再度`wazuh-logtest`を実行した。
その後、`Alert to be generated`のステータスを確認した。

&emsp;最後に、実環境でハニーポットへ攻撃を行い、Wazuhのイベントログ上に`Cowrie Hneypot Event:CMD:ls -al`といったアラートが、レベル５以上の脅威としてリアルタイムに可視化されることを確認した。
