# 3月25日 2:13までの作業報告

### 成果 
　今回の作業では攻撃の実行、analyze_alerts.pyの動作検証そしてWazuhのActiveResponse機能の設定と実行を行った。


攻撃の実行ではKaliにおいてhydraを用いてパスワードリストを使った辞書攻撃を行った。  
初めの出力された報告書では、被害者サーバーの再起動時における更新のみを読み取り正確に攻撃を検知しなかった。  
報告書については20260325_003640.mdを参照するとわかる。


これはOS再起動による大量のログとKaliからの攻撃のログが混じり最初の１件のみを取得するという関数が偶然ノイズを拾ってしまった。こういうノイズと攻撃を正しく判別しないことを「アラート疲れ」というらしい。


そこで、まずanalyze_alerts.pyのfetch_latest_alert間数内にKaliのIPを直接書き込むことでWazuhが正常に動いていることを確認した。これを「スレッドハンティング」というらしい。  
　次に、ActiveResponse機能を追加した。方法としてはossec.conf内に以下の内容を追記した。
```XML:ossec.conf
<active-response>
#Wazuhに搭載されているスクリプトでLinuxのiptablesやufwを自動操作し通信を破棄する
<command>firewall-drop</command>
<location>local</location>
#SSH認証失敗(5760),SSHブルートフォース(5712,5720)のルールを指定
<rules_id>5760,5712,5720</rules_id>
#誤検知による永久締め出しを防ぐために60秒で自動解除する（実務では10分以上らしい）
<timeout>60</timeout>
</active-response>
```
  
とくに、今回の作業ではこの設定の書き込みにおいてエラーをたくさん吐いて対処に時間がかかった。
  最後に締め出しが正常に動作しているのかをkaliでip コマンドを用いて確認した。
