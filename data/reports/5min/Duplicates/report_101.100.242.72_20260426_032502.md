# SOC自動分析レポート: 101.100.242.72

**生成日時:** 2026-04-26 03:26:40

## SOCアナリスト レポート：SSH ブルートフォース攻撃と永続化の試み

### 1. 攻撃の概要と目的

攻撃元IP `101.100.242.72` から、SSHサービスに対する継続的なブルートフォース攻撃が確認されました。攻撃者は、まず一般的なユーザー名とパスワードの組み合わせを複数回試行しました。その後、複数回にわたり `root` ユーザーとしてログインに成功しています。

ログイン成功後、攻撃者は `~/.ssh/authorized_keys` ファイルに自身の公開鍵を書き込むことで、パスワードなしでシステムに再接続できる永続的なバックドアを確立しようと試みました。この目的は、検出を回避しつつ、将来的にシステムへのアクセスを維持し、さらなる偵察活動、データ窃取、マルウェアの展開、または他のシステムへの横展開を行うための足がかりを築くことであると推測されます。

### 2. 推測される手法・使用ツール

*   **手法**:
    *   **SSHブルートフォース/パスワードスプレー攻撃**:
        *   `frappe`, `deployer`, `ubuntu`, `admin`, `tester`, `test`, `oracle`, `user`, `zhaoxj` といった一般的な、またはアプリケーション固有のユーザー名を標的とし、それぞれに複数のパスワードを組み合わせて総当たり攻撃を試みています。
        *   `root` ユーザーに対しては、`qwer1234QWER!@#$`, `Ni123456`, `ubuntu@123`, `Password.123`, `a12348765` など、比較的複雑なパスワードを含む複数の認証情報でのログインに成功しています。これは、辞書攻撃やパスワードスプレーリストに強力なパスワードが含まれていることを示唆しています。
    *   **永続化**:
        *   ログイン成功後、`cd ~ && rm -rf .ssh && mkdir .ssh && echo "ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr">>.ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~` というコマンドを実行し、`~/.ssh/authorized_keys` ファイルに攻撃者のSSH公開鍵を書き込むことで、パスワードなしでのアクセスを可能にするバックドアを設置しようとしています。鍵のコメントは `mdrfckr` となっており、攻撃者の識別子と考えられます。

### 3. 脅威レベルとその理由

**脅威レベル: 非常に高い (Critical)**

**理由**:
*   **root権限でのログイン成功**: 攻撃者は複数回にわたり、システム上で最も高い権限を持つ `root` ユーザーとしてログインに成功しています。これにより、攻撃者はシステムの完全な制御権を獲得し、任意の操作（データの窃取、マルウェアのインストール、システムの改ざんなど）を実行できる状態になります。
*   **永続的なアクセス経路の確立試行**: `authorized_keys` への攻撃者の公開鍵の追加は、恒久的なバックドアの設置を目的としています。これが成功した場合、攻撃者はシステムへのアクセスを維持し、パスワード変更などの対策後もシステムに侵入し続けることが可能となります。
*   **複数の認証情報の漏洩可能性**: ログに記録された `root` ユーザーの複数のパスワードが実環境で使用されている場合、それらのパスワードが他のシステムやサービスでも再利用されている可能性があり、組織全体のセキュリティリスクを増大させます。
*   **組織的・自動化された攻撃の可能性**: 攻撃手法が一貫しており、複数のログイン試行が短時間で行われていることから、これは単独の偶然の試行ではなく、広範な自動化された攻撃キャンペーンの一部であると推測されます。
*   **さらなる攻撃への準備**: `cowrie.session.file_download` のイベントは詳細が不明ですが、これまでの行動と合わせて考えると、さらなるペイロード（マルウェアなど）のダウンロードやC2（コマンド＆コントロール）通信の準備である可能性があり、被害拡大の懸念があります。

### 4. 推奨アクション

この攻撃はハニーポットで捕捉されたものですが、もし実システムで発生した場合を想定し、以下の緊急対応と予防的対策を直ちに実施すべきです。

#### 4.1. 緊急対応 (Immediate Actions)

1.  **攻撃元IPのブロック**: 攻撃元IPアドレス `101.100.242.72` を、境界ファイアウォール、IPS/IDS、またはホストベースファイアウォールで即座にブロックし、今後のアクセスを遮断します。
2.  **既存の認証情報の変更**:
    *   ログに記録された `root` ユーザーで使用された全てのパスワード (`qwer1234QWER!@#$`, `Ni123456`, `ubuntu@123`, `Password.123`, `a12348765` など) を直ちに、より強力で一意なものに変更します。
    *   ログイン試行に使用された他のユーザーアカウント (`frappe`, `deployer`, `ubuntu`, `admin` など) のパスワードもレビューし、もし存在する場合は変更を検討します。
    *   これらのパスワードが他のシステムやサービスで使い回されている場合は、それらも同様に変更します。
3.  **`authorized_keys` の確認とクリーンアップ**:
    *   全てのユーザー（特に `root`）の `~/.ssh/authorized_keys` ファイルを徹底的にレビューし、攻撃者によって追加された可能性のある不正な公開鍵（特にコメントが `mdrfckr` のもの）があれば、直ちに削除します。
    *   正規の公開鍵以外のエントリが存在しないことを確認します。
4.  **システム侵入有無の確認**:
    *   ログからコマンド実行が確認されているため、実際にシステムが侵害されたものと仮定し、以下の点を詳細に調査します。
        *   システムログ（`auth.log`, `syslog` など）に不正なログインや不審な活動の痕跡がないか。
        *   ファイルシステムの変更履歴（特に `/tmp`, `/var/tmp`, `/usr/local/bin` など）にマルウェアや不審なファイルが作成されていないか。
        *   不審なプロセスやネットワーク接続（C2通信など）がないか。
        *   他のユーザーアカウントが作成されていないか。
    *   可能であれば、フォレンジック調査のためにシステムのディスクイメージを取得します。

#### 4.2. 予防的対策 (Preventive Measures)

1.  **SSHサービス設定の強化**:
    *   **`PermitRootLogin no`**: `root` ユーザーのSSH直接ログインを禁止し、一般ユーザーでログイン後、`sudo` を利用して昇格する運用に切り替えます。
    *   **`PasswordAuthentication no`**: パスワード認証を完全に無効化し、公開鍵認証のみを許可します。
    *   **強力なユーザー名とパスワードポリシーの強制**: 全てのシステムユーザーに対して、複雑で推測されにくいパスワード（最低12文字以上、英数字記号混合）の使用を義務付け、パスワードスプレー攻撃に対する耐性を高めます。
    *   SSHのデフォルトポート22番以外のポートを使用することを検討します（ただし、これは検出回避のための手段であり、根本的なセキュリティ対策ではありません）。
2.  **多要素認証 (MFA) の導入**: SSHログインに多要素認証（TOTPなど）を導入し、認証の強度を飛躍的に高めます。
3.  **WAF/IPS/IDSの導入と強化**: ブルートフォース攻撃や不正なアクセスパターンを自動的に検知・ブロックできるセキュリティソリューションを導入または設定強化します。
4.  **ログ監視とアラートの強化**:
    *   SSHログイン失敗の試行回数や頻度に対する閾値設定を強化し、異常なログイン試行を検知した場合に即座にアラートが発報されるよう、SIEM (Security Information and Event Management) システムなどを設定します。
    *   `authorized_keys` ファイルの変更を監視し、不正な変更があればアラートを生成します。
5.  **不必要なサービスの停止/アクセス制限**: SSHサービスが外部に公開されている必要がない場合は、外部からのアクセスを制限するか、サービス自体を停止します。特定の信頼できるIPアドレスからのアクセスのみを許可する設定（ACL）を導入します。
6.  **定期的な脆弱性診断とパッチ適用**: システムおよびアプリケーションの脆弱性を定期的にスキャンし、最新のセキュリティパッチを適用して、既知の脆弱性を悪用されるリスクを低減します。
