#### 1. 攻撃の概要と目的

攻撃者は、SSHハニーポット（Cowrie）に対し、複数の異なるユーザー名とパスワードを用いた総当たり攻撃を試みました。この攻撃は、以下の段階で構成されています。

1.  **偵察と認証情報の推測:** 攻撃者は `user7`, `deployer`, `frappe`, `ubuntu`, `user`, `mary`, `admin`, `zhaoxj` といった一般的なユーザー名を含む複数の認証情報を試行しました。
2.  **rootアカウントでの不正ログイン成功:** 攻撃者は最終的に `root` ユーザーに対して、複数の異なるパスワード（例: `abcd1234.`, `3245gs5662d34`, `Password.123`, `MoeClub.org`, `a12348765`, `abc@123456`）でログインに成功しています。
3.  **永続化の試み:** ログイン成功後、攻撃者は自身のSSH公開鍵をシステムに登録し、恒久的なアクセスポイントを確立しようとしました。具体的には、既存の `.ssh` ディレクトリを削除し、新しい `.ssh/authorized_keys` ファイルに `mdrfckr` というコメントの公開鍵を追加し、適切なパーミッションを設定するコマンドを実行しています。
4.  **防御回避の試み:** 攻撃者は `chattr -ia .ssh` および `lockr -ia .ssh` というコマンドを試行しており、これはおそらく `.ssh` ディレクトリの属性を変更して、削除や変更から保護しようとする防御メカニズムを解除する意図があったと考えられます。

これらの行動から、攻撃の主な目的は、root権限でのシステムアクセスを獲得し、SSH公開鍵を通じて永続的なバックドアを設置することであると推測されます。

#### 2. 推測される手法・使用ツール

*   **手法:**
    *   **SSH ブルートフォース攻撃 / パスワードリスト攻撃:** 攻撃ログに見られる短期間での多様なユーザー名とパスワードの試行、および複数回にわたるログイン成功は、自動化されたSSHブルートフォースツールやパスワードリストを用いた攻撃であることを強く示唆しています。
    *   **永続化 (Persistence):** `rm -rf .ssh && mkdir .ssh && echo "..." >>.ssh/authorized_keys` コマンドは、システムへの将来的なアクセスを確保するための典型的な永続化手法です。公開鍵認証を使用することで、パスワードなしで再ログインが可能になります。
    *   **防御回避 (Defense Evasion):** `chattr -ia .ssh` の使用は、システムの監査ログやファイル属性設定による防御メカニズムを回避しようとする試みです。`lockr` は標準的なLinuxコマンドではないため、攻撃者が使用しているカスタムツールやスクリプトの一部である可能性があります。

*   **使用ツール:**
    *   自動化されたSSHクライアント（例: Pythonのparamiko、Go言語のsshライブラリを使用するカスタムスクリプトなど）。
    *   SSHブルートフォースツール（例: Hydra, NmapのNSEスクリプト、Medusaなど）。
    *   特定のマルウェアまたはボットネットに組み込まれた自動化スクリプト。

#### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**

1.  **root権限でのログイン成功:** 攻撃者はシステム上で最も高い権限を持つ `root` アカウントの認証情報（パスワード）を複数回取得し、ログインに成功しています。現実のシステムであれば、これにより攻撃者はシステムを完全に制御下に置くことが可能になります。
2.  **永続化メカニズムの積極的な確立試行:** 攻撃者が自身のSSH公開鍵を `authorized_keys` に追加しようと試みていることは、単なる一時的な侵入ではなく、将来にわたるシステムへの恒久的なアクセス経路を確保する意図が明確であることを示しています。
3.  **自動化された攻撃:** ログのパターンから、手動ではなく自動化されたスクリプトやツールによる攻撃である可能性が高いです。これは、攻撃者が広範囲なスキャンと攻撃の一部としてこのシステムを標的としていることを示唆し、他のシステムも同様のリスクに晒されている可能性があります。
4.  **防御回避の試み:** `chattr` や `lockr` コマンドの使用は、攻撃者がシステムを乗っ取った後の行動を計画し、自身の痕跡を隠蔽したり、バックドアを保護したりする意図があることを示しています。
5.  **複数の異なるパスワードでの成功:** 複数の異なるパスワードで `root` ログインに成功しているという事実は、攻撃者が強力なパスワードリストを持っているか、この環境でよく使われるパスワードの推測に成功していることを意味し、他のシステムでの認証情報管理にも脆弱性が存在する可能性を示唆します。

#### 4. 推奨アクション

**A. 緊急対応 (Immediate Actions):**

1.  **攻撃元IPのブロック:** 攻撃元IPアドレス `103.172.205.139` を直ちにファイアウォール、IDS/IPS、またはACLでブロックし、今後のアクセスを完全に遮断してください。
2.  **SSH認証情報の変更と無効化:**
    *   ログに記録された `root` ユーザーのすべてのパスワード（`abcd1234.`, `3245gs5662d34`, `Password.123`, `MoeClub.org`, `a12348765`, `abc@123456`）を直ちに無効化し、非常に強力で複雑な新しいパスワードに変更してください。
    *   `deployer`, `frappe`, `ubuntu`, `user`, `mary`, `admin`, `zhaoxj` など、攻撃者が試行した他のユーザーアカウントのパスワードも確認し、必要に応じて変更または無効化してください。
3.  **Authorized_keysファイルのレビューと削除:**
    *   すべてのユーザー（特に `root` および特権ユーザー）の `.ssh/authorized_keys` ファイルを緊急でレビューし、攻撃者によって追加された可能性のある公開鍵（特に `ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK_rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`）を削除してください。
    *   システム全体で不審なSSH鍵ファイル（`id_rsa`, `id_dsa`, `id_ecdsa`, `id_ed25519` など）がないかスキャンしてください。
4.  **システム侵害調査:**
    *   対象システム（ハニーポットではない本番環境を想定した場合）に対して、完全なフォレンジック調査を実施し、他のバックドア、不正なプロセス、設定変更、マルウェアの存在がないか徹底的に確認してください。
    *   他の認証ログ、システムログ、ネットワークログ（ファイアウォール、プロキシなど）を確認し、この攻撃元IPからの他の活動や、SSH以外のポートへの攻撃がないか調査してください。

**B. 長期的な対策 (Long-term Measures):**

1.  **SSHセキュリティ強化:**
    *   **パスワード認証の無効化:** 鍵認証のみを許可し、パスワード認証は完全に無効化することを強く推奨します。
    *   **rootログインの禁止:** `PermitRootLogin no` を設定し、`root` ユーザーの直接ログインを禁止し、一般ユーザーでログイン後に `sudo` を利用する運用に切り替えてください。
    *   **強度の高いパスワードポリシー:** すべてのシステムユーザーに対して、文字数、複雑性、有効期間などを含む厳格なパスワードポリシーを強制し、定期的なパスワード変更を義務付けてください。
    *   **SSHポートの変更:** 標準の22番ポートから別のポートに変更することを検討してください。これにより、一般的なポートスキャンからの可視性を低減できます。
    *   **多要素認証 (MFA) の導入:** 可能であれば、SSHログインにMFAを導入し、セキュリティを強化してください。
    *   **Fail2ban/CrowdSecなどの導入:** ブルートフォース攻撃を自動的に検知・ブロックするツール（例: Fail2ban, CrowdSec）を導入し、不正なログイン試行を繰り返すIPアドレスを自動的にブロックする仕組みを構築してください。
2.  **IDS/IPSの導入・強化:** 異常なログイン試行、不正なコマンド実行、または他の疑わしいネットワークトラフィックを検知・ブロックできるIDS/IPSソリューションを導入または設定を強化してください。
3.  **定期的なセキュリティ監査と脆弱性管理:** 定期的なセキュリティ監査（ペネトレーションテスト、脆弱性スキャン）を実施し、システムの脆弱性を特定・修正するプロセスを確立してください。
4.  **ユーザー教育:** 強力なパスワードの重要性、不審な活動の報告方法などについて、システム利用者への定期的なセキュリティ教育を実施してください。
5.  **ログ監視の強化:** ハニーポットを含むすべてのセキュリティログを一元的に収集・監視するシステム（SIEMなど）を導入し、異常な活動や攻撃パターンをリアルタイムで検知できる体制を強化してください。
