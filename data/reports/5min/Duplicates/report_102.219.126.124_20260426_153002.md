### 1. 攻撃の概要と目的

攻撃元IPアドレス 102.219.126.124 から、SSHサービスに対して継続的な不正アクセス試行が検出されました。この攻撃は、まず一般的なユーザー名とパスワードの組み合わせによるログイン試行（ブルートフォース/パスワードリスト攻撃）を行い、その後、`root`ユーザーとしてログインに成功したと見せかけるイベントを複数回発生させています。

ログイン成功と見せかけた後、攻撃者は以下の目的で一連のコマンド実行を試みています。

1.  **永続的なアクセス経路の確立（バックドアの設置）:** `root`ユーザーのホームディレクトリにある`.ssh/authorized_keys`ファイルに自身のSSH公開鍵を追加することで、以降はパスワード認証なしでシステムにアクセスできるよう、バックドアを設置しようとしています。
2.  **防御機構の回避:** `.ssh`ディレクトリのファイル属性を変更する`chattr`コマンド（および不明な`lockr`コマンド）を試行し、正規のユーザーやシステム管理者によるファイル変更防止策を無効化しようとしています。

これらの行動から、攻撃者はシステムの完全な制御権を奪取し、将来にわたって継続的にアクセスできる状態を確立することを企図していると判断されます。

### 2. 推測される手法・使用ツール

*   **手法:**
    *   **SSH ブルートフォース/パスワードリスト攻撃:** ログには `drcom:drcom123` や `ubuntu:Aa12345!`, `user:user!2025` といった様々なユーザー名・パスワードの組み合わせでのログイン失敗が記録されており、さらに `root` ユーザーに対しては、`1qaz@WSX3edc$RFV!@`, `ali123456`, `Root8`, `Aa112211.`, `Config123`, `Aa123321` など、複数のパスワードを繰り返し試行し、ログイン成功（ハニーポット上）に至っています。これは典型的なブルートフォース攻撃またはパスワードリスト攻撃のパターンです。
    *   **永続化 (Persistence):** ログイン成功後、`cd ~ && rm -rf .ssh && mkdir .ssh && echo "ssh-rsa AAAAB3NzaC1y... mdrfckr" >> .ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~` という一連のコマンドを実行し、自身のSSH公開鍵を`root`ユーザーの`authorized_keys`に追加しています。これにより、パスワードが変更されても攻撃者がアクセスできる状態を作り出そうとしています。
    *   **防御回避 (Defense Evasion):** `chattr -ia .ssh` および `lockr -ia .ssh` のコマンド実行は、`.ssh`ディレクトリに対する不変属性や追記のみ属性（immutable/append-only attribute）といった保護を解除し、`authorized_keys`ファイルの改ざんを可能にしようとする意図が見受けられます。`lockr`は標準的なLinuxコマンドではないため、誤入力か、特定のマルウェアが導入するコマンドを想定している可能性があります。
*   **使用ツール:**
    *   **自動化されたSSHクライアントまたはスクリプト:** 複数の異なるクレデンシャルでのログイン試行と、ログイン成功後の定型的なコマンド実行パターンから、攻撃は自動化されたスクリプト（例: Python, Bashスクリプト）または専用のSSHブルートフォースツール（例: `Hydra`, `Medusa`など）によって行われている可能性が極めて高いです。
    *   **標準的なLinuxコマンド:** `cd`, `rm`, `mkdir`, `echo`, `chmod`など、システムに標準で備わっているコマンドを利用して、バックドアの設置を試みています。

### 3. 脅威レベルとその理由

*   **脅威レベル: 高 (High)**

*   **理由:**
    1.  **特権アカウント（root）の標的と成功:** 攻撃者はシステムの最高権限である`root`アカウントの奪取を試み、実際に複数のパスワードで「ログイン成功」イベントを発生させています。もしこれが実際のシステムであった場合、システム全体が完全に侵害される致命的な事態に繋がります。
    2.  **永続的なアクセス確立の試み:** `authorized_keys`への公開鍵追加は、攻撃者が一度侵入に成功すれば、パスワード変更などの防御策をすり抜けて、永続的にシステムへアクセスできるバックドアを設置する目的があります。これは、一般的なマルウェア感染と同等の深刻な脅威です。
    3.  **自動化された広範な攻撃の可能性:** 攻撃が自動化されていることから、特定のターゲットに限定されず、インターネット上の多数のSSH公開システムに対して広範囲に仕掛けられている可能性があり、同様の脆弱性を持つ他のシステムも標的となっている恐れがあります。
    4.  **積極的な防御回避の試み:** `chattr`コマンドによるファイル属性変更の試みは、攻撃者がシステム管理者の防御策を認識し、それを回避しようとする積極的な姿勢を示しています。

### 4. 推奨アクション

1.  **攻撃元IPアドレス (102.219.126.124) のブロック:**
    *   即座に組織のファイアウォール、IDS/IPS、およびSSHゲートウェイで、このIPアドレスからのすべての接続をブロックリストに追加してください。
2.  **SSH認証情報の厳重なレビューと強化:**
    *   **パスワードポリシーの強化:** 強固なパスワード（長さ、複雑性、履歴制限）を必須とし、定期的なパスワード変更を義務付けてください。ログに記録されたパスワード（例: `1qaz@WSX3edc$RFV!@`, `ali123456`, `Root8`など）が、他のシステムやアカウントで使用されていないか緊急に確認し、もし利用されている場合は直ちにパスワードを変更してください。
    *   **`root`ログインの制限:** `/etc/ssh/sshd_config`ファイルにて `PermitRootLogin no` を設定し、`root`ユーザーの直接ログインを禁止してください。特権操作が必要な場合は、一般ユーザーでログイン後、`sudo`コマンドを利用する運用を徹底してください。
    *   **SSH公開鍵認証の推奨とパスワード認証の無効化:** 可能であれば、パスワード認証を無効にし、よりセキュアなSSH公開鍵認証のみを許可する設定に変更してください。
    *   **二要素認証 (MFA) の導入:** SSH接続にMFAを導入し、認証セキュリティを強化することを強く推奨します。
3.  **システム内のSSH `authorized_keys`ファイルの監査:**
    *   組織内の全Linux/Unixサーバーにおいて、`~/.ssh/authorized_keys`ファイル（特に`root`ユーザーのもの）を監査し、ログに示された不審な公開鍵（コメントが`mdrfckr`の鍵）や、その他の未知の公開鍵が存在しないか確認してください。存在する場合は直ちに削除し、その鍵がいつ追加されたか、どのような経路で追加されたかを調査してください。
4.  **システムログの継続的な監視と相関分析:**
    *   SSH認証ログ（`/var/log/auth.log`、`secure`など）を継続的に監視し、ブルートフォース攻撃や不審なログイン試行がないか確認してください。特に、今回の攻撃で試されたユーザー名やパスワードの組み合わせに一致する試行がないか注意深く分析してください。
    *   SIEM（Security Information and Event Management）などを利用して、SSHログインイベントと他のシステムイベントを相関させ、異常な行動パターンを早期に検出できる体制を強化してください。
5.  **脅威インテリジェンスの活用と共有:**
    *   攻撃元IPアドレス (102.219.126.124) および公開鍵情報（特に`mdrfckr`というコメント）を脅威インテリジェンスプラットフォーム（VirusTotal, Shodan, AbuseIPDBなど）で検索し、既知の攻撃キャンペーンやアクターに関連するかどうかを調査してください。
    *   これらの情報を関連組織や情報共有コミュニティと共有し、他のシステムへの攻撃を防ぐための協力を促進してください。
6.  **脆弱性スキャンとセキュリティパッチの適用:**
    *   SSHサービスを含む、公開されているサービスに対して定期的な脆弱性スキャンを実施し、既知の脆弱性がないか確認してください。
    *   すべてのシステムでOSおよびアプリケーションのセキュリティパッチを最新の状態に維持してください。

この攻撃はハニーポットで検出されたものであり、実際のシステムへの直接的な被害は発生していませんが、これは組織内の他のシステムも同様の攻撃に晒されている可能性を示す強力な警告です。上記推奨アクションを速やかに実施し、セキュリティ体制を強化してください。
