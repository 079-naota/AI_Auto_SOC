# SOC自動分析レポート: 191.255.52.36

**生成日時:** 2026-04-26 09:02:26

---

## SOC分析レポート

### 1. 攻撃の概要と目的

攻撃元IP `191.255.52.36` から、SSHサービスを標的としたブルートフォース攻撃が複数回観測されました。この攻撃者は、複数のユーザー名とパスワードの組み合わせを試行し、最終的に`root`ユーザーで複数のパスワード（`Hello@123`, `3245gs5662d34`, `123123Aa`, `odoo123`, `Aa111111!`）を用いてログインに成功しています。

ログイン成功後、攻撃者は以下の目的で行動していると推測されます。

*   **永続的なアクセス経路の確保**: `~/.ssh/authorized_keys`ファイルに自身の公開鍵を登録することで、パスワードなしでの再接続を可能にするバックドアを設置しようとしています。
*   **アカウントの乗っ取りと特権維持**: `root`ユーザーのパスワード変更を試み、システムに対する完全な制御を確立しようとしています。
*   **システム環境の偵察**: CPU情報、メモリ情報、実行プロセス、cronジョブ、ログインユーザー、システムアーキテクチャなど、多岐にわたるシステム情報を収集し、次の攻撃段階（マルウェア展開、データ窃取、さらなる横展開など）のための情報を収集しています。
*   **競合するマルウェアの排除**: `/tmp`ディレクトリ内の不審なスクリプトの削除や`/etc/hosts.deny`のクリアを試みることで、他の攻撃者によるバックドアを削除し、自身の支配を確固たるものにしようとしています。

### 2. 推測される手法・使用ツール

*   **初期アクセス**:
    *   **SSHブルートフォース攻撃**: `ubuntu`, `root`, `wang`, `345gs5662d34`といった一般的なユーザー名や数字を含む推測しやすいユーザー名を標的とし、辞書攻撃や一般的なパスワードリストを用いたと思われるブルートフォース攻撃によってSSH認証を突破しました。
*   **永続化**:
    *   **SSH公開鍵認証の悪用**: `rm -rf .ssh && mkdir .ssh && echo "..." >> .ssh/authorized_keys && chmod -R go= ~/.ssh`というコマンドを用いて、攻撃者の公開鍵（コメント: `mdrfckr`）を`root`ユーザーの`authorized_keys`に書き込み、パスワードなしでの永続的なアクセス経路を確立しようとしています。
    *   **アカウントパスワードの変更**: `echo "root:V7x7C26rMbb4"|chpasswd|bash`コマンドにより、`root`ユーザーのパスワード変更を試みています。
*   **特権昇格/防御回避**:
    *   **ファイル属性変更の試み**: `chattr -ia .ssh; lockr -ia .ssh`といったコマンドで、`.ssh`ディレクトリの変更不可属性を解除しようとしていますが、これはハニーポット環境では失敗しています。
*   **偵察**:
    *   `cat /proc/cpuinfo`, `free -m`, `crontab -l`, `w`, `uname -m`, `top`, `uname`, `uname -a`, `whoami`, `lscpu`など、多数の標準的なLinuxコマンドを連続して実行し、システムリソース、稼働状況、OS/ハードウェア情報、特権レベルなどを詳細に調査しています。
*   **競合排除**:
    *   `rm -rf /tmp/secure.sh; rm -rf /tmp/auth.sh; pkill -9 secure.sh; pkill -9 auth.sh; echo > /etc/hosts.deny; pkill -9 sleep;`といったコマンドにより、他の攻撃者によって設置された可能性のあるマルウェアやバックドアのスクリプトを削除し、実行中のプロセスを停止しようとしています。また、`/etc/hosts.deny`をクリアすることで、IPアドレスによるアクセス制限を無効にしようとしています。

これらの行動パターンから、攻撃者は自動化されたスクリプトやツール（例えば、SSHブルートフォースツールや、それに続く一連のコマンド実行スクリプト）を使用している可能性が高いと推測されます。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**

1.  **認証突破の成功**: 攻撃者は複数の異なるパスワードで`root`ユーザーとして複数回ログインに成功しており、これは最も高い特権を持つアカウントの認証情報が漏洩しているか、非常に推測しやすいパスワードが設定されていたことを示します。
2.  **永続的なアクセス確立の試み**: 攻撃者はログイン後すぐに`authorized_keys`に自身の公開鍵を追加しようとしており、一度ログインが成功すると、パスワード変更やブルートフォース対策が講じられても、継続的にアクセスできるバックドアを設置する意図があります。
3.  **システムに対する完全な制御の試み**: `root`パスワードの変更や、他のマルウェアの排除を試みる行動は、システムに対する完全な支配を目的としていることを明確に示しています。
4.  **広範な偵察活動**: ログイン後の偵察活動は非常に広範であり、攻撃者がシステムの詳細な情報を収集し、次の攻撃フェーズ（例: マルウェアの展開、データ窃取、ラテラルムーブメント）に進む準備をしていることを示唆しています。
5.  **現実世界での深刻な影響**: これが実際のシステムで発生した場合、システムは完全に侵害され、データの漏洩、破壊、サービス停止、他のシステムへの攻撃基盤としての悪用など、壊滅的な被害につながる可能性があります。

### 4. 推奨アクション

このログはハニーポットの記録ですが、実際のシステムで同様の事態が発生したと仮定し、以下の緊急対応および再発防止策を推奨します。

**A. 緊急対応（即時実施）**

1.  **攻撃元IPのブロック**: ファイアウォールやACLで、攻撃元IPアドレス `191.255.52.36` からのSSH接続を即座にブロックします。
2.  **侵害されたアカウントのパスワードリセット**:
    *   **`root`ユーザー**を含む、ログイン成功に使用された可能性のあるすべてのユーザーアカウントのパスワードを、複雑かつ強固なものに直ちにリセットします。
    *   パスワード変更後、すべての既存セッションを終了させます。
3.  **SSH `authorized_keys`の確認と削除**:
    *   `root`ユーザーの`~/.ssh/authorized_keys`ファイルを確認し、攻撃者によって追加された公開鍵（特にコメントが`mdrfckr`のもの）があれば、直ちに削除します。
    *   他のユーザーの`authorized_keys`も合わせて確認し、不審な鍵がないかチェックします。
4.  **システムからの隔離**: 該当システムをネットワークから一時的に隔離し、攻撃者からのさらなるアクセスや横展開を防ぎます。
5.  **システムイメージの取得**: フォレンジック調査のため、侵害された可能性のあるシステムのディスクイメージを取得します。

**B. フォレンジック調査**

1.  **ログの詳細分析**:
    *   SSHログ、認証ログ、システムログ、Webサーバーログ、ネットワーク機器のログなどを広範囲に分析し、攻撃者の活動時間、アクセス元、実行されたコマンド、作成・変更されたファイル、外部への通信など、攻撃の全貌を特定します。
    *   特に、公開鍵が追加された時刻以降の不審なSSH接続がないかを確認します。
2.  **マルウェアスキャンとバックドアの特定**:
    *   システム全体に対してマルウェアスキャンを実施し、潜在的なマルウェア、ルートキット、バックドアの有無を特定します。
    *   特に`/tmp`ディレクトリや`cron`設定、システムサービス、LD_PRELOADなどの箇所を重点的に確認します。
3.  **脆弱性評価**: システムに他の脆弱性が存在しないか、OS、ミドルウェア、アプリケーションレベルで脆弱性スキャンおよび評価を実施します。

**C. 再発防止策**

1.  **強力なパスワードポリシーの適用**:
    *   すべてのユーザーアカウントに対し、複雑性、長さ、有効期限に関する強力なパスワードポリシーを強制します。
    *   特に特権アカウント（`root`など）には、デフォルトパスワードや推測されやすいパスワードを使用しないよう徹底します。
2.  **SSH鍵認証の推奨とパスワード認証の無効化**:
    *   可能な限り、SSHのパスワード認証を無効化し、鍵認証のみを許可する設定に変更します。鍵にはパスフレーズを設定し、安全に管理する手順を確立します。
3.  **ブルートフォース対策の強化**:
    *   Fail2banなどの侵入検知・防止システムを導入し、SSHのログイン試行回数制限を設定し、不正なログイン試行を自動的にブロックします。
4.  **SSHポートの変更とアクセス制限**:
    *   SSHサービスのポートを標準の22番以外に変更することを検討します（ポートスキャンによる発見リスクは依然ありますが、自動化された攻撃の一部を防ぐ効果はあります）。
    *   SSHアクセスを特定の信頼できるIPアドレス範囲に限定するファイアウォールルールを設定します。
5.  **多要素認証 (MFA) の導入**:
    *   SSHログインに多要素認証を導入し、認証情報の窃取だけではアクセスできないようにセキュリティを強化します。
6.  **定期的なセキュリティパッチ適用**:
    *   OS、SSHサーバーを含むすべてのソフトウェアを最新の状態に保ち、既知の脆弱性を悪用した攻撃を防ぎます。
7.  **セキュリティ監視の強化**:
    *   SIEM (Security Information and Event Management) システムを導入し、SSHログイン成功/失敗、特権アカウントの活動、不審なコマンド実行などの異常なイベントをリアルタイムで監視し、早期検知・対応を可能にします。
8.  **侵入検知システム (IDS/IPS) の導入**:
    *   ネットワークレベルで不正アクセスや攻撃パターンを検知・防御するシステムを導入し、多層的な防御を構築します。
