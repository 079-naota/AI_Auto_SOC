## SOCアナリストレポート

### 1. 攻撃の概要と目的

攻撃元IPアドレス `43.155.21.198` から、複数のSSH接続が試行されました。この攻撃の主要な目的は、システムの不正侵入と、それに続く永続的なアクセス経路（バックドア）の確立であると推測されます。

**概要:**
1.  **初期偵察とブルートフォース/辞書攻撃**: 攻撃者はまず、`rootwww`などの一般的なユーザー名と弱いパスワードの組み合わせでログインを試行しました。これは、自動化されたツールによるブルートフォース攻撃または辞書攻撃の兆候です。
2.  **ルートアカウントへのログイン成功**: その後、攻撃者は`root`ユーザーに対し、複数の異なるパスワード（例: `6yhnji90-`, `Pan5202614`, `zhaojia`, `P@$$w0rd12`, `Login!@456`, `mail@1234`, `3245gs5662d34`）を繰り返し試行し、最終的にシステムへのログインに成功しています（ログはハニーポットの挙動を反映していますが、実システムであれば侵害が成立していたことを意味します）。
3.  **防御回避の試み**: ログイン成功後、攻撃者はまず `cd ~; chattr -ia .ssh; lockr -ia .ssh` というコマンドを実行し、`.ssh`ディレクトリに設定されている可能性のある保護属性（不変属性など）を解除しようとしました。これにより、後続の操作を可能にし、検出を回避する意図があると考えられます。
4.  **永続化メカニズムの確立**: 続けて、`cd ~ && rm -rf .ssh && mkdir .ssh && echo "ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr">>.ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~`というコマンドを実行しました。これは、既存の`.ssh`ディレクトリを削除し、再作成した上で、自身の公開鍵 (`ssh-rsa AAAAB3NzaC1yc2E... mdrfckr`) を`.ssh/authorized_keys`ファイルに書き込むことで、SSH鍵認証による永続的なバックドアを設置しようとするものです。これにより、将来パスワードが変更されても、攻撃者は自身の秘密鍵を用いてシステムに再アクセスできる状態を確立しようとしました。

### 2. 推測される手法・使用ツール

*   **手法**:
    *   **初期アクセス**: SSHブルートフォース攻撃 / 辞書攻撃 (MITRE ATT&CK T1110.001)
    *   **永続化**: SSH Authorized Keysを利用したバックドアの設置 (MITRE ATT&CK T1098.004)
    *   **防御回避**: `chattr`コマンドによるファイル属性変更の試み (MITRE ATT&CK T1562.001)
*   **使用ツール**:
    *   攻撃者が手動で操作しているというよりは、自動化されたスクリプトやボットネットの一部として、SSHブルートフォースツール（例: Hydra, Nmapの`ssh-brute`スクリプトなど）が使用されている可能性が高いです。
    *   ログイン成功後に実行された一連のコマンドは定型的であり、マルウェア（ボットネットのローダーなど）の機能として事前にスクリプト化されたものであると推測されます。
    *   特に、`.ssh/authorized_keys`に書き込まれた公開鍵には`mdrfckr`というコメントが付されており、この公開鍵はMiraiの亜種など、過去に多数のLinuxベースのIoTデバイスを狙ったボットネットのキャンペーンで利用されたことが報告されている既知の悪意ある公開鍵です。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**
1.  **ルート権限でのログイン成功**: 攻撃者が `root` ユーザーとしてログインに成功している点が最も重大です。ハニーポットのログであるため実システムへの侵害ではないものの、この攻撃は実システムであればシステムに対する完全な制御権が攻撃者に奪われたことを意味します。これにより、データ窃取、破壊、他のシステムへの攻撃基盤としての利用など、深刻な被害が発生する可能性があります。
2.  **永続的アクセスの確立試行**: 攻撃者は、SSH公開鍵認証によるバックドアを設置しようとしています。これは、パスワードが変更されても攻撃者が自由に再ログインできる永続的なアクセス経路を確立するものであり、単発の攻撃ではなく、長期的なシステム侵害やボットネットへの組み込みを目的としていることを強く示唆します。
3.  **既知の悪意ある公開鍵の使用**: `mdrfckr`というコメント付きの公開鍵が使用されていることは、この攻撃が広範な自動化されたマルウェアキャンペーン（ボットネット活動）の一部であることを示しており、攻撃の背景に組織的な脅威が存在する可能性が高いです。
4.  **防御回避の試み**: `chattr`コマンドによるファイル属性の変更試行は、システム管理者が設定した防御策を無効化し、より深くシステムを改竄しようとする意図を示しており、攻撃者の悪意の度合いが高いことを示します。

### 4. 推奨アクション

このログはハニーポットから得られたものですが、実環境で同様の攻撃が発生した場合を想定した緊急対応と予防的対策を以下に示します。

**A. 緊急対応（Immediate Response）**

1.  **攻撃元IPの即時ブロック**:
    *   ファイアウォール、IPS/IDS、セキュリティグループなどで、攻撃元IPアドレス `43.155.21.198` からの全ての通信を即座にブロックしてください。
2.  **侵害されたアカウントのパスワード変更**:
    *   攻撃者がログインに成功した `root` ユーザーのパスワードを直ちに変更し、強力で推測されにくいものに設定してください。
    *   可能であれば、他のシステムアカウント（特に管理者権限を持つアカウント）についてもパスワード変更を検討してください。
3.  **SSH公開鍵の確認と削除**:
    *   全てのユーザー（特に`root`ユーザー）のホームディレクトリ内の`.ssh/authorized_keys`ファイルを緊急で確認してください。
    *   ログに記載された不正な公開鍵（`ssh-rsa AAAAB3NzaC1yc2E... mdrfckr`）を含む、身に覚えのない公開鍵が追加されていないか確認し、発見された場合は直ちに削除してください。
4.  **システム侵害の包括的な調査（フォレンジック）**:
    *   攻撃者がログイン後に他の不正な活動（マルウェアのダウンロード・実行、他の設定変更、ログの削除など）を行っていないか、詳細なフォレンジック調査を実施してください。
        *   **システムログの確認**: `/var/log/auth.log`、`/var/log/secure`、`/var/log/syslog`などで、不審なログイン、コマンド実行、ファイル変更、エラーログがないか確認します。
        *   **実行中のプロセスとネットワーク接続の確認**: `ps aux`, `netstat -tulnp`などで、未知のプロセスや不審なネットワーク接続がないか調査します。
        *   **ファイルシステムの整合性チェック**: `/etc/passwd`, `/etc/shadow`, `/etc/ssh/sshd_config`などの重要な設定ファイルや実行ファイルが改ざんされていないか確認します。
        *   **マルウェアスキャン**: 既知のマルウェアスキャナーを用いて、システム全体をスキャンし、マルウェアの有無を確認します。
        *   **コマンド履歴の確認**: 各ユーザーの`.bash_history`などを確認し、不正なコマンド実行がないか確認します。

**B. 予防的対策（Preventive Measures）**

1.  **SSH設定の厳格化**:
    *   **`PermitRootLogin no`**: `/etc/ssh/sshd_config`ファイルで`root`ユーザーのSSHログインを禁止してください。管理者作業は一般ユーザーでログイン後、`sudo`を使用するように徹底してください。
    *   **`PasswordAuthentication no`**: パスワード認証を無効にし、公開鍵認証のみを許可してください。
    *   **`AllowUsers`または`AllowGroups`**: SSHアクセスを許可するユーザーを明示的に指定し、最小限のユーザーに限定してください。
    *   **デフォルトポートの変更**: SSHサービスをデフォルトの22番ポートから別のポート番号に変更する（セキュリティ強化策の一つとして）。
2.  **ブルートフォース対策ツールの導入**:
    *   Fail2banなどのツールを導入し、不正なログイン試行を自動的に検知・ブロックする仕組みを構築してください。
3.  **ネットワークアクセス制限**:
    *   SSHサービスをインターネットに直接公開せず、VPNやBastionホスト経由でのアクセスに限定してください。
    *   必要な場合でも、特定の信頼できるIPアドレスからのSSHアクセスのみを許可するようにファイアウォールルールを設定してください。
4.  **強力な認証情報の利用**:
    *   全てのシステムアカウントに対し、複雑で推測困難なパスワードポリシーを強制し、定期的なパスワード変更を義務付けてください。
    *   可能であれば、SSH鍵認証と合わせて多要素認証 (MFA) の導入を検討してください。
5.  **システムの監視強化**:
    *   SIEM（Security Information and Event Management）などのログ管理システムを導入し、SSHログイン試行、成功、コマンド実行などのログをリアルタイムで監視し、異常を自動的にアラートする仕組みを構築してください。
    *   ファイル整合性監視ツール（AIDE、Tripwireなど）を導入し、重要なシステムファイルやディレクトリ（特に`.ssh`ディレクトリ）の改ざんを検知する体制を整えてください。
6.  **定期的な脆弱性診断とパッチ適用**:
    *   システムの脆弱性スキャンを定期的に実施し、見つかった脆弱性に対して速やかにパッチを適用してください。
    *   不要なサービスやオープンポートを無効化してください。
