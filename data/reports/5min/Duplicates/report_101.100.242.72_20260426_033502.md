# SOC自動分析レポート: 101.100.242.72

**生成日時:** 2026-04-26 03:37:16

## SOCアナリストレポート

### 1. 攻撃の概要と目的

攻撃元IPアドレス `101.100.242.72` から、組織の公開されているSSHサービスに対して執拗なブルートフォース攻撃および辞書攻撃が観測されました。攻撃は、複数の異なる一般的なユーザー名（`frappe`, `deployer`, `ubuntu`, `admin`, `tester`, `test`, `oracle`, `user`, `zhaoxj` など）と、特に管理者権限を持つ `root` ユーザーに対して、複数の強力なパスワード（`qwer1234QWER!@#$`, `Ni123456`, `ubuntu@123`, `Password.123`, `a12348765` など）を試行する形で実施されています。

ログイン成功（ハニーポット上での模擬的成功）後、攻撃者は以下の活動を複数回にわたり試行しています。

1.  **アカウント奪取**: 試行錯誤の結果、`root` ユーザーの認証情報（ユーザー名とパスワードの組み合わせ）を特定することに成功しています。
2.  **永続化 (Persistence)**: ログイン後、`cd ~ && rm -rf .ssh && mkdir .ssh && echo "ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr">>.ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~` というコマンドを実行し、`root` ユーザーの `~/.ssh/authorized_keys` ファイルに特定の公開鍵を追加しようとしています。これは、今後パスワードなしでSSHアクセスを確立するためのバックドア設置を目的としています。
3.  **防御回避 (Defense Evasion)**: `cd ~; chattr -ia .ssh; lockr -ia .ssh` というコマンドを試行し、`.ssh` ディレクトリに設定されている可能性のある `immutable` や `append-only` などのファイル属性を解除しようとしています。これは、永続化の試みを妨げるセキュリティ対策を無効化する意図があると推測されます。

この攻撃は、ターゲットシステムへの不正アクセスを確立し、長期的な支配を確立することを明確な目的としています。

### 2. 推測される手法・使用ツール

*   **初期アクセス**:
    *   **ブルートフォース/辞書攻撃**: 攻撃者は、自動化されたツールを使用して、SSHサービスに対して大量のユーザー名とパスワードの組み合わせを試行しています。特に `root` ユーザーに対する試行は執拗であり、複数の異なるパスワードが成功しています。
*   **永続化**:
    *   **SSH公開鍵の追加**: 認証に成功した後、攻撃者は `~/.ssh/authorized_keys` ファイルに自身が管理する公開鍵を追加することで、将来的にパスワードなしでの再アクセス経路を確立しようとしています。これにより、たとえパスワードが変更されてもアクセスを維持できます。
*   **防御回避**:
    *   **ファイル属性変更**: `chattr -ia .ssh` コマンドは、Linuxシステムにおけるファイルやディレクトリの属性を変更するもので、`i` (immutable) や `a` (append-only) 属性が設定されている場合、ファイルの削除や変更ができなくなります。攻撃者はこの属性を解除することで、`.ssh` ディレクトリの変更を可能にし、永続化の試みを成功させようとしています。
    *   `lockr` コマンドは、Cowrieハニーポットのログでは `cowrie.command.failed` と記録されていることから、認識されないか、ハニーポットがエミュレートしていない特殊なコマンドである可能性があります。

*   **使用ツール**:
    *   **自動化されたSSHクライアント**: 大量のログイン試行を高速に行っていることから、HydraやMedusa、またはカスタムスクリプトなどのSSHブルートフォースツールを使用していると考えられます。
    *   **シェルコマンド**: ログイン後の活動には、標準的なLinuxシェルコマンド（`cd`, `rm`, `mkdir`, `echo`, `chmod`, `chattr`）が使用されています。
    *   **公開鍵**: 攻撃者が使用している公開鍵 (`ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`) は、特定のボットネットやマルウェアキャンペーンで利用されている既知の鍵である可能性があります。

### 3. 脅威レベルとその理由

*   **脅威レベル**: **高 (High)**

*   **理由**:
    1.  **rootアカウントの認証情報奪取**: 攻撃者は複数の異なるパスワードで `root` ユーザーのログインに繰り返し成功しています。実システムであれば、これはシステムに対する完全な制御権を攻撃者に与えることに等しく、最も深刻な結果を招きます。
    2.  **永続化の確立**: ログイン成功後の公開鍵の追加は、攻撃者が一度の侵入で終わらず、将来にわたって継続的にシステムへアクセスしようとしていることを明確に示しています。これにより、システムのセキュリティ対策が一時的に強化されても、バックドアを通じて再侵入されるリスクが生じます。
    3.  **防御回避の試み**: `chattr` コマンドの使用は、管理者が設定している可能性のあるファイル保護メカニズムを攻撃者が理解し、それを無効化しようとしていることを示します。これは、攻撃者が高度な知識と明確な悪意を持っていることを裏付けます。
    4.  **継続的かつ自動化された攻撃**: 短期間にわたる多数のログイン試行と、ログイン成功後のコマンド実行の繰り返しは、攻撃が自動化されたツールやボットネットによって行われていることを強く示唆します。これは、標的型攻撃ではないにしても、組織のシステムが広範な悪意あるキャンペーンの一部として狙われている可能性が高いことを意味します。
    5.  **既知の攻撃パターン**: SSHブルートフォース、`root` ユーザーの奪取、`authorized_keys` への公開鍵追加、`chattr` による防御回避は、Linuxサーバーを狙うマルウェア（例: Mirai、XOR DDoS）やクリプトマイナーなどによる侵害活動で頻繁に見られる典型的なパターンです。

これらの理由から、この攻撃は非常に高い脅威レベルと評価され、もし実システムで発生していれば、重大な侵害につながる可能性がありました。

### 4. 推奨アクション

このログはハニーポットからのものですが、実システムで同様の事象が発生した場合を想定し、以下の推奨アクションを実施してください。

#### 1. 緊急対応 (Immediate Response)

*   **攻撃元IPのブロック**: 攻撃元IPアドレス `101.100.242.72` を組織のファイアウォール、IPS/IDS、またはSSHデーモン設定（`denyhosts`, `fail2ban` など）で即座にブロックし、さらなる通信を遮断してください。
*   **侵害システムの隔離**: もしこのログが実際のシステムで発生していた場合、当該システムをネットワークから直ちに隔離し、被害の拡大を防止してください。
*   **認証情報の変更**:
    *   ログに記録されたすべての成功した `root` パスワード（`qwer1234QWER!@#$`, `3245gs5662d34`, `Ni123456`, `ubuntu@123`, `Password.123`, `a12348765`）を、非常に強力でユニークなものに直ちに変更してください。
    *   システム上の全てのアカウント（特に管理者権限を持つアカウント）のパスワードを見直し、もし脆弱なものがあれば変更してください。
*   **不正なSSH公開鍵の削除**:
    *   `root` ユーザーの `~/.ssh/authorized_keys` ファイルを確認し、ログに記録された公開鍵 (`ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`) が存在すれば即座に削除してください。
    *   他のユーザーアカウントの `~/.ssh/authorized_keys` ファイルも同様に確認し、不審な鍵がないかチェックしてください。
*   **ログの詳細調査とフォレンジック**:
    *   システムの認証ログ (`/var/log/auth.log` など)、システムログ、アプリケーションログを詳細に調査し、攻撃者の活動範囲、ログイン日時、実行されたコマンド、不正なファイルのアップロード/ダウンロード、新しいユーザーアカウントの作成、特権昇格、異常なプロセスの有無などを特定してください。
    *   疑わしいファイルのハッシュ値をVirusTotalなどで確認してください。

#### 2. 予防策 (Preventive Measures)

*   **SSH認証の強化**:
    *   **多要素認証 (MFA) の導入**: SSHログインにMFAを必須とし、パスワードが漏洩しても不正ログインを防ぐ仕組みを導入してください。
    *   **公開鍵認証への移行**: パスワード認証を完全に無効化し、より安全な公開鍵認証のみを許可する運用に切り替えてください。公開鍵の管理も厳格に行う必要があります。
    *   **rootログインの制限**: SSH経由での `root` ユーザーの直接ログインを禁止し、一般ユーザーでログイン後に `sudo` を利用する運用に変更してください。
*   **強力なパスワードポリシーの実施**:
    *   全てのアカウントに対して、最低14文字以上で、英大文字・小文字、数字、記号を組み合わせた複雑なパスワードの使用を義務付けてください。定期的なパスワード変更も奨励してください。
*   **アクセス制御の厳格化**:
    *   **IPアドレス制限**: SSHサービスへのアクセスを、特定の信頼できるIPアドレス範囲に限定してください。
    *   **ポート変更**: SSHの標準ポート22番を非標準ポートに変更する（セキュリティバイオブスキュリティに過ぎませんが、自動化されたスキャンからのヒットを減らす効果はあります）。
*   **システムセキュリティ hardening**:
    *   **不要なアカウントの削除/無効化**: 未使用のシステムアカウントやデフォルトアカウントを特定し、削除または無効化してください。
    *   **最新のセキュリティパッチ適用**: OSやSSHサービスを含むすべてのソフトウェアを常に最新の状態に保ち、既知の脆弱性が悪用されないようにしてください。
    *   **侵入防止システム (IPS) / ファイアウォールの設定強化**: SSHへのブルートフォース攻撃を検知し、自動的にIPブロックを行う Fail2ban などのツールを導入してください。
*   **定期的なセキュリティ監査**:
    *   SSH設定（`/etc/ssh/sshd_config`）の定期的なレビューを行い、不適切な設定がないか確認してください。
    *   ユーザーアカウント、特に特権アカウントの定期的な監査を実施してください。

#### 3. 検知強化 (Detection Enhancement)

*   **ログ監視とアラート**:
    *   SSHのログイン成功/失敗、特に `root` ユーザーや特権アカウントのログイン試行に対するアラートをSIEMやログ管理システムに設定してください。
    *   短時間での多数のログイン失敗や、異常なログイン元IPアドレスからのアクセスを検知するルールを設定してください。
    *   `authorized_keys` ファイルへの変更、`.ssh` ディレクトリへの変更、`chattr` コマンドの実行など、重要なファイルや設定に対する変更を監視し、アラートを発するよう設定してください。
*   **挙動ベースの検知**:
    *   不明なプロセスや異常なネットワーク通信（特に外部へのアウトバウンド通信）を監視し、マルウェア感染やC2通信の兆候を検知してください。

これらの推奨アクションを実施することで、将来的な類似攻撃から組織のシステムを保護し、セキュリティ体制を強化することができます。
