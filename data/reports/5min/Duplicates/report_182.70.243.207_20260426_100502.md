# SOC自動分析レポート: 182.70.243.207

## SOCアナリストレポート

### 1. 攻撃の概要と目的

この攻撃は、攻撃元IP `182.70.243.207` から当システム（Cowrieハニーポット）のSSHサービスに対して行われました。攻撃者は、主にブルートフォース/パスワードリスト攻撃によってSSH認証情報の突破を試み、複数回 `root` ユーザーでのログインに成功しています。

ログイン成功後、攻撃者は自身のSSH公開鍵を`authorized_keys`ファイルに追加することで、バックドアを設置し、将来にわたってパスワードなしでシステムへの永続的なアクセス経路を確立しようとしました。これは、システムを乗っ取り、継続的なアクセスを維持することを目的とした明確な永続化（Persistence）の試みです。

### 2. 推測される手法・使用ツール

*   **初期アクセス**:
    *   **SSHブルートフォース/パスワードリスト攻撃**: 攻撃者は短時間のうちに多数のユーザー名とパスワードの組み合わせを試行しています。特に`root`ユーザーを標的とし、「`QAZwsx`」「`Root1234567@@`」「`1q2w3e..`」「`root123123!!`」「`@a123456`」「`Root11111111@#`」といった、辞書攻撃やパスワードリスト攻撃でよく使われる、あるいは過去の漏洩情報に含まれる可能性のあるパスワードを複数回試行し、ログインに成功しています。また、「`steam`」「`tunnel`」「`cloud`」「`gitlab`」など、一般的なユーザー名も試されています。
*   **永続化**:
    *   **SSH公開鍵認証バックドアの設置**: ログインに成功した後、攻撃者は以下のコマンドを実行し、`~/.ssh/authorized_keys`ファイルに自身のSSH公開鍵を追記することで、パスワードなしでの再ログインを可能にしようとしました。

    *   特定のツール名は特定できませんが、一連の自動化されたログイン試行とコマンド実行パターンから、スクリプトまたは自動化ツール（例: `Hydra`, `Medusa`などのブルートフォースツール）が使用されている可能性が高いです。また、`chattr`や`lockr`といったコマンドの試行（Cowrie上では失敗）は、ファイル属性の変更を通じて永続化を強固にしようとする意図、あるいは攻撃者が使用するカスタムスクリプトの一部である可能性を示唆します。

### 3. 脅威レベルとその理由

*   **脅威レベル: 高 (High)**

*   **理由**:
    1.  **複数回のrootログイン成功**: 攻撃者はシステム上で最も高い権限を持つ`root`ユーザーとして複数回認証に成功しています。これにより、システムに対する完全な制御権を取得する可能性が極めて高まります。
    2.  **永続化の試み**: 攻撃者はログイン後すぐにSSH公開鍵を挿入し、バックドアを設置しようとしています。これは、一時的なアクセスに留まらず、将来にわたって継続的にシステムにアクセスしようとする明確な意図を示しており、システムの長期的な侵害リスクを増大させます。
    3.  **攻撃の継続性**: ログイン成功後も、異なるパスワードやユーザー名での試行が継続しており、攻撃者がより多くのシステムアカウントへのアクセスを試みるか、あるいはバックドアの動作確認、さらなる偵察活動などを行おうとしている可能性があります。
    4.  **広範な攻撃範囲**: 複数の異なるパスワードを試行していることから、総当たり攻撃またはパスワードリスト攻撃が組織的に行われていると推測され、他のシステムへの同様の攻撃も懸念されます。

### 4. 推奨アクション

1.  **緊急対応 (Immediate Actions)**
    *   **攻撃元IPのブロック**: ファイアウォールやIDS/IPSにおいて、攻撃元IP `182.70.243.207` からのSSH (TCP/22) およびその他の通信を直ちにブロックリストに追加してください。
    *   **侵害されたアカウントのパスワード変更**: ログに記録されたログイン成功に使用されたすべての`root`パスワード (`QAZwsx`, `3245gs5662d34`, `Root1234567@@`, `1q2w3e..`, `root123123!!`, `@a123456`, `Root11111111@#`) を、本番環境のシステムで使用している場合は、直ちに強力なものに変更してください。
    *   **不正なSSH公開鍵の削除**: 実際のシステム上で、`~/.ssh/authorized_keys`ファイルを確認し、ログに記録された以下の公開鍵が存在する場合は直ちに削除してください。
        ```
        ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr
        ```
    *   **システム侵害の有無の詳細調査**: `root`権限でのログインに成功しているため、さらなる侵害（マルウェアのインストール、データの窃取、他のシステムへの横展開、アカウントの追加など）が行われている可能性が高いです。システムログ（auth.log, syslog）、プロセスリスト、ネットワーク接続、ファイルシステムの変更などを詳細に調査し、不審なアクティビティがないか確認してください。
    *   **関連する脅威情報の検索**: ログに記録されたSSH公開鍵 (`mdrfckr`のコメント付き) や攻撃に使用されたパスワードについて、脅威インテリジェンスデータベース（VirusTotal, MISPなど）で検索し、既知のマルウェアキャンペーンや攻撃グループとの関連性を確認してください。

2.  **予防的措置 (Preventative Measures)**
    *   **SSHサービスの設定強化**:
        *   **パスワード認証の無効化**: 可能であれば、SSHパスワード認証を無効にし、公開鍵認証のみに限定してください。
        *   **rootログインの禁止**: `PermitRootLogin no` を設定し、`root`ユーザーでのSSH直接ログインを禁止してください。必要に応じて一般ユーザーでログインし、`su`または`sudo`で権限昇格を行う運用を推奨します。
        *   **SSHポートの変更**: 標準の22番ポートから非標準のポートに変更することで、自動化された攻撃スクリプトからのスキャンを軽減できます。
        *   **アクセス制御の強化**: `AllowUsers`や`AllowGroups`設定により、SSHアクセスを許可するユーザーやグループを厳密に制限してください。
    *   **アカウントとパスワードポリシーの強化**:
        *   すべてのシステムアカウントで、複雑性、長さ、定期的な変更を強制する強力なパスワードポリシーを導入してください。
        *   多要素認証 (MFA) の導入を検討し、認証情報の漏洩時のリスクを低減してください。
    *   **侵入防御システムの導入**: Fail2Banのようなツールを導入し、一定回数のログイン失敗があったIPアドレスからのアクセスを自動的にブロックする設定を適用してください。
    *   **セキュリティログの監視強化**: SSH認証ログ（`auth.log`など）やシステムログをリアルタイムで監視し、不審なログイン試行や認証成功を早期に検知できる体制を構築してください。SIEMソリューションの活用も有効です。
    *   **脆弱性管理**: 定期的にシステムやアプリケーションの脆弱性スキャンを実施し、最新のセキュリティパッチを適用してください。
