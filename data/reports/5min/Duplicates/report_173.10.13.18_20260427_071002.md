## SOCアナリストレポート：SSHサービスへの継続的な攻撃分析

### 1. 攻撃の概要と目的

攻撃元IPアドレス `173.10.13.18` から、当組織のSSHサービスに対して継続的なブルートフォース攻撃が確認されました。この攻撃は複数の異なるユーザー名とパスワードを試行し、最終的に `root` ユーザーの認証情報を特定し、システムへの永続的なアクセス経路を確立しようとしたものです。

攻撃の主な目的は以下の通りと推測されます。
*   **初期アクセス（Initial Access）**: 既知の認証情報や弱いパスワードを試行し、SSHサービスへの不正ログインを試みる。
*   **永続化（Persistence）**: `root` ユーザーとしてログイン後、自身のSSH公開鍵を `~/.ssh/authorized_keys` ファイルに追加することで、パスワードなしでの再ログインを可能にするバックドアを設置し、システムへの永続的なアクセス経路を確保する。
*   **防御回避（Defense Evasion）**: `chattr` コマンドを用いて追加したファイルの属性を変更し、検出や削除を困難にしようと試みる。
*   **影響（Impact）/（追加マルウェアの展開）**: `cowrie.session.file_download` イベントの発生は、さらなるマルウェア（例: ボットネットのエージェント、クリプトマイナーなど）をダウンロードし、実行を試みた可能性を示唆しています。

### 2. 推測される手法・使用ツール

**推測される攻撃手法:**

*   **ブルートフォース攻撃/辞書攻撃**: `dev`, `trans`, `root` といった一般的なユーザー名に対し、`!dev`, `trans`, `123456Aa`, `Aa000000`, `Sp0rt`, `123456789a@`, `qaz123!@#`, `Aa12488261` などのパスワードを繰り返し試行しています。これは自動化されたスクリプトやツールによる攻撃の特徴です。
*   **アカウントの侵害（Compromise Accounts）**: `root` ユーザーの認証情報の特定に成功し、最高権限でのアクセスを獲得しています。
*   **認証情報の変更（Modify Authentication Credentials）**: ログイン後、`authorized_keys` ファイルを操作し、攻撃者の公開鍵を追記することで、鍵ベースの認証によるアクセス経路を確保しようとしています。これはシステムへのバックドア設置の典型的な手法です。
*   **リモートコード実行（Remote Code Execution）**: ログイン後に `cd`, `rm`, `mkdir`, `echo`, `chmod` といったコマンドを実行し、ファイルシステムの変更を行っています。`chattr` や `lockr` コマンドも試行しており、システム操作の意図が明確です。
*   **ファイルダウンロード（File Download）**: `cowrie.session.file_download` イベントは、外部からの追加のペイロードやマルウェアのダウンロードを試みた可能性を示しています。

**推測される使用ツール:**

*   **SSHクライアント**: 標準的なSSHクライアントが使用されています。
*   **ブルートフォースツール**: `Hydra` や `Nmap` の `ssh-brute` スクリプト、またはカスタムのブルートフォーススクリプトなど、自動でSSHログイン試行を行うツールが使用された可能性が高いです。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**

1.  **root権限での複数回のログイン成功**: ログには`root`ユーザーとして、複数の異なるパスワードで繰り返しログインに成功した記録があります。`root`はLinuxシステムにおける最高権限ユーザーであり、このアカウントが侵害された場合、攻撃者はシステムに対する完全な制御権を獲得できます。
2.  **永続的なアクセス経路の確立試行**: 攻撃者はログイン後すぐに、自身のSSH公開鍵を `authorized_keys` に追加するコマンドを実行しています。これは、パスワードが変更された場合でも、公開鍵認証を用いてシステムに再接続するためのバックドアを設置する明確な意図を示しており、システムへの持続的な侵害リスクを高めます。
3.  **自動化された攻撃の継続性**: 攻撃は断続的ではなく、短期間のうちに多数のログイン試行と、ログイン成功後のコマンド実行を繰り返しています。これは自動化された攻撃であり、特定のパスワードが機能しなくなった場合でも、攻撃者が他の手段で侵入を試み続けることを示唆しています。
4.  **潜在的なマルウェア感染の可能性**: `cowrie.session.file_download` イベントは、システムにさらなるマルウェアを導入しようとした可能性を示唆しており、これが成功した場合、システムの機能停止、データ漏洩、リソースの悪用（例: クリプトマイニング、ボットネット活動）など、広範な被害につながるリスクがあります。
5.  **既知のパスワードリストの使用**: ログインに成功したパスワード（例: `123456Aa`, `Sp0rt`, `qaz123!@#`）は、一般的なブルートフォース攻撃や辞書攻撃で試行される弱いパスワードのパターンに属しています。

### 4. 推奨アクション

**緊急対応 (Immediate Actions):**

1.  **攻撃元IPのブロック**: 攻撃元IPアドレス `173.10.13.18` をファイアウォールやIDS/IPSで直ちにブロックリストに追加し、今後の接続を拒否します。
2.  **（実システムの場合）システムのネットワーク隔離**: もしこのログが実システムのものであれば、該当するサーバーをネットワークから隔離し、攻撃の拡散を阻止します。
3.  **パスワードの緊急変更**: ログに記録されたログイン成功パスワード (`123456Aa`, `Aa000000`, `Sp0rt`, `123456789a@`, `qaz123!@#`, `Aa12488261`, `3245gs5662d34`) が、組織内の他のシステムやサービスで使われていないかを確認し、使用されている場合は即座にこれら全てのパスワードを、強力でユニークなものに変更します。
4.  **authorized_keysの確認と不正鍵の削除**: 組織内の全システムを対象に、`~/.ssh/authorized_keys` ファイルに攻撃者の公開鍵 (`ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`) が追加されていないか緊急で確認し、存在する場合は直ちに削除します。
5.  **フォレンジック調査**: 影響を受けた可能性のあるシステムに対し、不正なプロセス、不審なファイル、予期しないネットワーク接続、他の永続化メカニズム（例: cronジョブ、サービス、スタートアップスクリプト）がないか、徹底的なフォレンジック調査を実施します。

**予防的対策 (Preventive Measures):**

1.  **SSH設定の強化**:
    *   **パスワード認証の無効化**: 可能な限り公開鍵認証のみを許可し、パスワード認証を無効にします。
    *   **Rootログインの禁止**: `PermitRootLogin no` を設定し、`root`ユーザーでの直接SSHログインを禁止します。代わりに一般ユーザーでログイン後、`su`または`sudo`を使用するようにします。
    *   **ユーザー制限**: `AllowUsers`または`AllowGroups`ディレクティブを使用し、SSHアクセスを許可するユーザーを厳密に制限します。
    *   **ポート変更**: SSHサービスを標準ポート (22) から非標準ポートに変更することを検討します (自動スキャンからの可視性を減らす効果があります)。
2.  **多要素認証 (MFA) の導入**: SSHアクセスを含む全ての重要なシステムアクセスに多要素認証を義務付け、認証セキュリティを大幅に強化します。
3.  **強力なパスワードポリシーの適用**:
    *   最低文字数、大文字・小文字・数字・記号の組み合わせを強制するなど、複雑なパスワードポリシーを設定します。
    *   一般的な辞書ワードや過去に漏洩したパスワードの使用を禁止する仕組みを導入します。
    *   定期的なパスワード変更を推奨または強制します。
4.  **侵入検知・防止システム (IDS/IPS) の導入と強化**:
    *   SSHブルートフォース攻撃を検知し、自動的に攻撃元IPをブロックするFail2banなどのツールを導入します。
    *   SSHサービスへの不審なログイン試行や成功、`authorized_keys`ファイルへの変更、不審なコマンド実行を検知するアラートルールを設定します。
5.  **定期的な脆弱性スキャンとパッチ管理**:
    *   システムとアプリケーションの脆弱性スキャンを定期的に実施し、発見された脆弱性には速やかにパッチを適用します。
6.  **ログ監視の強化**:
    *   Cowrieのようなハニーポットのログを継続的に監視し、攻撃の傾向や新たな手法を特定します。
    *   実際のサーバーの認証ログ (`/var/log/auth.log`など) やセキュリティログを一元的に収集・監視するシステム（SIEMなど）を導入し、リアルタイムでの検知と対応能力を高めます。
