### 1. 攻撃の概要と目的

攻撃元IPアドレス `186.10.86.130` から、SSHサービスに対して複数の認証試行が確認されました。最初の試行から約7分後に、`root`ユーザーに対するブルートフォース/辞書攻撃が成功し、その後、攻撃者はシステムへの永続的なアクセス経路を確立するための明確な試みを行いました。

攻撃の主な目的は以下の通りと推測されます。

*   **初期アクセス**: `root`ユーザーの有効な認証情報（ユーザー名とパスワード）を発見し、システムへの不正なログインを成功させること。
*   **永続化**: ログイン成功後、`root`ユーザーのホームディレクトリ配下の`.ssh/authorized_keys`ファイルに自身のSSH公開鍵を追記することで、パスワードなしで将来的にシステムへ再接続できるバックドアを設置すること。これにより、認証情報の有効期限切れや変更に影響されずに、恒久的なアクセス権を確保しようとしていました。
*   **さらなる活動の足がかり**: 永続化に成功した場合、攻撃者はシステム上でマルウェアの展開、データ窃取、ボットネットへの参加、他のシステムへの横展開など、さらなる悪意ある活動を行うための足がかりを得ることが目的と推測されます。

### 2. 推測される手法・使用ツール

本攻撃で推測される手法と使用ツールは以下の通りです。

*   **ブルートフォース/辞書攻撃**:
    *   攻撃者は、`rootalias`やランダムな文字列（例: `345gs5662d34`）を含む複数のユーザー名やパスワードを試行しており、特に`root`ユーザーに対しては様々なパスワード（例: `p2puser@123`, `3245gs5662d34`, `g`, `root1234567`, `uway2026`, `c*o5^V:^[^^`, `cnp200@HW`, `root1.2026`）を短期間に大量に試行しています。これは、自動化されたブルートフォースツールや辞書攻撃ツールを使用している典型的な挙動です。
*   **SSH公開鍵による永続化**:
    *   ログイン成功後、攻撃者は以下のコマンドを実行しようとしました。
        ```bash
        cd ~ && rm -rf .ssh && mkdir .ssh && echo "ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr">>.ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~
        ```
    *   このコマンドは、`~/.ssh`ディレクトリを再作成し、攻撃者の公開鍵（コメント部分に`mdrfckr`と記載）を`authorized_keys`ファイルに書き込むことで、SSH公開鍵認証によるバックドアを設置するものです。これにより、攻撃者はパスワードを知らなくても、いつでもシステムにSSHで接続できるようになります。
*   **ファイル属性操作の試行**:
    *   バックドア設置の前に、`cd ~; chattr -ia .ssh; lockr -ia .ssh`というコマンドを試行しています。これは、`.ssh`ディレクトリのimmutable属性やappend-only属性を解除し、書き込みを可能にしようとする意図があると推測されます（`lockr`コマンドは一般的なLinuxコマンドではないため、誤字か特定の環境に依存するコマンドである可能性があります）。この試行はログ上では失敗しています。
*   **自動化ツール**:
    *   一連のログイン試行、コマンド実行のパターンは非常に機械的で反復的であり、攻撃者が特定のSSHブルートフォースおよびバックドア設置スクリプトまたはツールを使用している可能性が高いことを示唆しています。

### 3. 脅威レベルとその理由

**脅威レベル**: **高 (High)**

**理由**:

1.  **`root`アカウントへの認証成功**: 攻撃者が最も特権の高い`root`アカウントの認証情報（ユーザー名とパスワード）を発見し、ログインに成功している点は極めて深刻です。`root`権限はシステムへの完全な制御を可能にします。
2.  **永続化の試み**: ログイン成功後、攻撃者は直ちにSSH公開鍵によるバックドアを設置しようとしています。これは単一のセッションでの侵害に留まらず、将来にわたるシステムへの恒久的なアクセス権を確立しようとする明確な意図を示しており、長期的な侵害リスクを伴います。
3.  **攻撃の自動化**: 攻撃が自動化されたツールによって行われていることから、組織が広範なインターネットスキャンと無差別攻撃の標的になっている可能性が高く、同様の攻撃が他のサービスやシステムに対しても行われるリスクがあります。
4.  **潜在的な二次攻撃**: バックドアが設置された場合、攻撃者はシステムをマルウェア感染、データ窃取、DDoS攻撃の踏み台、または内部ネットワークへの横展開のための拠点として悪用する可能性があります。

### 4. 推奨アクション

この攻撃を受けて、以下の緊急対応と恒久対策を直ちに実行することを推奨します。

#### 緊急対応 (Immediate Actions):

1.  **攻撃元IPのブロック**: ファイアウォール、IDS/IPSなどのセキュリティデバイスで、攻撃元IPアドレス `186.10.86.130` からのすべての通信を即座にブロックしてください。
2.  **SSH認証情報の変更**:
    *   ログに記録されたすべての認証成功パスワード（`p2puser@123`, `3245gs5662d34`, `g`, `root1234567`, `uway2026`, `c*o5^V:^[^^`, `cnp200@HW`, `root1.2026`）を使用しているすべてのユーザーアカウント（特に`root`）について、直ちにパスワードを強力でユニークなものに変更してください。
    *   これらのパスワードが他のシステムやサービスで使い回されていないかを確認し、もしあればそれらも変更してください。
3.  **SSH公開鍵の確認と削除**:
    *   すべてのSSHサーバ（特に`root`ユーザー）において、`~/.ssh/authorized_keys`ファイルの内容を精査し、不正なSSH公開鍵（特にログに記録されたコメント`mdrfckr`を持つ鍵）が登録されていないか確認してください。
    *   不正な鍵が発見された場合は、直ちにそのエントリを削除し、ファイルのパーミッションが適切（通常は`600`）であることを確認してください。
4.  **システム全体でのセキュリティ監査**:
    *   SSHサービスが稼働しているシステムに対し、マルウェアスキャン、設定の確認、追加の不審なファイルやプロセスの有無、他のバックドアの有無など、包括的なセキュリティ監査を迅速に実施してください。
    *   他のユーザーアカウントに対する不審な活動がないかも確認してください。
5.  **関連ログの収集と分析**:
    *   SSHサービスだけでなく、OSの認証ログ、システムログ、ネットワーク機器のログなど、関連するすべてのログを収集し、この攻撃に関連するさらなる兆候（例えば、他のポートへのアクセス、不明なプロセス起動、設定変更など）がないか詳細に分析してください。

#### 恒久対策 (Long-Term Measures):

1.  **SSHパスワードポリシーの強化**:
    *   SSHアクセスに利用されるすべてのパスワードに対して、複雑性、長さ、変更頻度に関する厳格なポリシーを適用してください。
    *   辞書攻撃やブルートフォース攻撃に強い、ランダム性の高いパスワードの利用を義務付けてください。
2.  **多要素認証 (MFA) の導入**:
    *   SSHログインにおいて、パスワード認証だけでなく、ハードウェアトークンやTOTPなどの多要素認証を必須とすることで、認証情報の漏洩リスクを大幅に軽減できます。
3.  **SSH公開鍵認証の厳格な管理とパスワード認証の無効化**:
    *   可能であれば、SSHのパスワード認証を無効化し、公開鍵認証のみを許可する設定に変更することを強く推奨します。
    *   公開鍵の登録、配布、管理プロセスを厳格化し、不正な鍵が登録されないように定期的に監査してください。
4.  **SSHアクセスの制限**:
    *   SSHポート（デフォルト22番）をインターネットに直接公開するのではなく、VPN経由でのアクセスを必須とするか、特定の許可されたIPアドレスからの接続のみをホワイトリスト形式で許可するなどのアクセス制限を導入してください。
    *   SSHのデフォルトポートを変更することも、攻撃のノイズを減らすために有効です（ただし、セキュリティの本質的な強化にはなりません）。
5.  **ログ監視とアラートの強化**:
    *   SSHログイン失敗回数の閾値設定、ログイン成功後の不審なコマンド実行、`authorized_keys`ファイルへの変更、新たなファイルの作成やダウンロードなど、異常な挙動を検知するためのリアルタイム監視とアラートシステムを強化してください。
    *   SIEM（Security Information and Event Management）システムへのログ連携を強化し、迅速な検知と対応を可能にしてください。
6.  **侵入検知/防御システム (IDS/IPS) の導入と更新**:
    *   ネットワーク層でSSHブルートフォース攻撃や不正なアクティビティを検知・防御できるIDS/IPSソリューションを導入・更新し、最新のシグネチャを適用してください。
7.  **脅威インテリジェンスの活用**:
    *   攻撃元IPアドレス（186.10.86.130）や公開鍵のフィンガープリント、コメント（`mdrfckr`）などの情報を脅威インテリジェンスプラットフォームと照合し、既知の攻撃グループやマルウェアとの関連性を調査することで、より深い洞察と対策に繋げてください。
