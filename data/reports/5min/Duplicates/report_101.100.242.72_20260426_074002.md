## SOC分析レポート

### 1. 攻撃の概要と目的

攻撃元IP `101.100.242.72` から、約20分間にわたるSSHサービスへのブルートフォース攻撃が確認されました。この攻撃では、複数の一般的なユーザー名とパスワードの組み合わせが試行された後、特に `root` ユーザーに対して複数回ログインが成功しています。

ログイン成功後、攻撃者は以下のコマンドを実行し、自身の永続的なアクセス経路を確立しようとしています。
1.  `cd ~; chattr -ia .ssh; lockr -ia .ssh`
    *   `.ssh` ディレクトリの属性変更（immutable/append-only属性の無効化）を試み、改ざん防止機能を回避しようとする意図が見られます。`lockr` コマンドは通常存在しないため、失敗しています。
2.  `cd ~ && rm -rf .ssh && mkdir .ssh && echo "ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr">>.ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~`
    *   既存の `.ssh` ディレクトリを削除し、新規作成した上で、特定の公開鍵（コメント: `mdrfckr`）を `authorized_keys` ファイルに追加しようとしています。これは、パスワード認証を必要とせず、自身の秘密鍵でシステムにアクセスできるバックドアを設置し、永続的なアクセス権を獲得することを目的としています。

このログはハニーポット (Cowrie) 上で観測されたものですが、もしこれが本番環境で発生した場合、攻撃者は `root` 権限でのアクセスを獲得し、システムを完全に制御することが可能になります。

### 2. 推測される手法・使用ツール

*   **手法:**
    *   **偵察（Reconnaissance）:** SSHサービスの公開を特定し、攻撃対象としました。
    *   **ブルートフォース攻撃 / パスワードスプレー攻撃:** 複数の異なるユーザー名とパスワードの組み合わせを自動的に試行し、認証情報を特定しようとしました。特に `root` ユーザーを執拗に狙っています。
    *   **初期アクセス（Initial Access）:** ブルートフォース攻撃により `root` ユーザーのパスワードを特定し、SSH経由でシステムへの不正な初期アクセスを確立しました。
    *   **永続化（Persistence）:** ログイン成功後、自身の公開鍵を `~/.ssh/authorized_keys` に追加することで、将来的にパスワードなしでシステムにアクセスできる永続的なバックドアを確立しようとしました。
    *   **防御回避（Defense Evasion）:** `chattr` コマンドを使用して、ファイルシステムのセキュリティ属性を無効化し、`authorized_keys` ファイルへの変更を容易にしようとしました。
*   **使用ツール:**
    *   **自動化されたブルートフォースツール:** 短期間に多数のログイン試行を行っていることから、Hydra、Nmapの `ssh-brute` スクリプト、Medusa、またはカスタムスクリプトなどの自動化ツールが使用された可能性が高いです。
    *   **標準SSHクライアント:** 接続自体は一般的なSSHクライアントによって行われています。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**

*   **特権アカウント（root）の侵害:** 攻撃者が `root` ユーザーとして複数回ログインに成功しているため、システムに対する完全な制御権を獲得したと判断されます。
*   **永続的なアクセス確立の試み:** 攻撃者が `authorized_keys` に自身の公開鍵を追加しようとしていることは、将来にわたって継続的にシステムへアクセスする意図を示しており、長期的な脅威となります。
*   **影響範囲の広さ:** `root` 権限が奪われた場合、攻撃者はシステム内のあらゆるデータへのアクセス、改ざん、削除、マルウェアのインストール、他のシステムへの横展開など、極めて広範かつ深刻な被害を引き起こす可能性があります。
*   **自動化された攻撃:** 攻撃は自動化されたツールによって実行されている可能性が高く、同様の手法が他のシステムに対しても広く展開される可能性があります。

このログはハニーポットでの観測ですが、もし実環境でこの事象が発生していた場合、即座にインシデント対応が必要となる深刻な侵害に分類されます。

### 4. 推奨アクション

#### 4.1. 緊急対応 (Immediate Actions)

1.  **攻撃元IPのブロック:** Firewall/IDS/IPSにて、攻撃元IP `101.100.242.72` からの全ての通信を即座にブロックしてください。
2.  **本番環境のSSHログの緊急確認:**
    *   本番環境のSSHログ（例: `/var/log/auth.log`）にて、`101.100.242.72` からの接続履歴がないか確認してください。
    *   本ログに記載された成功パスワード（`qwer1234QWER!@#$`, `3245gs5662d34`, `Ni123456`, `ubuntu@123`, `Password.123`, `a12348765`）が、実際に運用中のどのシステムでも使用されていないか緊急で確認してください。
3.  **rootアカウントのパスワードリセット:** もし上記の成功パスワードが本番環境の `root` アカウントで一致した場合、または類似する弱いパスワードを使用している場合は、直ちに全ての特権アカウント（`root`、`admin` など）のパスワードを複雑で推測困難なものにリセットしてください。
4.  **SSH公開鍵の確認と削除:**
    *   全てのLinux/Unixサーバーにおいて、`root` ユーザーの `~/.ssh/authorized_keys` ファイルの内容を確認してください。
    *   本ログに記載された公開鍵 `ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr` が存在しないか確認し、もし存在する場合は即座に削除してください。
    *   また、不明な他の公開鍵も併せて確認し、必要に応じて削除してください。
5.  **侵害調査（Forensic Analysis）の開始:**
    *   万が一、本番環境でログイン成功が確認された場合、当該システムは侵害された可能性が高いです。専門家によるフォレンジック調査を開始し、不正なプロセス、ファイル、ネットワーク接続、データの窃取や改ざん、追加マルウェアの有無などを詳細に調査してください。
    *   侵害されたシステムが他の内部システムに影響を与えていないか、横展開の有無も確認してください。

#### 4.2. 恒久的な対策 (Long-term Measures)

1.  **SSH認証の強化:**
    *   **パスワード認証の無効化:** SSHアクセスにおいて、パスワード認証を完全に無効化し、公開鍵認証のみを許可することを強く推奨します。
    *   **多要素認証 (MFA) の導入:** 可能であれば、SSHログインにMFAを導入し、セキュリティを強化してください。
    *   **root直接ログインの禁止:** `sshd_config` にて `PermitRootLogin no` を設定し、`root` ユーザーの直接ログインを禁止してください。特権操作は一般ユーザーでログイン後、`sudo` を使用して実行するように徹底してください。
2.  **パスワードポリシーの強化と周知:**
    *   複雑性、長さ、有効期限など、より強力なパスワードポリシーをシステム全体で適用してください。
    *   パスワードの定期的な変更と、使い回しの禁止をユーザーに周知徹底してください。
3.  **ブルートフォース対策ツールの導入:**
    *   Fail2banなどのツールを導入し、一定回数以上のSSHログイン失敗があったIPアドレスを自動的にブロックするように設定してください。
4.  **ネットワークアクセス制御の最適化:**
    *   SSHサービスをインターネットに直接公開するのではなく、VPN接続経由でのみアクセスを許可するか、特定の信頼できるIPアドレス範囲からのみアクセスを許可するようFirewallルールを厳格化してください。
    *   ポートノッキングなどの技術導入も検討してください。
5.  **ログ監視体制の強化:**
    *   SSHログイン試行、認証成功、特権コマンド実行などのシステムログを集中管理・監視し、不審なアクティビティをリアルタイムで検知し、アラートを発するシステムを構築してください。
6.  **脆弱性管理とパッチ適用:**
    *   OSやSSHサーバーソフトウェア（OpenSSHなど）の脆弱性情報を常に収集し、最新のセキュリティパッチを迅速に適用してください。
7.  **セキュリティ意識向上トレーニング:**
    *   従業員に対し、フィッシング詐欺、ソーシャルエンジニアリング、強力なパスワードの重要性など、基本的なセキュリティ対策に関する定期的なトレーニングを実施してください。
