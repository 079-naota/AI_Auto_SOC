## SOCアナリストレポート

### 1. 攻撃の概要と目的

このレポートは、攻撃元IPアドレス `190.244.39.232` からのSSHサービスに対する不審な活動について分析したものです。

**攻撃の概要:**
2026年4月26日07:54:05Zから08:35:12Zにかけて、約40分間にわたり、攻撃元IPアドレス `190.244.39.232` からSSHサービスへの連続的なアクセス試行が観測されました。この活動は、複数のユーザー名とパスワードの組み合わせを使用したブルートフォース攻撃から始まり、最終的に特権ユーザーである `root` アカウントでの複数回の認証成功を記録しました。

認証成功後、攻撃者はシステムへの永続的なアクセス経路を確立するため、自身の公開鍵を `~/.ssh/authorized_keys` ファイルに書き込むコマンドを実行しました。また、ファイルの属性を変更しようとするコマンドも試行されましたが、一部はハニーポットの環境で失敗しています。

**攻撃の目的:**
1.  **初期アクセスと認証情報の窃取:** SSHサービスの認証情報を、特に特権ユーザーである `root` のパスワードをブルートフォース攻撃により特定し、システムへの不正アクセスを確立すること。
2.  **永続的なアクセス経路の確保 (Persistence):** ログイン成功後、`~/.ssh/authorized_keys` に攻撃者自身の公開鍵 (`ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`) を追加することで、パスワードを知らなくても再接続できるバックドアを設置すること。
3.  **痕跡隠蔽/設定改ざんの試行:** `chattr -ia .ssh` や存在しない `lockr -ia .ssh` といったコマンドを試行しており、バックドアの隠蔽や、`.ssh` ディレクトリの属性変更による永続化を試みた可能性があります。

### 2. 推測される手法・使用ツール

*   **ブルートフォース/パスワードスプレー攻撃:**
    *   攻撃者は、`odoo`, `user0`, `shiv`, `oracle`, `ovpn`, `windows` などの一般的なユーザー名や、不明な文字列 (`345gs5662d34`) を含む多数のユーザー名とパスワードの組み合わせを繰り返し試行しています。特に `root` ユーザーに対する試行が目立ち、最終的に複数の異なるパスワードでログインに成功しています。
*   **自動化されたスクリプト/ツール:**
    *   ログイン試行の速度、認証成功後のコマンド実行の迅速さ、および一貫性から、この攻撃は手動で行われたものではなく、特定のSSHブルートフォースツール（例: Hydra, NmapのNSEスクリプト, MetasploitのSSHモジュールなど）やカスタムスクリプトによって自動化されている可能性が高いと推測されます。
*   **SSH公開鍵バックドア設置:**
    *   ログイン成功後に `cd ~ && rm -rf .ssh && mkdir .ssh && echo "ssh-rsa ..." >> .ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~` という一連のコマンドを実行しています。これはSSHの公開鍵認証メカニズムを悪用し、`authorized_keys` ファイルに攻撃者自身の公開鍵を追加することで、パスワードなしで特権ユーザーとしてシステムに再アクセスできる永続的なアクセス経路を確立する典型的な手法です。攻撃者の公開鍵には `mdrfckr` という特徴的なコメントが付与されています。
*   **ファイル属性変更の試行:**
    *   `chattr -ia .ssh; lockr -ia .ssh` コマンドの試行は、`.ssh` ディレクトリへの変更を永続化させたり、他のユーザーからの変更を防いだりする目的があったと考えられます。`lockr` コマンドが `command.failed` と記録されていることから、このコマンドはハニーポット環境には存在しないか、正常に実行されませんでした。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**

*   **特権ユーザーの認証成功:** 攻撃者は `root` ユーザーとして複数回認証に成功しています。これが本番環境のシステムであれば、攻撃者はシステムに対する完全な制御権を獲得し、任意の操作（データ窃取、マルウェア展開、システム破壊、他のシステムへの横展開など）が可能になります。
*   **永続化メカニズムの確立試行:** ログイン後にSSH公開鍵を設置しようとしていることは、攻撃者が一時的なアクセスだけでなく、恒久的なシステムへのアクセス経路を確立し、長期的な支配を目論んでいることを明確に示しています。これはMITRE ATT&CKフレームワークにおけるPersistence（永続化）の段階に該当し、攻撃の深刻度を大幅に高めます。
*   **自動化された攻撃の継続性:** 約40分間にわたり執拗に攻撃を継続し、複数のパスワードでログイン成功を記録していることから、組織内の他の露出しているSSHサービスに対しても同様の攻撃が試行されている可能性があり、広範なリスクを示唆します。
*   **攻撃手法の複雑性:** 単純なブルートフォースだけでなく、認証成功後のシステム設定改ざん（SSH公開鍵の設置）まで一連の流れで試行しているため、高度な攻撃スクリプトまたはツールが使用されていると判断できます。

この攻撃は、初期アクセスから特権昇格、そして永続化へと進む典型的なサイバーキルチェーンに沿っており、もし実際のシステムで成功していれば、壊滅的な影響をもたらす可能性があったため、「高」と評価します。

### 4. 推奨アクション

**緊急対応（即時実施を推奨）:**

1.  **攻撃元IPアドレスのブロック:**
    *   直ちに `190.244.39.232` を、境界ファイアウォール、IDS/IPS、または各サーバーのホストベースファイアウォール（iptables/firewalldなど）でブラックリストに登録し、今後の接続を完全にブロックしてください。
2.  **成功したパスワードの確認と変更:**
    *   ログに記録されている `root` ユーザーで成功したパスワード (`135792468`, `abcdefgh`, `Aa112211.`, `Gp123456`, `nPSpP4PBW0`, `3245gs5662d34`) が、**本番環境のシステムで現在使用されていないことを緊急で確認**してください。
    *   もしこれらのパスワードが使用されている、または過去に使用されていた場合は、直ちに当該アカウントのパスワードを非常に複雑で推測されにくいものに変更し、関連する全てのシステムで強制的なパスワードリセットを徹底してください。
3.  **攻撃者のSSH公開鍵の確認と削除:**
    *   ログに記載された攻撃者の公開鍵 (`ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`) のフィンガープリントを生成し、管理対象の全サーバー（特にSSHサービスを提供しているLinuxサーバー）において、`~/.ssh/authorized_keys` ファイル内に一致する公開鍵が存在しないか緊急で確認してください。特に `root` ユーザーや他の特権ユーザーのディレクトリを重点的に調査します。
    *   もし当該公開鍵が発見された場合、直ちにこれを削除してください。そして、そのシステムはすでに侵害されているとみなし、詳細なフォレンジック調査を実施し、システムを安全な状態（必要に応じてバックアップからのリストア、OSの再インストールなど）に復旧させる計画を立ててください。

**恒久的な対策（今後同様の攻撃を防ぐため）:**

1.  **SSHセキュリティ設定の強化:**
    *   **パスワード認証の無効化:** 最も推奨される対策として、可能であればSSHへのパスワード認証を完全に無効化し、より安全な公開鍵認証のみに移行してください。
    *   **Rootログインの禁止:** `/etc/ssh/sshd_config` で `PermitRootLogin no` を設定し、`root` ユーザーの直接ログインを禁止します。管理者ユーザーは一般ユーザーでログイン後、`sudo` を利用するようにします。
    *   **デフォルトポートの変更:** SSHの標準ポート22番を別の高位ポートに変更することで、一般的な自動スキャンからの可視性を低減させます。
    *   **アクセス元IP制限:** 厳密に許可されたIPアドレスからのSSH接続のみを許可するよう、ファイアウォールルールを設定します。
    *   **レートリミット/ブルートフォース対策ツールの導入:** Fail2banなどのツールを導入し、一定回数のログイン失敗があったIPアドレスを自動的にブロックする設定を行います。
    *   **多要素認証(MFA)の導入:** 可能であれば、SSHログインにMFAを導入し、認証セキュリティを強化します。
2.  **強力なパスワードポリシーの強制:**
    *   全システムユーザーに対し、複雑性（大文字、小文字、数字、記号の組み合わせ）、長さ（最低12文字以上）、および推測困難な文字列を含むパスワードの使用を強制するポリシーを確立し、定期的なパスワード変更を促します。
3.  **定期的なセキュリティ監査と脆弱性スキャン:**
    *   定期的にシステムのセキュリティ設定の監査と脆弱性スキャンを実施し、潜在的な脆弱性を特定して修正します。特にSSH設定が推奨されるベストプラクティスに準拠しているかを確認します。
4.  **ログ監視体制の強化:**
    *   本番環境のSSH認証ログ（`/var/log/auth.log` や `/var/log/secure` など）をSIEM（Security Information and Event Management）システムなどに集約し、リアルタイムでの監視と、ブルートフォース攻撃や異常なログイン試行を検知するルールを強化してください。
5.  **インシデント対応計画の見直しと訓練:**
    *   このような攻撃が発生した場合に、迅速かつ効果的に対応できるよう、インシデント対応計画を定期的に見直し、関連チームとの合同訓練を実施してください。
