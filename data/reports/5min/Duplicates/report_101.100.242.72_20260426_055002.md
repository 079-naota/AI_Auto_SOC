### 1. 攻撃の概要と目的

攻撃元IP `101.100.242.72` から、約22分間にわたりSSHサービスへの継続的な攻撃が観測されました。この攻撃は、多様なユーザー名とパスワードの組み合わせを試行するブルートフォース攻撃から始まり、複数回にわたって `root` アカウントへのログインに成功しています。

ログイン成功後、攻撃者は以下の主要な行動を試みています。

1.  **防御メカニズムの無効化試行**: `cd ~; chattr -ia .ssh; lockr -ia .ssh` コマンドを実行し、`.ssh` ディレクトリに設定されている可能性のある保護属性（不変属性など）を解除しようとしています。`lockr` コマンドはシステムに存在しないため失敗していますが、攻撃者が`.ssh`ディレクトリへの変更を妨げる要因を取り除こうとしていたことが伺えます。
2.  **永続的なアクセスの確立**: `cd ~ && rm -rf .ssh && mkdir .ssh && echo "ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O7b...mdrfckr">>.ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~` という一連のコマンドを実行し、自身のSSH公開鍵（コメント部に`mdrfckr`と記載）を `root` ユーザーの `authorized_keys` ファイルに追加しようとしています。これは、将来的にパスワードなしでSSH鍵認証を利用してシステムにアクセスできる永続的なバックドアを確立することを目的としています。

この攻撃の最終的な目的は、SSH経由でシステムへの初期アクセスを獲得し、最も特権の高い `root` アカウントを制御下に置き、永続的なアクセス経路を確保することであると推測されます。

    *   **永続化 (Persistence)**: ログイン後に `.ssh/authorized_keys` ファイルに自身の公開鍵を追加することで、システムへの永続的なアクセス経路を確立しようとしています。これは、攻撃者が長期的にシステムを制御しようとする意図を示しています。
    *   **防御回避 (Defense Evasion)**: `chattr` や `lockr` コマンドを試行することで、システムに実装されている可能性のあるセキュリティ機構を迂回し、鍵の設置を確実にしようとしています。

### 3. 脅威レベルとその理由

*   **脅威レベル**: **高 (High)**
*   **理由**:
    1.  **特権アカウントへのログイン成功**: 攻撃者はHoneypot上とはいえ、システムで最も高い権限を持つ `root` アカウントへのログインに複数回成功しています。本番環境であれば、これはシステム全体に対する完全な制御を意味します。
    2.  **永続化の試行**: 攻撃者が自身のSSH公開鍵を`authorized_keys`に追加しようとしたことは、一度アクセスに成功すれば、その後のアクセスを容易にし、システムへの永続的なバックドアを設置しようとする深刻な意図を示しています。
    3.  **既知の攻撃パターン**: SSHブルートフォース攻撃は一般的な初期アクセス手法であり、それに続く鍵認証による永続化は、攻撃者がシステムを悪用するための典型的なステップです。これが成功した場合、データ窃取、マルウェア感染、他のシステムへの水平展開など、広範囲な被害につながる可能性があります。
    4.  **Honeypotへの攻撃**: 今回のログはCowrie Honeypotで収集されたものであり、実際のシステムが侵害されたわけではありませんが、同様の攻撃が本番環境で観測された場合、極めて深刻なセキュリティ侵害として対処する必要があります。

### 4. 推奨アクション

以下の推奨アクションを実施し、類似の攻撃に対する防御を強化してください。

1.  **攻撃元IPアドレスのブロック**:
    *   直ちにファイアウォールまたはIPS/IDSにて、攻撃元IP `101.100.242.72` からの全ての接続をブロックしてください。

2.  **SSHパスワード認証の強化**:
    *   **多要素認証 (MFA) の導入**: SSHログインに対して多要素認証を必須とすることで、パスワードが漏洩した場合でも不正アクセスを防ぎます。
    *   **強力なパスワードポリシーの適用**: 複雑で推測困難なパスワードを強制し、定期的なパスワード変更を義務付けます。
    *   **`root`ユーザーの直接ログイン禁止**: `root`アカウントのSSH直接ログインを無効にし、一般ユーザーでログイン後、`sudo`コマンドを利用して特権操作を行う運用に変更します。
    *   **ブルートフォース対策**: Fail2Banなどのツールを導入し、SSHログイン失敗回数が多いIPアドレスを自動的にブロックする仕組みを導入・強化します。

3.  **SSH鍵認証の厳格な管理**:
    *   **パスワード認証の無効化**: 可能であればパスワード認証を完全に無効にし、より安全なSSH鍵認証のみを許可します。
    *   **`authorized_keys`ファイルの監査**: 全てのシステムにおいて、`authorized_keys`ファイルの内容を監査し、既知の不正な公開鍵（特に今回の攻撃で使用された`mdrfckr`というコメントを持つ鍵）が含まれていないことを確認し、不正な鍵は削除します。
    *   **ディレクトリ・ファイルのパーミッション設定**: `~/.ssh`ディレクトリのパーミッションを`700` (`drwx------`) に、`authorized_keys`ファイルのパーミッションを`600` (`-rw-------`) に厳格に設定し、他のユーザーからの読み取りや書き込みを禁止します。
    *   **不変属性 (Immutable Attribute) の設定**: `chattr +i ~/.ssh` コマンドを使用して、`.ssh`ディレクトリに不変属性を設定することを検討します。これにより、rootユーザーであっても属性を解除しない限りファイルやディレクトリの変更・削除ができなくなり、攻撃者が鍵を設置するのを困難にします。

4.  **ログ監視とアラート体制の強化**:
    *   SSHログインの成功・失敗イベント、特に`root`アカウントに対するログイン試行や特権コマンドの実行履歴について、リアルタイムでの監視とアラートシステムを導入・強化します。
    *   SIEM（Security Information and Event Management）システムなどを活用し、ログの長期保存と相関分析を行います。

5.  **セキュリティ診断と脆弱性管理**:
    *   SSHサービスを含む全ての公開サービスに対して定期的な脆弱性スキャンを実施し、既知の脆弱性がないかを確認します。
    *   SSHサーバーソフトウェアおよびOSのセキュリティパッチを常に最新の状態に保ちます。

6.  **脅威インテリジェンスの活用**:
    *   今回の攻撃で使用された公開鍵情報（`mdrfckr`）などの脅威インテリジェンスを、組織内のセキュリティ製品（IPS/IDS、SIEMなど）に登録し、監視・検知能力を向上させます。
