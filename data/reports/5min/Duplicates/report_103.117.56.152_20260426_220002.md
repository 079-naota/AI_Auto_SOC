# SOC自動分析レポート: 103.117.56.152

**生成日時:** 2026-04-26 22:01:16

## SOCアナリストレポート

**件名: 2026年04月26日 攻撃元IP 103.117.56.152 からのSSHブルートフォース/永続化攻撃に関する分析レポート**

### 1. 攻撃の概要と目的

2026年4月26日06:54:07から07:36:31までの約40分間にわたり、攻撃元IPアドレス `103.117.56.152` から標的システム（Cowrieハニーポット）に対し、SSHサービスへの組織的なブルートフォース/辞書攻撃が継続的に仕掛けられました。

攻撃者は、`testuser`、`bot`、`postgres`、`user1`、`steam`、`ubuntu`、`wang`、`oracle`といった一般的なユーザー名を含む多様な認証情報を試行しました。複数の試行の末、`root`ユーザーに対して複数の異なるパスワード(`Aa@12345`, `odoo123`, `nPSpP4PBW0`, `xxZZ1234`, `Aa112211.`, `3245gs5662d34`)でログインに成功しました。

ログイン成功後、攻撃者はシステムへの永続的なアクセス経路を確立することを目的として、以下の悪意あるコマンドを実行しました。
1.  `.ssh`ディレクトリの属性を変更しようと試みました (`chattr -ia .ssh; lockr -ia .ssh`。ただし`lockr`は無効なコマンド)。これは、`authorized_keys`ファイルへの書き込みを妨げる可能性のある属性を無効化する意図があったと推測されます。
2.  既存の`.ssh`ディレクトリを削除し、新しい`.ssh`ディレクトリを作成しました。
3.  作成した新しい`.ssh/authorized_keys`ファイルに、自身のSSH公開鍵（コメント: `mdrfckr`）を書き込み、適切なパーミッションを設定しました。これにより、パスワード認証を必要とせず、自身の公開鍵を用いてシステムに再アクセスできるバックドアを設置しようとしました。

この攻撃は、ターゲットシステムのSSHサービスを介して不正なアクセス権を獲得し、そのアクセスを永続化することを明確な目的としていました。

### 2. 推測される手法・使用ツール

*   **SSHブルートフォース/辞書攻撃:**
    ログに記録された大量のログイン失敗試行 (`cowrie.login.failed`) から、攻撃者が自動化されたツールを用いて、一般的なユーザー名とパスワードの組み合わせを網羅的または辞書的に試行していることが明らかです。`Hydra`や`Medusa`、`Nmap`のSSHスクリプトなどが使用された可能性があります。
*   **アカウント侵害 (Credential compromise):**
    `root`アカウントに対するログインが複数回成功しており、攻撃者が強力なパスワードや一般的なパスワード辞書に含まれるパスワードを効率的に特定したことを示しています。
*   **永続化メカニズムの確立:**
    ログイン成功後、`.ssh/authorized_keys`ファイルに攻撃者の公開鍵を追加する一連のコマンド(`rm -rf .ssh && mkdir .ssh && echo "ssh-rsa ..." >> .ssh/authorized_keys && chmod -R go= ~/.ssh`)が実行されています。これは、SSHを介した永続的なバックドアを設置する一般的な手法であり、一度侵入が成功すると、パスワードが変更されたとしても攻撃者はアクセスを維持できます。
*   **自動化された攻撃スクリプト:**
    ログイン成功後のコマンド実行シーケンスが複数回にわたって一貫していることから、攻撃は手動ではなく、あらかじめ用意されたスクリプトによって自動的に実行されている可能性が高いです。

### 3. 脅威レベルとその理由

**脅威レベル: 重大 (Critical)**

**理由:**

1.  **最高権限アカウント (root) の侵害:** 攻撃者はシステム上で最高の権限を持つ`root`アカウントへのログインに成功しています。`root`権限の奪取は、システム全体の制御が攻撃者の手に渡ることを意味し、データの窃取、システム設定の改ざん、マルウェアのインストール、他のシステムへの攻撃の中継点としての利用など、あらゆる種類の深刻な被害をもたらす可能性があります。
2.  **永続化の試み:** 攻撃者は、自身の公開鍵を`authorized_keys`ファイルに書き込むことで、永続的なアクセス経路を確立しようとしました。この手法が成功した場合、システム管理者がパスワードを変更しても、攻撃者は継続的にシステムにログインできる状態となります。
3.  **自動化された広範囲な攻撃:** このログはハニーポット (Cowrie) のものであり、実際の運用システムではありませんが、攻撃が自動化されていることから、同様の脆弱性を持つ実システムが多数狙われている可能性を示唆しています。もし、この攻撃が実際のシステムで成功していた場合、その被害は計り知れません。
4.  **複数のログイン成功:** 短時間に複数回`root`アカウントで異なるパスワードでログインが成功しており、攻撃者が多様な認証情報を利用してログインを試みていることがわかります。

### 4. 推奨アクション

このログはハニーポットのものですが、もし実際のシステムで同様の攻撃が発生した場合を想定し、以下の緊急対応および予防策を強く推奨します。

#### 緊急対応 (Critical Actions):

1.  **攻撃元IPアドレスのブロック:** 攻撃元IPアドレス `103.117.56.152` をファイアウォール、IDS/IPS、またはルーターのACLで即座にブロックし、さらなるアクセスを遮断してください。
2.  **侵害された可能性のあるアカウントのパスワード変更:** ログに記録された`root`ユーザーおよびその他の関連ユーザーのパスワードを直ちに、かつ強力なものに変更してください。
3.  **SSH公開鍵認証情報の監査:** 全てのユーザー（特に`root`）の`~/.ssh/authorized_keys`ファイルの内容を精査し、不正な公開鍵（特にログに記載された`mdrfckr`のコメント付きの鍵）が存在しないか確認し、存在する場合は削除してください。
4.  **システムログの徹底的な監査:** SSH、認証、プロセス実行などのシステムログを詳細に分析し、上記ログに記録された時刻以前や以降で、他の不正なログイン試行、成功、不審なプロセス実行、ファイル変更、ネットワーク通信がないか確認してください。
5.  **システムイメージの取得 (フォレンジック目的):** 侵害の痕跡調査のために、可能であれば現在のシステムの状態のスナップショットまたはフォレンジックイメージを取得してください。

#### 予防策 (Preventive Measures):

1.  **SSH設定の強化:**
    *   **パスワード認証の無効化:** 可能であればSSHのパスワード認証を無効にし、公開鍵認証のみを許可するように設定してください。
    *   **`PermitRootLogin no`:** `root`ユーザーの直接SSHログインを禁止し、必要であれば一般ユーザーでログイン後、`su`または`sudo`を使用する運用にしてください。
    *   **強力なパスワードポリシー:** 全てのシステムアカウントに対し、複雑性、長さ、有効期限を含む強力なパスワードポリシーを適用し、定期的なパスワード変更を義務付けてください。
    *   **ポート変更:** SSHの標準ポート22番を、より高位なポート番号に変更することを検討してください（セキュリティ・バイ・オブスキュリティですが、一般的なスキャンからの検出を軽減できます）。
    *   **`AllowUsers`/`DenyUsers`によるアクセス制御:** SSHへのアクセスを許可するユーザーを明示的に指定し、不要なアカウントからのアクセスを制限してください。
2.  **二要素認証 (MFA) の導入:** SSHアクセスに対し、パスワードまたは公開鍵に加えて、追加の認証要素（例: TOTP、FIDO2キーなど）を導入することを強く推奨します。
3.  **ブルートフォース対策ツールの導入:** `Fail2ban`や`DenyHosts`などのツールを導入し、一定回数のログイン失敗があったIPアドレスを自動的にブロックするように設定してください。
4.  **IDS/IPSの導入と運用:** 侵入検知システム/侵入防御システム (IDS/IPS) を導入し、SSHブルートフォース攻撃やログイン後の異常なコマンド実行を検知・防御できるようにルールを調整してください。
5.  **脆弱性管理の徹底:** システムOS、アプリケーション、ミドルウェアの定期的な脆弱性スキャンとパッチ適用を確実に行い、既知の脆弱性を解消してください。
6.  **ログ監視とSIEMの活用:** ログを一元的に収集・分析するSIEM (Security Information and Event Management) を導入し、リアルタイムで不審な活動を検知し、アラートを発する体制を構築してください。
