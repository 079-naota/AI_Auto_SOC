# SOC自動分析レポート: 171.104.143.176

## SOC分析レポート

### 1. 攻撃の概要と目的

攻撃元IPアドレス `171.104.143.176` から、対象システムへのSSHサービスに対するブルートフォース/辞書攻撃が確認されました。最初の試行は一般的なユーザー名/パスワード (`deploy/admin`) で失敗しましたが、その後、`root` ユーザーで異なる2つのパスワード (`1234ASDF`, `QWEasd123456`) を用いて計2回ログインに成功しています。

ログイン成功後、攻撃者は以下の目的で一連のコマンドを実行しました。

*   **永続的アクセスの確立**: `root` ユーザーのSSH公開鍵認証によるバックドア (`.ssh/authorized_keys` への自身の公開鍵追加) を設置しようとしました。これにより、パスワードなしで将来的にアクセスすることを試みています。
*   **独占的アクセスの確保**: `root` ユーザーのパスワードを複数回変更し、他のユーザー（または他の攻撃者）がアクセスできないようにし、自身のアクセスを排他的に確保しようとしました。また、他の攻撃者による痕跡の削除 (`/tmp/secure.sh`, `/tmp/auth.sh` の削除とプロセス停止) や、システムへのアクセス制限の緩和 (`/etc/hosts.deny` のクリア) も試みています。
*   **システム情報の収集 (偵察)**: CPU情報、メモリ、OSバージョン、ディスク使用量、cronジョブ、実行中のプロセス、システムアーキテクチャなど、広範囲なシステム情報を収集しています。これは、後の攻撃フェーズ（マルウェア展開、横展開、データ窃取など）に備えた偵察活動と考えられます。

これらの行動から、攻撃者はシステムの完全な掌握と、それを利用したさらなる悪意のある活動（例: 仮想通貨マイニングマルウェアの展開、C2サーバーとの通信確立、ランサムウェア展開、データ窃取、ボットネットへの組み込みなど）を企図していると推測されます。

### 2. 推測される手法・使用ツール

*   **認証情報攻撃 (Credential Stuffing / Brute-force / Dictionary Attack)**: SSHサービスに対して、一般的なユーザー名と推測しやすいパスワードの組み合わせを試行しました。

    *   **パスワード変更**: `chpasswd` コマンドを使用して `root` ユーザーのパスワードを何度も変更し、他のアクセス経路を無効化しようとしました。
*   **偵察 (Reconnaissance) 手法**:
    *   標準的なLinuxコマンド (`cat /proc/cpuinfo`, `free -m`, `crontab -l`, `w`, `uname -a`, `top`, `lscpu`, `whoami`, `df -h`, `which ls`, `ls -lh`) を用いて、システムのリソース、OS情報、ユーザー、ディスク使用状況などを広範囲に調査しました。
*   **防御回避 (Defense Evasion) / 影響 (Impact)**:
    *   他の攻撃者による可能性のあるスクリプトやプロセス (`/tmp/secure.sh`, `/tmp/auth.sh`) を削除・停止し、システムの掌握を排他的に進めようとしました。
    *   `/etc/hosts.deny` をクリアすることで、IPベースのアクセス制限を無効化し、将来のアクセスを容易にしようとしました。
*   **使用ツール**: ログに特定のマルウェアのダウンロードや実行は確認されていませんが、一連のコマンドは自動化されたシェルスクリプトによって実行されている可能性が高いです。`lockr` コマンドが試行されましたが、これは一般的に存在しないコマンドであり、ハニーポット環境では実行に失敗しています。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**

1.  **`root` 権限での侵入**: システムの最高権限である `root` ユーザーでのログインに成功しており、攻撃者はシステムに対するあらゆる操作（ファイルの読み書き、実行、設定変更、ユーザー追加・削除など）が可能になりました。これはシステムの機密性、完全性、可用性に対する最大級の脅威となります。
2.  **永続化の試み**: SSH公開鍵の設置や `root` パスワードの変更は、攻撃者がシステムへの永続的なアクセス経路を確保しようとしている明確な証拠です。これにより、現在のセッションが終了しても、攻撃者はいつでも再侵入できる状態を作り出そうとしました。
3.  **広範な偵察活動**: システムリソースや設定に関する詳細な情報収集は、攻撃者が次にどのような行動をとるかを計画するための準備段階です。これにより、システムを最大限に悪用するための計画が立てられている可能性が高いです。
4.  **競合排除の行動**: 他の攻撃者や防御メカニズムの排除を試みる行動は、システムを完全に掌握し、自身の支配下に置こうとする強い意図を示しています。

これらの理由から、この攻撃はシステムに対して非常に高いリスクをもたらすものであり、早急な対応が求められます。

### 4. 推奨アクション

#### 4.1. 緊急対応 (Immediate Actions)

1.  **システム隔離**: ログインが成功したシステム（ハニーポット環境が模倣する実システム）を直ちにネットワークから隔離し、これ以上の被害拡大や横展開を防ぎます。
2.  **攻撃元IPのブロック**: 攻撃元IPアドレス `171.104.143.176` からのアクセスをファイアウォールやIDS/IPSでブロックリストに追加します。
3.  **SSH認証情報の変更**:
    *   `root` ユーザーを含む全てのユーザーのパスワードを直ちに、かつ複雑なものに強制変更します。
    *   `~/.ssh/authorized_keys` ファイルの内容を確認し、ログに記録された攻撃者の公開鍵 (`ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`) を含む全ての不正な鍵を削除します。正当な鍵も一度すべて削除し、再登録を検討します。
4.  **バックドアの確認と除去**:
    *   `crontab -l` や `/etc/systemd/system/`, `/etc/rc.d/` などのスタートアップスクリプト、その他の永続化メカニズムが設定されていないか確認し、不正なものを削除します。
    *   `/etc/hosts.deny` ファイルがクリアされていないか確認し、必要に応じてセキュリティポリシーに基づいた設定に戻します。
5.  **システムスナップショットの取得**: フォレンジック調査のため、現在のシステムの状態（ディスクイメージやメモリダンプ）を保全します。

#### 4.2. フォレンジック調査 (Forensic Investigation)

1.  **ログ分析の深化**:
    *   Cowrieログだけでなく、実際のシステムログ（`/var/log/auth.log`、`/var/log/secure`、Webサーバーログなど）を詳細に調査し、攻撃の全容（初期侵入経路、実行コマンド、アクセスファイルなど）を特定します。
    *   不正なログインに使用されたパスワードが、他のシステムやサービスで使い回されていないか確認します。
2.  **侵害範囲の特定**: 攻撃者が他のシステムへの横展開を試みた形跡がないか、システム内部の通信ログなどを確認します。
3.  **マルウェアの有無確認**: 不正なファイルやプロセス（特にマイニング関連や、C2通信を行うバックドアツール）がシステム内に存在しないか、セキュリティツールや手動でスキャン・確認します。
4.  **ユーザーアカウントの監査**: 不審な新規ユーザーアカウントが作成されていないか、既存アカウントの権限が不正に変更されていないかを確認します。

#### 4.3. 再発防止策 (Preventive Measures)

1.  **SSH設定の強化**:
    *   `sshd_config` で `PermitRootLogin no` を設定し、`root` ユーザーでの直接ログインを禁止します。
    *   `PasswordAuthentication no` を設定し、パスワード認証を無効化し、公開鍵認証のみを許可します。
    *   公開鍵認証を使用する場合は、強力なパスフレーズ付きのSSH鍵を生成・使用し、定期的に鍵を更新します。
2.  **認証強化**: 多要素認証 (MFA) の導入を検討し、SSHログインのセキュリティを向上させます。
3.  **不正ログイン対策ツールの導入**: Fail2Banなどのツールを導入し、一定回数以上のログイン失敗があった場合に攻撃元IPを自動的にブロックする設定を行います。
4.  **セキュリティパッチの適用**: OSやSSHサーバーを含む全てのソフトウェアを最新の状態に保ち、既知の脆弱性を悪用されるリスクを低減します。
5.  **定期的なセキュリティ監査**:
    *   不要なユーザーアカウントやサービスを削除し、最小権限の原則を適用します。
    *   SSH公開鍵認証で利用されている鍵を定期的に監査し、不正な鍵が追加されていないか確認します。
6.  **監視体制の強化**:
    *   SSHログインの成功・失敗、不審なコマンド実行、`authorized_keys` ファイルへの変更、`/etc/passwd` や `/etc/shadow` の変更など、重要なシステムイベントに対するログ監視とアラート設定を強化します。
    *   EDR (Endpoint Detection and Response) ソリューションの導入を検討します。
