# SOC自動分析レポート: 103.172.205.139

## SOCアナリストレポート

### 1. 攻撃の概要と目的

攻撃元IPアドレス `103.172.205.139` から、ターゲットシステムに対する継続的なSSHブルートフォース攻撃が確認されました。攻撃者は、`root` ユーザーを含む複数のユーザー名に対して様々なパスワードを試行し、合計で5回の `root` ユーザーログインに成功しています。

ログイン成功後、攻撃者は以下の目的のために定型的なコマンドを実行しようとしています。
*   **永続的なアクセス経路の確立**: `~/.ssh/authorized_keys` ファイルに自身のSSH公開鍵を追加することで、パスワードなしでシステムに再接続できるバックドアを設置しようとしています。
*   **バックドアの隠蔽・保護の試み**: 追加した `authorized_keys` ファイルが容易に削除・変更されないよう、`chattr` コマンドや `lockr` コマンド（ハニーポット環境では失敗）を用いてファイル属性の操作を試みています。

この攻撃は、ターゲットシステムへの不正なアクセス権を取得し、そのアクセスを永続化させることを目的とした、一般的なSSHブルートフォースおよびバックドア設置の試行と判断されます。

### 2. 推測される手法・使用ツール

1.  **SSHブルートフォース攻撃（辞書攻撃）**:
    *   複数のユーザー名（`user7`, `deployer`, `frappe`, `ubuntu`, `user`, `mary`, `admin`, `zhaoxj`, `345gs5662d34`）とパスワードの組み合わせを短時間に多数試行しています。
    *   特に `root` ユーザーに対しては、「`abcd1234.`」「`Password.123`」「`MoeClub.org`」「`a12348765`」「`abc@123456`」といったパスワードでログインに成功しており、これらは一般的に利用される弱いパスワードや辞書攻撃リストに含まれる可能性が高いものです。
    *   これらの試行は、自動化されたスクリプトやブルートフォースツール（例: Hydra, Nmapの`ssh-brute`スクリプトなど）によって行われている可能性が極めて高いです。

2.  **SSH公開鍵認証を利用したバックドア設置**:
    *   ログイン成功後、攻撃者は以下のコマンドを複数回実行しています。

3.  **ファイル属性変更の試み**:
    *   `cd ~; chattr -ia .ssh; lockr -ia .ssh` というコマンドも実行されています。
    *   `chattr` はLinuxファイルシステムで不変属性（Immutable）や追記専用属性（Append-only）を設定するために使用されます。攻撃者はこれらの属性を設定することで、`authorized_keys` ファイルがシステム管理者によって削除されたり、内容が変更されたりするのを防ごうとした可能性があります。`lockr` は一般的なコマンドではありませんが、何らかのロック機構を試みたものと推測されます。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**

*   **特権アカウントへのログイン成功**: 攻撃者はシステム上で最も高い権限を持つ `root` ユーザーとして、合計5回にわたってログインに成功しています。これは、実際のシステムであれば完全なシステム侵害につながる重大な事態です。
*   **永続的なアクセスの確立試行**: ログイン後にバックドアとなるSSH公開鍵を設置しようとしていることは、単なる一時的な侵入ではなく、将来にわたってシステムへのアクセスを維持する明確な意図があることを示しています。これにより、検知されにくい持続的な脅威となる可能性があります。
*   **ブルートフォース攻撃の継続性**: 攻撃が一時的なものではなく、異なる認証情報を次々と試行し、ログイン成功後も定型的なマルウェア（SSH公開鍵の設置）を実行しようとしていることから、自動化された攻撃フレームワークが稼働していると判断されます。このような攻撃元は、他の脆弱なシステムを探し続けている可能性が高いです。
*   **複数の認証情報侵害の可能性**: ログからは `root` ユーザーの複数の異なるパスワードでのログイン成功が確認されており、もしこれらのパスワードが他のシステムでも使用されている場合、そのシステムも同様に危険にさらされている可能性があります。

### 4. 推奨アクション

1.  **緊急対応（即時実施）**:
    *   **攻撃元IPのブロック**: 攻撃元IPアドレス `103.172.205.139` からのすべての通信を、ファイアウォールまたはIDS/IPSで直ちにブロックリストに追加し、アクセスを遮断してください。
    *   **漏洩パスワードの変更**: ログに記録されている、ログインに成功した全てのユーザー名とパスワード（特に `root` ユーザーのパスワード `abcd1234.`, `Password.123`, `MoeClub.org`, `a12348765`, `abc@123456`）を、強力でユニークなものに直ちに変更してください。他のシステムでこれらのパスワードを流用している場合は、それらも全て変更してください。
    *   **不正なSSH公開鍵の削除**: ログに示された公開鍵（`ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK4rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`）が本番環境の `~/.ssh/authorized_keys` に存在しないか確認し、万が一存在する場合は直ちに削除してください。

2.  **SSHサービス強化**:
    *   **rootログインの無効化**: `/etc/ssh/sshd_config` で `PermitRootLogin no` を設定し、`root` ユーザーのSSH直接ログインを禁止してください。特権操作が必要な場合は、一般ユーザーでログイン後 `sudo` を使用するようにしてください。
    *   **パスワード認証の無効化**: 可能であれば、`PasswordAuthentication no` を設定し、公開鍵認証のみを許可するようにしてください。
    *   **強力な認証情報と多要素認証 (MFA) の導入**: 全てのSSHユーザーに対して、強力で推測されにくいパスワードの使用を義務付け、可能であれば多要素認証 (MFA) を導入してください。
    *   **レートリミットの設定**: `fail2ban` などのツールを導入し、SSHログイン試行回数にレートリミットを設け、ブルートフォース攻撃を自動的に検知・ブロックするメカニズムを確立してください。
    *   **デフォルトポートの変更**: SSHのデフォルトポート（TCP 22）を非標準のポートに変更することを検討してください。これにより、一般的なポートスキャンからの攻撃を減らすことができます。

3.  **システム監視と監査**:
    *   **ログ監視の強化**: SSH認証ログ、コマンド履歴、システムログなどを継続的に監視し、不審なログイン試行やコマンド実行を早期に検知できるよう、SIEM (Security Information and Event Management) システムへのログ連携とアラート設定を強化してください。
    *   **脆弱性診断**: システム全体に対して定期的な脆弱性スキャンを実施し、OS、ミドルウェア、アプリケーションの脆弱性を特定し、速やかにパッチを適用してください。
    *   **セキュリティ監査**: 定期的にシステム設定のセキュリティ監査を実施し、ベストプラクティスが適用されていることを確認してください。特に、不要なアカウントの削除、最小権限の原則の適用、不適切なファイルパーミッションの修正などを行ってください。

4.  **インシデントレスポンス計画の確認**:
    *   今後のセキュリティインシデントに備え、インシデントレスポンス計画が適切に機能するかを確認し、必要に応じて訓練を実施してください。
