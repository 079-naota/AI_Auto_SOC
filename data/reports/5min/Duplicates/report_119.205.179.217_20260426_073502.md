# SOC自動分析レポート: 119.205.179.217

**生成日時:** 2026-04-26 07:37:47

## SOCアナリストレポート

### 1. 攻撃の概要と目的

攻撃元IPアドレス `119.205.179.217` からのSSHサービスに対するブルートフォース攻撃が確認されました。この攻撃者は、様々なユーザー名とパスワードの組み合わせを試行し、最終的に`root`アカウントでのログインに複数回成功しています。ログイン成功後、攻撃者は公開鍵認証による持続的なアクセス経路を確立することを目的とし、以下の不正な活動を試みていました。

**攻撃のタイムラインの要約:**
*   **初期フェーズ (05:44Z - 05:49Z):** `kamil`, `an`, `ubuntu`などの一般的なユーザー名と推測されやすいパスワードを用いたSSHログイン試行。全て失敗。
*   **成功フェーズ (05:50Z, 05:52Z, 05:53Z, 05:58Z, 06:00Z):**
    *   `root`ユーザーに対して、複数の異なるパスワード（`qazwsxedc123456`, `3245gs5662d34`, `Aa112211.`, `dlgwbn`, `3edc4rfv#`, `Qazwsx999!`）を用いてログインに成功。
    *   ログイン成功後、`root`ユーザーのホームディレクトリに不正なSSH公開鍵（コメント: `mdrfckr`）を設置しようとするコマンドを実行。具体的には、既存の`.ssh`ディレクトリを削除し、新たなディレクトリを作成後、攻撃者の公開鍵を`authorized_keys`ファイルに追加し、適切なパーミッションを設定する一連の操作を行いました。
    *   `chattr -ia .ssh; lockr -ia .ssh`コマンドによるファイル属性の変更も試みられましたが、これはハニーポット環境では失敗しています。
*   **中間失敗フェーズ (05:51Z, 05:54Z - 05:57Z, 05:59Z):** `bitrix`, `ubuntu`, `b`, `testadmin`, `user`など、様々なユーザー名に対するログイン試行が継続的に失敗。

この攻撃の主目的は、初期の認証突破に続き、公開鍵認証による永続的なバックドアを確立し、将来的な無制限なアクセスを確保することであると判断されます。

### 2. 推測される手法・使用ツール

*   **手法:**
    *   **SSH ブルートフォース攻撃 / 辞書攻撃:** 既知のユーザー名や一般的なユーザー名（`root`, `ubuntu`など）と、辞書に含まれる可能性のあるパスワードを組み合わせた総当たり攻撃が用いられています。比較的短時間に多数の認証試行が行われていることから、自動化されたツールが使用されている可能性が高いです。
    *   **永続化 (Persistence):** ログイン成功後、ユーザーのSSH設定ディレクトリ（`~/.ssh`）を操作し、攻撃者の公開鍵を`authorized_keys`ファイルに追加することで、以降はパスワード認証なしでSSH接続できる状態を作り出そうとしています。これは、システムの再起動やパスワード変更後もアクセスを維持するための典型的な手法です。
    *   **特権昇格 (Privilege Escalation) の意図:** `root`ユーザーへのログインに執拗に成功しようとしており、実際に複数回成功していることから、システムに対する完全な制御権の獲得を意図していることが伺えます。
*   **使用ツール:**
    *   ログからは特定のツール名を断定できませんが、SSHブルートフォース攻撃には [Hydra](https://github.com/vanhauser-thc/thc-hydra) や [Medusa](https://github.com/joxean/medusa) などの自動化ツールが一般的に使用されます。複数のセッションがほぼ同時に開始され、異なるユーザー名/パスワードの組み合わせが試行されていることから、スクリプトまたは専用のブルートフォースツールが活用されている可能性が高いです。
    *   ログイン後のコマンド実行には、標準的なUnix/Linuxコマンド（`cd`, `rm`, `mkdir`, `echo`, `chmod`, `chattr`）が使用されています。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**
1.  **`root`アカウントへの複数回のログイン成功:** ハニーポット環境でのイベントであるとはいえ、攻撃者はシステムの最高権限である`root`アカウントに対するログイン認証を複数回突破しています。これは、現実のシステムであれば深刻な侵害を意味し、システムへの完全な制御を許すことになります。
2.  **永続的なアクセス経路の確立試行:** ログイン成功後、`authorized_keys`ファイルに攻撃者の公開鍵を追加しようとする明確な意図が確認されています。これにより、攻撃者はシステムのバックドアを設置し、将来にわたって認証なしでアクセスを継続する能力を得ようとしていました。これは、検出・対応を困難にし、長期的な脅威をもたらす可能性があります。
3.  **組織的な攻撃の可能性:** 使用されている複数の異なるパスワードが、特定の辞書やリストに基づくものであること、また一連のブルートフォースから公開鍵設置までの手順が自動化されていることから、個人の試みというよりは、より組織的または広範なスキャン活動の一部である可能性が考えられます。
4.  **攻撃手法の悪質性:** ブルートフォース攻撃は一般的な手法ですが、`root`ユーザーに対する成功と永続化の試みは、その後のマルウェア感染、データ窃取、ボットネットへの組み込みなど、さらに深刻な二次攻撃につながる高いリスクを伴います。

### 4. 推奨アクション

この攻撃はハニーポットで観測されたものですが、同様の攻撃が実環境のシステムに対して行われる可能性を考慮し、以下の緊急対応および予防的措置を推奨します。

**緊急対応 (Immediate Actions):**
*   **攻撃元IPのブロック:** 攻撃元IPアドレス `119.205.179.217` からのすべての通信を、境界ファイアウォールやIDS/IPSで即座にブロックリストに追加してください。
*   **関連システムの確認:** 社内/組織内でSSHサービスを公開しているシステムがあれば、緊急で以下の項目を確認してください。
    *   `119.205.179.217` からのSSH接続履歴の有無。
    *   `root`ユーザーによる不正なログイン成功履歴の有無。
    *   提供された攻撃者の公開鍵 (`ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`) と一致する、または類似の不正な公開鍵が`~/.ssh/authorized_keys`に設置されていないか確認。もし発見された場合は、直ちに削除し、該当システムを隔離し詳細なフォレンジック調査を実施してください。
    *   上記のログイン成功パスワードが他のシステムで使用されていないか確認し、使用されている場合は即座に変更を指示してください。

**予防的措置 (Preventive Measures):**
*   **SSHセキュリティの強化:**
    *   **パスワード認証の無効化:** 可能であれば、パスワード認証を無効にし、公開鍵認証のみを許可するようにSSHサービスを設定してください。
    *   **`root`ユーザーの直接ログイン禁止:** `/etc/ssh/sshd_config`内で `PermitRootLogin no` を設定し、`root`ユーザーの直接ログインを禁止してください。必要に応じて一般ユーザーでログイン後、`su`または`sudo`で昇格する運用を徹底してください。
    *   **強固なパスワードポリシーの強制:** 全てのユーザーアカウントに対して、複雑性、長さ、有効期限などを含む厳格なパスワードポリシーを適用してください。
    *   **非標準ポートの使用:** SSHのデフォルトポートである22番ポートを非標準のポートに変更することを検討してください。これにより、一般的なスキャンからの可視性を低減できます。
    *   **ホワイトリストによるアクセス制御:** ファイアウォールやセキュリティグループを設定し、信頼できる特定のIPアドレスからのSSHアクセスのみを許可するホワイトリスト方式を導入してください。
    *   **ブルートフォース対策ツールの導入:** Fail2banなどのツールを導入し、一定回数のログイン失敗があったIPアドレスを自動的にブロックする仕組みを導入してください。
    *   **多要素認証 (MFA) の導入:** SSH接続にMFAを導入することで、認証情報の漏洩によるリスクを大幅に低減できます。
*   **ログ監視とアラート:**
    *   SSHログインログ（成功・失敗の両方）をセキュリティ情報イベント管理（SIEM）システムに集約し、リアルタイムでの監視と異常検知アラートを設定してください。特に`root`ユーザーへのログイン成功、短時間での多数のログイン失敗、未知のIPアドレスからの接続に対してアラートを設定することが重要です。
*   **セキュリティ教育:** ユーザーに対して、強固なパスワードの重要性やフィッシング詐欺などの脅威に関する定期的なセキュリティ教育を実施してください。

これらの推奨事項を実施することで、同様の攻撃に対するシステムの耐性を向上させ、将来的な侵害のリスクを低減することができます。
