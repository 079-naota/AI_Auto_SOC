**SOCアナリストレポート**

**件名: SSHサービスに対するブルートフォース攻撃と`root`権限奪取、および永続化の試みに関する分析報告**

### 1. 攻撃の概要と目的

攻撃元IPアドレス `103.117.56.152` から、当社のSSHサービスに対し、継続的なブルートフォース攻撃が確認されました。この攻撃は、2026-04-26T06:54:07Zから少なくとも07:36:31Zまでのおよそ40分間にわたり行われました。

攻撃者は、`testuser`, `bot`, `postgres`, `user1`, `steam`, `ubuntu`, `wang`, `oracle`といった一般的なユーザー名を含む多様なユーザー名とパスワードの組み合わせを繰り返し試行しました。

ログ分析の結果、攻撃者は以下のパスワードを使用して`root`アカウントへのログインに複数回成功しています。
*   `Aa@12345` (07:28:18Z)
*   `odoo123` (07:31:22Z)
*   `nPSpP4PBW0` (07:33:26Z)
*   `xxZZ1234` (07:34:25Z)
*   `Aa112211.` (07:36:30Z)

さらに、これらのログイン成功時とは別に、`root`ユーザーに対してパスワード`3245gs5662d34`でのログインも4回成功していることが確認されました。

ログイン成功後、攻撃者は以下のコマンドを連続して実行し、システムへの永続的なアクセス経路の確立を試みています。

```bash

cd ~ && rm -rf .ssh && mkdir .ssh && echo "ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr">>.ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~
```

これらのコマンドの主な目的は、`root`ユーザーのホームディレクトリにある既存のSSH設定（`.ssh`ディレクトリ）を削除し、攻撃者自身の公開鍵（コメントに`mdrfckr`と記載されたもの）を`authorized_keys`ファイルに追加することで、パスワードなしでいつでもシステムにアクセスできるバックドアを設置することです。また、`chattr -ia .ssh`はファイル属性の変更不可フラグを解除しようとする試みであり、永続化の強固さを高める意図がうかがえます（`lockr -ia .ssh`コマンドは実行に失敗しています）。

### 2. 推測される手法・使用ツール

*   **侵入手法**:
    *   **パスワードリスト攻撃 / ブルートフォース攻撃**: 短時間で多種多様なユーザー名とパスワードの組み合わせを試行しており、これは自動化された辞書攻撃またはブルートフォース攻撃の典型的な挙動です。
    *   **アカウント乗っ取り (Account Takeover)**: `root`ユーザーの認証情報を不正に取得し、ログインに成功しています。
    *   **永続化 (Persistence)**: ログイン成功後、速やかにSSH公開鍵認証を利用した永続的なアクセス経路を確立しようとしています。これは、`~/.ssh/authorized_keys`ファイルへの攻撃者自身の公開鍵の追加、および関連するパーミッション変更によって行われます。
*   **使用ツール**:
    *   **自動化されたSSHクライアント**: 大量のログイン試行と、ログイン成功後の定型コマンド実行から、一般的なSSHクライアントを基盤とし、それを自動化するスクリプトまたは専用のブルートフォースツール（例: Hydra, NmapのNSEスクリプト、Medusaなど）が使用された可能性が高いです。
    *   **汎用シェルコマンド**: `cd`, `rm`, `mkdir`, `echo`, `chmod`, `chattr`などのLinux標準コマンドを組み合わせて、`authorized_keys`ファイルの操作を行っています。`lockr`コマンドは標準コマンドではなく、実行に失敗していますが、ファイルの属性変更やロックに関連する意図があったと推測されます。

### 3. 脅威レベルとその理由

*   **脅威レベル**: **高 (High)**

*   **理由**:
    1.  **最高権限アカウントの侵害**: システム上で最も強力な権限を持つ`root`アカウントへのログインが複数回成功しており、攻撃者は対象システムに対する完全な制御権を掌握したと見なされます。これは最も深刻なセキュリティ侵害の一つです。
    2.  **永続的なアクセス経路の確立**: 攻撃者はログイン成功後、直ちに自身の公開鍵を`authorized_keys`に登録し、パスワード認証なしでSSH経由でいつでもシステムにアクセスできる状態を作り出しています。これにより、検知や対処が遅れた場合、攻撃者がシステムに常駐し、さらなる攻撃活動を展開するリスクが極めて高まります。
    3.  **複数の異なる成功パスワード**: 異なるパスワードで`root`アカウントへのログインが複数回成功しており、これは攻撃者が非常に広範囲なパスワードリストを持っているか、または対象システムに複数の脆弱なパスワードが存在することを示唆します。
    4.  **活動の自動化**: 攻撃活動が自動化されているため、特定の脆弱性が修正されない限り、繰り返し攻撃が行われる可能性が高いです。

### 4. 推奨アクション

この攻撃は深刻なセキュリティ侵害を示唆しており、迅速かつ徹底した対応が必要です。

1.  **緊急対応 (Immediate Actions)**:
    *   **対象システムの緊急停止またはネットワーク隔離**: 攻撃者によるさらなる操作、データ窃取、または他のシステムへの横展開を防ぐため、直ちに影響を受けたシステムをネットワークから隔離するか、緊急停止してください。
    *   **SSHサービスの停止またはポート変更**: SSHサービスを一時的に無効化するか、標準ポート（22番）から、すぐに特定が困難な高位ポートに変更してください。
    *   **`root`ユーザーのパスワードリセット**: ログに記載されている`root`ユーザーのログイン成功パスワード（`Aa@12345`, `odoo123`, `nPSpP4PBW0`, `xxZZ1234`, `Aa112211.`, `3245gs5662d34`）は全て漏洩している可能性が高いため、直ちに非常に強力で推測困難な新しいパスワードに変更してください。
    *   **`authorized_keys`ファイルの精査と不正鍵の削除**: `root`ユーザーのホームディレクトリ（`~/.ssh/authorized_keys`）の内容を確認し、ログに記載された攻撃者の公開鍵（コメント`mdrfckr`の鍵）および、認識できない全ての鍵を削除してください。必要であれば、`.ssh`ディレクトリ全体を削除し、適切なパーミッションで再作成してください。
    *   **その他ユーザーアカウントの確認**: ログに登場した他のユーザー名（`testuser`, `bot`, `postgres`など）がシステム内に存在する場合は、これらのアカウントのパスワードも脆弱である可能性があるため、パスワードのリセットとポリシー強化を検討してください。
    *   **ログの保全**: 現在の攻撃ログを含む全ての関連ログ（システムログ、認証ログ、ネットワークデバイスログなど）を、将来のフォレンジック調査のために保全してください。

2.  **フォレンジック調査 (Forensic Investigation)**:
    *   **侵入経路の特定**: どのパスワードが最初に侵害され、どのように漏洩したのかを徹底的に調査してください。
    *   **侵害範囲の特定**: 攻撃者がログイン後にどのような操作を行ったか（他のユーザーの作成、マルウェアの配置、設定ファイルの改ざん、データの窃取、他のシステムへの接続など）を詳細に分析し、システム全体の健全性を評価してください。
    *   **永続化メカニズムの確認**: `authorized_keys`以外の方法（cronジョブ、システムサービス、スタートアップスクリプト、rootkitなど）でバックドアが仕掛けられていないか、徹底的に検査してください。
    *   **通信先の分析**: 攻撃元IPアドレスからの通信だけでなく、攻撃者がログイン後に外部のC2（Command and Control）サーバーなどと通信していないか、ネットワークログを詳細に調査してください。

3.  **再発防止策 (Preventive Measures)**:
    *   **パスワードポリシーの厳格化**: 全てのシステムアカウントに対し、複雑性、長さ、有効期限などを含む厳格なパスワードポリシーを適用し、定期的なパスワード変更を義務付けてください。辞書攻撃で一般的に使用されるパスワードの利用を禁止するシステムを導入してください。
    *   **SSH認証の強化**:
        *   **公開鍵認証への移行**: パスワード認証を無効化し、公開鍵認証のみを許可することを強く推奨します。
        *   **多要素認証 (MFA) の導入**: 可能な場合、SSHログインに多要素認証を導入し、認証セキュリティを大幅に向上させてください。
        *   **`root`ログインの制限**: SSH経由での`root`ユーザーの直接ログインを禁止し、一般ユーザーでログイン後、`sudo`や`su`で昇格する運用に変更してください（`PermitRootLogin no`）。
    *   **ファイアウォールによるアクセス制限**: SSHアクセスを、特定の信頼できるIPアドレスまたはIPアドレス範囲に限定するファイアウォールルールを適用してください。
    *   **侵入検知/防御システム (IDS/IPS) の導入と強化**: 不審なログイン試行パターン（短時間での多数の失敗など）を検知し、自動的にブロックできるIDS/IPSソリューションを導入、または既存のシステムのルールを強化してください。
    *   **ログ監視の強化**: 認証ログやシステムログをリアルタイムで監視し、異常なログイン試行やログイン後の不審なコマンド実行を即座に検知できる体制を構築してください。
    *   **脆弱性管理とパッチ適用**: 定期的にシステムの脆弱性スキャンを実施し、OSやアプリケーションのセキュリティパッチを最新の状態に保ってください。
    *   **Cowrie Honeypotの活用**: このログがCowrie Honeypotからのものである場合、Honeypotが正しく機能していることを確認し、収集された脅威インテリジェンス（攻撃元IP、ユーザー名、パスワード、攻撃コマンド、公開鍵など）を、他のセキュリティシステム（ファイアウォール、IDS/IPS、SIEMなど）に連携し、防御能力向上に役立ててください。
