## SOCアナリストレポート：SSHブルートフォース攻撃およびバックドア設置の試み

### 1. 攻撃の概要と目的

攻撃元IPアドレス `102.219.126.124` から、システムに対するSSHブルートフォース攻撃が継続的に行われました。この攻撃の主な目的は、SSHサービスの脆弱な認証情報を特定し、特に特権ユーザーである`root`アカウントへのログインを成功させること、そしてログインに成功した際には、自身のSSH公開鍵をシステムに登録することで、持続的なアクセス（バックドア）を確立することにあります。

### 2. 推測される手法・使用ツール

*   **手法:**
    *   **SSH ブルートフォース攻撃 (辞書攻撃):** `drcom`, `root`, `ubuntu`, `user`などの一般的なユーザー名と、`drcom123`, `1qaz@WSX3edc$RFV!@`, `ali123456`, `Root8`, `Aa112211.`, `Config123`, `Aa123321`といった様々なパスワードを組み合わせて、繰り返しSSHログインを試行しています。特に`root`ユーザーが執拗に狙われ、複数のパスワードでログイン成功しています。
    *   **持続的アクセス確立のためのSSH鍵インジェクション:** ログイン成功後、以下のコマンド群を自動的に実行しています。
        *   `cd ~; chattr -ia .ssh; lockr -ia .ssh`: `.ssh`ディレクトリのファイル属性（immutable/append-only）を解除しようとしています。これは、`authorized_keys`ファイルの書き込みを阻害する可能性のある保護を無効化する試みです。`lockr`は標準的なLinuxコマンドではないため、ハニーポット上では失敗しています。
        *   `cd ~ && rm -rf .ssh && mkdir .ssh && echo "..." >> .ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~`: 既存の`.ssh`ディレクトリを削除し、新規作成した上で、自身のSSH公開鍵を`authorized_keys`ファイルに書き込み、適切なパーミッションを設定しています。
*   **使用ツール:**
    *   ログの一貫したパターンと自動的なコマンド実行から、攻撃者は何らかの自動化されたスクリプト、ボットネットの一部、またはSSHブルートフォースツール（例: Hydra、Medusa、Nmapの`ssh-brute`スクリプトなど）を使用していると推測されます。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**

*   **root権限でのログイン成功:** 攻撃者はシステムの最高権限を持つ`root`ユーザーとして複数回ログインに成功しており、これは実際の環境であれば完全なシステム侵害を意味します。これにより、攻撃者はシステムのあらゆる操作（データの閲覧、改ざん、削除、マルウェアの実行、新たなユーザーの作成など）が可能になります。
*   **持続的アクセス確保の試み:** ログイン成功後、すぐにSSH公開鍵を挿入することで、恒久的なバックドアを確立しようとしています。これは、一時的な侵入だけでなく、将来にわたるシステムへのアクセスを確保しようとする明確な意図を示しており、非常に悪質です。
*   **自動化された攻撃:** 攻撃が繰り返し、連続的に行われていることから、自動化された攻撃であることが明らかです。このような攻撃は、ターゲットシステムが多数存在する場合に、広範囲に被害を及ぼす可能性があります。
*   **脆弱なパスワードの利用:** 複数の異なるパスワードで`root`ログインが成功していることから、標的システム（この場合はハニーポット）が脆弱なパスワードを使用しているという脆弱性を突かれていることを示します。

### 4. 推奨アクション

このログはハニーポットのものであるため、実際のシステムへの直接的な被害は発生していませんが、もし実際のシステムで同様のログが確認された場合、以下の緊急かつ重要なアクションを迅速に実施する必要があります。

1.  **攻撃元IPアドレスの即時ブロック:**
    *   ファイアウォール、ACL、IPS/IDS等のセキュリティデバイスを用いて、攻撃元IPアドレス `102.219.126.124` からのSSHポート（デフォルトは22/TCP）へのアクセスを即座にブロックしてください。
2.  **侵害された可能性のある認証情報の変更:**
    *   ログに記載されているログイン成功パスワード（`1qaz@WSX3edc$RFV!@`, `3245gs5662d34`, `ali123456`, `Root8`, `Aa112211.`, `Config123`, `Aa123321`など）が、システム内のどのユーザーアカウントでも使用されていないことを確認し、もし存在する場合は即座に**すべて**強力なパスワードに変更してください。特に`root`アカウントのパスワードは最優先で変更が必要です。
3.  **SSH公開鍵認証情報の確認と削除:**
    *   `root`ユーザーの`~/.ssh/authorized_keys`ファイルに、攻撃者の公開鍵 (`ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`) が存在しないことを確認してください。もし存在する場合は、即座に該当する行を削除し、ファイルのパーミッションが適切であること（`~/.ssh`は`700`、`authorized_keys`は`600`）を確認してください。
4.  **SSHパスワード認証の無効化と公開鍵認証への移行:**
    *   可能であれば、SSHデーモンの設定（`/etc/ssh/sshd_config`）で`PasswordAuthentication no`を設定し、パスワード認証を無効化し、より安全な公開鍵認証のみの使用を強制することを強く推奨します。
5.  **アカウントの棚卸しとロックアウト設定:**
    *   不要なユーザーアカウント（特に`drcom`, `ubuntu`, `user`など、ログイン試行されたアカウント）は削除または無効化してください。
    *   SSHサービスに`fail2ban`などのツールを導入し、短期間のログイン失敗回数に応じてIPアドレスを自動的にブロックする設定を適用してください。
6.  **パスワードポリシーの強化:**
    *   全ユーザーに対して、複雑性、長さ、有効期限などを含む強力なパスワードポリシーを強制し、定期的なパスワード変更を促してください。
7.  **システム全体のマルウェアスキャンと脆弱性診断:**
    *   もしログイン成功のログが実際のシステムで発生した場合は、システムがマルウェアに感染していないか、他にバックドアが仕掛けられていないかを確認するため、包括的なマルウェアスキャンと脆弱性診断を実施してください。
8.  **ログ監視体制の強化:**
    *   SSHログイン試行や認証イベントに関するログを継続的に監視し、異常なアクセスパターンや予期せぬログインイベントを早期に検知できる体制を強化してください。SIEM（Security Information and Event Management）システムへのログ連携を検討してください。
