# SOC自動分析レポート: 116.99.174.17

**生成日時:** 2026-04-27 06:16:41

---

## SOC分析レポート

### 1. 攻撃の概要と目的

**概要:**
2026年4月27日 02:13:34Zから02:31:53Zにかけて、攻撃元IPアドレス `116.99.174.17` からターゲットシステムに対し、SSHサービスへのブルートフォース攻撃（または辞書攻撃）が実行されました。この攻撃では、複数の異なるユーザー名とパスワードの組み合わせが試行され、最終的に `root` ユーザーで複数回ログインに成功しています。

**目的:**
攻撃者の主な目的は、SSHサービスへの不正アクセスを通じてシステムの制御を奪取することであると推測されます。特に、システム上で最も高い権限を持つ `root` アカウントでのログインを狙っており、これはシステムへの完全な侵入と、その後の悪意のある活動（例: データ窃取、マルウェア展開、システム破壊、他のシステムへの攻撃基点化など）を企図している可能性が高いことを示唆しています。ログイン成功後に`direct-tcpip`イベントが発生していることから、ポートフォワーディングなどを利用して、さらに内部ネットワークへのアクセスを試みた可能性も考えられます。

### 2. 推測される手法・使用ツール

**手法:**
*   **ブルートフォース攻撃 / 辞書攻撃**: 短時間のうちに多様なユーザー名とパスワードの組み合わせを繰り返し試行していることから、自動化されたスクリプトまたはツールによるブルートフォース攻撃、もしくは辞書攻撃が用いられたと考えられます。
*   **ターゲットアカウントの選定**: `root` だけでなく、`admin`, `user`, `ubnt`, `config`, `squid`, `support`, `guest`, `1234`, `nikita`, `ftpuser` など、よく使われるデフォルトアカウントや一般的なアカウント名を網羅的に試行しています。
*   **パスワードリストの利用**: 試行されたパスワードには、`admin`, `user`, `1234`, `password`, `default`, `alpine`, `12345` といった、推測しやすい単純な文字列や既知のデフォルトパスワードが含まれており、一般的なパスワードリストを利用している可能性が高いです。

**使用ツール:**
ログからは特定のツール名を断定できませんが、このような自動化されたブルートフォース攻撃を実行するために、`Hydra`, `Medusa`, `Nmap`のSSH関連スクリプト、または攻撃者が独自に開発したスクリプトなどが使用された可能性が高いです。ログイン後の`direct-tcpip`イベントは、ログイン後にマルウェアのダウンロードやコマンド＆コントロール（C2）通信のためのシェルを確立しようとした兆候である可能性があります。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**
1.  **複数回のログイン成功**: 攻撃者は `root` ユーザーとして複数回（`root/admin`, `root/@`, `root/root123`, `root/12345`, `root/abcd1234`, `root/ipscan`, `root/alpine`）ログインに成功しています。これはハニーポットのログですが、もしこれが実際のシステムであれば、システムが完全に侵害された状態を意味します。
2.  **特権アカウント (root) へのアクセス**: 攻撃者はシステム上で最高の権限を持つ `root` アカウントのクレデンシャルを特定し、アクセスしています。これにより、システムに対するあらゆる操作が可能となります。
3.  **広範な試行**: 多数のユーザー名とパスワードを体系的に試行しており、特定の脆弱性を狙うのではなく、総当たり的に侵入経路を探す執拗な攻撃意図が見られます。
4.  **侵害後の活動の兆候**: ログイン成功後に`direct-tcpip`というイベントが複数回発生しており、これは攻撃者がリモートポートフォワーディングやローカルポートフォワーディングを確立し、さらなる内部ネットワークへの足がかりを得ようとしたり、データを転送したり、マルウェアをダウンロード・実行したりする試みである可能性が高いです。

### 4. 推奨アクション

**緊急対応（即時実施）:**
1.  **攻撃元IPアドレスのブロック**: 攻撃元IPアドレス `116.99.174.17` をファイアウォール、またはIDS/IPSで即座にブロックし、これ以上のSSH接続試行を阻止してください。
2.  **SSHサービスのセキュリティ強化**:
    *   `root`ユーザーでの直接ログインを無効化する (`PermitRootLogin no`)。
    *   パスワード認証を無効化し、公開鍵認証のみを許可する。
    *   SSHのデフォルトポート22番以外のポートを使用する（セキュリティの強化ではなく、スキャン回避のため）。
    *   `Fail2Ban`のようなブルートフォース対策ツールを導入し、不正なログイン試行を自動的に検知・ブロックする。
    *   許可されたIPアドレスからの接続のみを許可するよう、アクセス制御リスト（ACL）を設定する。
3.  **既存アカウントのパスワード再設定と多要素認証 (MFA) の導入**:
    *   ログに示されたログイン成功クレデンシャル (`root/admin`, `root/@`, `root/root123`, `root/12345`, `root/abcd1234`, `root/ipscan`, `root/alpine`) を含む、すべてのシステムアカウントのパスワードを直ちに複雑なものへ変更する。
    *   可能であれば、SSHログインに多要素認証（MFA）を導入する。
4.  **システム侵害の確認（もし実システムの場合）**:
    *   不審なプロセス、設定変更、ファイル作成、ネットワーク接続がないか、システム全体の整合性チェックとマルウェアスキャンを実施する。
    *   SSH以外のサービス（ウェブサービス、データベースなど）についても、不審なアクセスがないかログを確認する。

**長期的な対策:**
1.  **強力なパスワードポリシーの徹底**: すべてのユーザーに対して、複雑なパスワードを強制し、定期的な変更を義務付ける。
2.  **ログ監視の強化**: SSHログインログやシステムログをSIEM（Security Information and Event Management）などの集中ログ管理システムに連携し、リアルタイムでの異常検知とアラート体制を構築する。特に、連続したログイン失敗、不明なIPアドレスからのログイン、特権ユーザーのログインなどを監視対象とする。
3.  **脆弱性管理とパッチ適用**: 定期的にシステムの脆弱性スキャンを実施し、OSやアプリケーションのセキュリティパッチを最新の状態に保つ。
4.  **ネットワークセグメンテーション**: 重要なシステムを隔離し、必要最小限のアクセスのみを許可するようネットワークセグメンテーションを見直す。
5.  **脅威インテリジェンスの活用**: 今回の攻撃元IPアドレスや試行されたクレデンシャル情報を脅威インテリジェンスとして記録し、今後の攻撃検知や防御に活用する。
