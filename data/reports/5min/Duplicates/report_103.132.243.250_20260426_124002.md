### 2. 推測される手法・使用ツール

*   **手法**:

    *   **永続的アクセス確立**: 認証成功後、SSH公開鍵を`~/.ssh/authorized_keys`に設置することで、パスワード認証以外の永続的なアクセス経路（SSH鍵認証によるバックドア）を確立しようとしています。
*   **使用ツール**:
    *   一連のログイン試行とコマンド実行が非常に迅速かつ定型的に繰り返されていることから、攻撃は自動化されたスクリプトまたはツールによって実行された可能性が高いです。具体的なツールとしては、SSHブルートフォース攻撃ツール（例: `Hydra`, `Medusa`など）や、カスタムで作成されたスクリプトが考えられます。

### 3. 脅威レベルとその理由

**脅威レベル: 緊急 (Critical)**

**理由**:
1.  **最高権限 (root) での認証成功**: 攻撃者はシステム上で最も高い権限を持つ`root`アカウントで複数回ログインに成功しています。これは、もしこれが実システムであれば、システム全体の完全な制御権が奪われた状態に等しく、データの窃取、改ざん、システムの破壊、あるいは他のシステムへの攻撃の踏み台化など、あらゆる悪意ある行為が可能になります。
2.  **永続的アクセスの確立試行**: 攻撃者は単一のセッションでのアクセスに留まらず、SSH公開鍵を設置することで、パスワード変更などの対策が講じられても継続的にシステムへアクセスできるバックドアを確立しようとしました。これは持続的な脅威となります。
3.  **攻撃の自動化と執拗さ**: 短時間に多数のログイン試行とコマンド実行が繰り返されており、攻撃が自動化されていることを示唆します。このような自動化された攻撃は、特定ターゲットだけでなく広範囲なシステムを標的としており、防御側が迅速に対応しなければ継続的にシステムを狙い続ける可能性があります。
4.  **複数の脆弱なrootパスワード**: `root`ユーザーに対し、複数の異なる脆弱なパスワードでログイン成功していることは、実環境であれば複数の管理アカウントに脆弱なパスワードが設定されている可能性を示唆し、極めて危険な状態です。

**A. 緊急対応（即時実施）**

1.  **攻撃元IPアドレスのブロック**: ファイアウォール、IDS/IPS、またはセキュリティグループ等で、攻撃元IPアドレス `103.132.243.250` からの全てのSSH接続を即座にブロックしてください。
2.  **SSH公開鍵の緊急確認と削除**:
    *   全SSHサーバーにおいて、全ユーザー（特に`root`）のホームディレクトリ下の`~/.ssh/authorized_keys`ファイルを確認し、ログに記載された攻撃者の公開鍵（コメントが `mdrfckr` のもの）が存在しないことを確認してください。
    *   万一発見された場合は、直ちに該当する公開鍵を`authorized_keys`ファイルから削除してください。
3.  **rootパスワードの即時変更**: ログに記載された認証成功パスワード（`135792468`, `3245gs5662d34`, `abcdefgh`, `qazwsxedc123`, `P@ssw0rd6`, `Admin88888888`, `Gp123456`）を含む、全ての`root`ユーザーパスワードを、予測困難で複雑なものに直ちに変更してください。
    *   これらのパスワードが他のシステムやサービスで使い回されていないかを確認し、もし使用されている場合は同様に直ちに変更してください。

**B. 予防とセキュリティ強化策**

1.  **SSH設定の強化**:
    *   **`PermitRootLogin no` の設定**: `/etc/ssh/sshd_config` ファイルで `PermitRootLogin no` を設定し、`root`ユーザーでの直接ログインを禁止してください。管理者アクセスは、一般ユーザーでログイン後、`sudo`コマンドを使用して権限昇格を行う運用に切り替えてください。
    *   **`PasswordAuthentication no` の設定**: 可能な場合は`PasswordAuthentication no`を設定し、SSH公開鍵認証のみを許可してください（これには十分な準備とテストが必要です）。
    *   **多要素認証 (MFA) の導入**: SSHアクセスにMFAを義務付け、セキュリティを強化してください。
    *   **SSHポートの変更**: 標準の22番ポートではなく、使用するSSHポートを非標準のポートに変更することを検討してください（これは基本的な防御策であり、高度な攻撃には効果が薄い可能性があります）。
2.  **ブルートフォース攻撃対策の導入/強化**:
    *   `Fail2Ban`などの侵入検知・防止システムを導入し、一定回数以上の認証失敗があったIPアドレスを自動的にブロックする仕組みを構築してください。
3.  **パスワードポリシーの厳格化**:
    *   システム全体で、複雑性、長さ、定期的な変更を義務付ける厳格なパスワードポリシーを適用してください。辞書攻撃に弱いパスワードの使用を禁止するルールを導入してください。
4.  **ログ監視とアラート体制の強化**:
    *   SSH認証成功、認証失敗（特に`root`ユーザー）、`authorized_keys`ファイルへの変更、および不正なコマンド実行などのイベントについて、ログ監視システムでリアルタイムに監視し、異常が検知された際にはSOCに即座にアラートが上がるよう設定してください。
5.  **定期的なセキュリティ監査と脆弱性管理**:
    *   システムの脆弱性スキャンとセキュリティ監査を定期的に実施し、未パッチの脆弱性がないかを確認し、速やかに修正してください。

**C. フォレンジック調査（実システムでの侵入を仮定する場合）**

1.  **感染範囲の特定**: 攻撃者が他のシステムへ横展開を試みていないか、またはこのシステムを踏み台として利用していないか、ネットワークトラフィックログや他のサーバーログなどを横断的に調査してください。
2.  **マルウェア感染の有無**: 不正な実行ファイルやスクリプトが設置されていないか、システムのファイル整合性チェック（FIM: File Integrity Monitoring）を実施し、マルウェア感染の有無を詳細に調査してください。
3.  **追加のバックドア確認**: SSH公開鍵以外にも、Webシェル、スケジュールタスク、サービス、ユーザーアカウントなど、他の永続的アクセス経路が設置されていないか広範に調査してください。
4.  **データ漏洩の確認**: 機密データのアクセスログや転送ログを確認し、データ漏洩の形跡がないか調査してください。
