## SOC分析レポート

### 1. 攻撃の概要と目的

一連の攻撃は自動化されたスクリプトまたはツールによって実行されており、ログイン成功のたびに永続化の試みを繰り返していることが確認できます。

### 2. 推測される手法・使用ツール

*   **ブルートフォース攻撃/辞書攻撃**:
    *   `steam`, `tunnel`, `cloud`, `gitlab` などの一般的なユーザー名や、特権ユーザーである `root` に対するパスワードの総当たり攻撃または辞書攻撃が確認されました。
    *   `root` ユーザーでは `QAZwsx`, `3245gs5662d34`, `Root1234567@@`, `1q2w3e..`, `root123123!!`, `@a123456`, `Root11111111@#` と、複数の異なるパスワードでログイン成功しています。これは、大規模なパスワードリストを用いた自動化された攻撃であることを強く示唆しています。
    *   ログイン成功後にも `345gs5662d34` というユーザー名でのログイン失敗が繰り返し記録されており、これは攻撃ツールが次の段階に進むための別の認証情報（あるいはデフォルト認証情報）を試している可能性があります。
*   **永続化 (Persistence)**:
    *   `cd ~ && rm -rf .ssh && mkdir .ssh && echo "ssh-rsa ... mdrfckr">>.ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~`
        *   これは、既存の`.ssh`ディレクトリを削除し、新しいディレクトリを作成した上で、攻撃者の公開鍵を`authorized_keys`ファイルに書き込むことで、SSH公開鍵認証によるバックドアを確立する典型的な手法です。`chmod -R go= ~/.ssh`は、`.ssh`ディレクトリのパーミッションを適切に設定し、公開鍵認証が機能するようにする目的と推測されます。
*   **防御回避の試み (Defense Evasion)**:
    *   `cd ~; chattr -ia .ssh; lockr -ia .ssh`
        *   `chattr -ia .ssh`: `.ssh`ディレクトリのファイル属性（特に不変属性`i`や追加のみ属性`a`）を解除しようとしています。これは、システム管理者がセキュリティ強化のために設定している属性を無効化し、`authorized_keys`ファイルの改変を可能にしようとする試みです。
        *   `lockr -ia .ssh`: `lockr`は一般的なLinuxコマンドではありません。これは攻撃者が誤入力したか、特定のマルウェアやカスタムツールの一部として使用される非標準コマンドである可能性があります。ログでは `cowrie.command.failed` と記録されており、このコマンドは実行に失敗しています。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**

1.  **root権限の奪取**: 攻撃者はシステム上で最も高い権限を持つ `root` ユーザーとして、複数回にわたりログインに成功しています。これにより、システムへの完全な制御権を取得したと見なされます。
2.  **永続化の試み**: 攻撃者はログイン成功後、速やかにSSH公開鍵を追加することで、長期的なバックドアの確立を試みています。これにより、パスワードが変更されたとしても、攻撃者はこの公開鍵と対応する秘密鍵を用いることで、いつでもシステムに再侵入できる状態を作り出そうとしました。
3.  **自動化された広範囲な攻撃**: ログから、攻撃は自動化されたツールによって、複数のユーザー名やパスワードを試行する広範囲なスキャン・攻撃の一部であることが示唆されます。このような攻撃者は、脆弱なシステムを発見し次第、迅速に支配下に置こうとします。
4.  **ハニーポットへの攻撃である点**: 今回のログはCowrieハニーポットのものであるため、実際のプロダクションシステムが侵害されたわけではありません。しかし、この攻撃パターンが本番環境のSSHサービスに向けられた場合、深刻な情報漏洩、システム破壊、マルウェア感染、他のシステムへの踏み台利用など、甚大な被害につながる可能性が極めて高いことを意味します。

### 4. 推奨アクション

#### 緊急対応 (Immediate Actions):

1.  **攻撃元IPのブロック**: 攻撃元IPアドレス `182.70.243.207` を組織のファイアウォール、IPS/IDS、CDN、WAFなどのセキュリティデバイスで即座にブロックリストに追加し、すべての通信を遮断してください。
2.  **SSH認証情報の変更と強化**:
    *   ログに記録された `root` ユーザーの成功パスワード（`QAZwsx`, `3245gs5662d34`, `Root1234567@@`, `1q2w3e..`, `root123123!!`, `@a123456`, `Root11111111@#`）を含む、すべての`root`アカウントのパスワードを、複雑で予測不能なものに直ちに変更してください。
    *   可能であれば、`root`ユーザーでのSSH直接ログインを無効化し、一般ユーザーでログイン後、`sudo`などを用いて権限昇格する運用に切り替えてください。
3.  **既存SSH公開鍵の確認と削除**:
    *   すべてのサーバーにおいて、`root`ユーザーの`.ssh/authorized_keys`ファイルを含め、すべてのユーザーの`.ssh/authorized_keys`ファイルを点検し、正規の鍵以外に、攻撃者が追加しようとした鍵（`ssh-rsa ... mdrfckr`）が存在しないか確認してください。万一存在した場合は、直ちに削除してください。
    *   正規の鍵についても、必要性や正当性を再確認し、定期的な鍵のローテーションを検討してください。
4.  **システム侵害の有無調査**:
    *   このハニーポットログの攻撃パターンが本番環境でも発生していないか、他のシステムログ（SSH認証ログ、認証ログ、コマンド履歴ログなど）を徹底的に調査してください。
    *   特に、攻撃者が公開鍵を追加した後に、他の不正なアクティビティ（マルウェアのダウンロード、他のシステムへの横展開、データ窃取など）がないか、システムの状態を詳細にフォレンジック調査してください。

#### 予防的対策 (Preventive Measures):

1.  **SSHパスワード認証の無効化**:
    *   SSHサービスにおけるパスワード認証を無効化し、多要素認証（MFA）を組み合わせた鍵認証のみに限定することを強く推奨します。
2.  **多要素認証 (MFA) の導入**:
    *   すべての外部公開SSHサービスに対して、MFAを導入し、認証セキュリティを大幅に強化してください。
3.  **厳格なパスワードポリシーの適用**:
    *   OSアカウントおよびアプリケーションアカウントに対して、十分な長さ、複雑性、有効期限を含む強固なパスワードポリシーを適用し、定期的なパスワード変更を義務付けてください。
4.  **SSHアクセスの制限**:
    *   SSHサービスへのアクセスは、信頼できるソースIPアドレス（例: オフィスネットワーク、VPN経由のIPなど）からのみに限定するよう、ファイアウォールルールを設定してください。
    *   可能であれば、SSHのデフォルトポート22番から別の高ポートに変更することも検討してください（セキュリティの"あいまいさ"に過ぎませんが、自動化されたスキャンからのヒット数を減らせる可能性があります）。
5.  **ログ監視と異常検知の強化**:
    *   SSHログインの成功/失敗試行、特に特権ユーザーへの試行に対するリアルタイムの監視とアラート機能を強化してください。
    *   Fail2banなどのツールを導入し、一定回数以上のログイン失敗があったIPアドレスを自動的にブロックする仕組みを導入してください。
6.  **脆弱性管理とパッチ適用**:
    *   すべてのOSおよびインストールされているアプリケーションに対し、最新のセキュリティパッチを適用し、既知の脆弱性を解消してください。
7.  **不要なアカウントの棚卸しと削除**:
    *   システム上の未使用または不要なユーザーアカウントを定期的に棚卸しし、削除してください。
8.  **セキュリティ意識向上トレーニング**:
    *   従業員に対し、フィッシング詐欺やソーシャルエンジニアリング攻撃に対する意識向上トレーニングを定期的に実施し、機密情報の取り扱いに関するセキュリティベストプラクティスを徹底してください。
9.  **脅威インテリジェンスの活用**:
    *   攻撃元IPアドレスを脅威インテリジェンスプラットフォームと照合し、既知の悪性IPアドレスリストに含まれているか、他の攻撃活動に関連しているかを確認してください。
