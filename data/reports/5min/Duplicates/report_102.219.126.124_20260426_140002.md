### 1. 攻撃の概要と目的

攻撃元IP `102.219.126.124` から、対象SSHサービスに対して繰り返し不正なログイン試行（ブルートフォース攻撃/辞書攻撃）が観測されました。
攻撃者は複数のユーザー名とパスワードの組み合わせを試行し、最終的に`root`ユーザーアカウントを含む複数のアカウントでログインに成功しています。

ログイン成功後、攻撃者はシステムへの永続的なアクセス経路を確立するため、以下のマルウェア的なコマンドを実行しようとしています。

1.  既存の`.ssh`ディレクトリの属性変更解除を試みる (`chattr -ia .ssh; lockr -ia .ssh`)
2.  `.ssh`ディレクトリを削除後、再作成し、攻撃者自身の公開鍵 (`ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`) を`authorized_keys`ファイルに追加し、パーミッションを設定する (`cd ~ && rm -rf .ssh && mkdir .ssh && echo "..." >>.ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~`)

攻撃の目的は、SSHサービスへの不正な初期アクセスを獲得し、その後、自身の公開鍵を登録することでシステムへの永続的なバックドアを確立することであると推測されます。

### 2. 推測される手法・使用ツール

*   **手法**:
    *   **初期アクセス (Initial Access)**: SSHブルートフォース攻撃 / 辞書攻撃。一般的なユーザー名 (`root`, `ubuntu`, `user`, `drcom`) や、複雑な組み合わせを含むパスワードを複数回試行しています。
    *   **永続化 (Persistence)**: ログイン成功後、`~/.ssh/authorized_keys`に攻撃者の公開鍵を挿入することで、パスワードなしでSSH接続できる裏口を確立しようとしています。これにより、パスワードが変更された後もシステムへのアクセスを維持する意図があります。
*   **使用ツール**:
    *   ブルートフォース攻撃には、HydraやMedusaのようなSSH辞書攻撃ツールが使用されている可能性が高いです。
    *   ログイン成功後のコマンド実行は、自動化されたスクリプトやボットネットの一部として実行されている可能性が考えられます。
    *   `lockr`コマンドは標準的なLinuxコマンドではないため、攻撃者の環境に固有のツールであるか、単なる誤入力の可能性もあります。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由**:

*   **不正ログインの成功**: ログには`cowrie.login.success`が複数回記録されており、特にシステムに対する最高権限を持つ`root`アカウントでのログイン成功が頻繁に見られます。これは、対象システムが攻撃者によって実質的に侵害された可能性を示唆します。
*   **永続化の明確な意図**: ログイン成功後、攻撃者が自身の公開鍵を`authorized_keys`ファイルに追加しようとしていることは、将来的なアクセス経路を確保し、永続的なシステム侵害を意図していることを明確に示しています。これにより、一度パスワードを変更しただけではアクセスを排除できないリスクが生じます。
*   **Root権限の悪用**: `root`ユーザーとしてログインしているため、もし本番環境であれば攻撃者はシステムに対する完全な制御権を取得し、任意のコマンド実行、データ窃取、マルウェア展開、他のシステムへの横展開などが可能になります。
*   **公開鍵の存在**: 挿入しようとした公開鍵のコメント部分に「`mdrfckr`」という特徴的な文字列が含まれており、特定の攻撃グループやキャンペーンに関連している可能性があります。

### 4. 推奨アクション

このログはハニーポット (Cowrie) 上のものであると認識していますが、もし本番環境で同様の事象が発生した場合、以下の緊急対応およびセキュリティ強化策を講じる必要があります。

**緊急対応 (即時実施)**:

1.  **攻撃元IPのブロック**: 攻撃元IP `102.219.126.124` をファイアウォール、IDS/IPSで即座にブロックし、アクセスを遮断します。
2.  **侵害されたアカウントのパスワード変更**: ログインに成功した全てのユーザーアカウント（特に`root`）のパスワードを緊急で変更し、推測されにくい強力なパスワードを設定します。
3.  **SSH鍵認証の確認とクリーンアップ**:
    *   `root`ユーザーの`~/.ssh/authorized_keys`ファイルを含め、全てのユーザーの`.ssh/authorized_keys`ファイルを調査し、攻撃者の公開鍵 (`ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`) が追加されていないか確認し、発見した場合は直ちに削除します。
    *   `.ssh`ディレクトリや`authorized_keys`ファイルのパーミッションが適切であるか確認します (`chmod 700 ~/.ssh`, `chmod 600 ~/.ssh/authorized_keys`など)。
4.  **フォレンジック調査の実施**: 侵害された可能性のあるシステムについて、より広範なフォレンジック調査を実施し、攻撃者が他に何を行ったか（マルウェアのインストール、データ窃取、他のシステムへの横展開、ログの改ざんなど）を確認します。
5.  **SSH設定の強化**:
    *   可能であれば、SSHのパスワード認証を無効化し、鍵認証のみに切り替えます。
    *   `PermitRootLogin no`を設定し、`root`ユーザーのSSH直接ログインを無効化します。
    *   SSHサービスを標準ポート (22) 以外に変更する（セキュリティ・バイ・オブスキュリティだが、自動化された攻撃の頻度を減らすのに役立ちます）。

**予防策とセキュリティ強化 (中長期実施)**:

1.  **多要素認証 (MFA) の導入**: SSHログインにMFAを導入し、セキュリティを強化します。
2.  **不正ログイン対策ツールの導入**: Fail2banなどのツールを導入し、SSHの不正ログイン試行を検知し、自動的にブロックするように設定します。
3.  **強力なパスワードポリシーの徹底**: 全てのシステムユーザーに対して、複雑性と定期的な変更を義務付ける強力なパスワードポリシーを適用します。
4.  **不要なアカウントの削除**: 使用されていないアカウントは削除し、最小権限の原則を徹底します。
5.  **監視体制の強化**: SSHログインログ (`/var/log/auth.log`など) やコマンド実行履歴を常時監視し、異常なアクセスや挙動を早期に検知できる体制を構築します。SIEMとの連携も検討します。
6.  **定期的な脆弱性診断とセキュリティ監査**: 定期的にシステムの脆弱性診断やセキュリティ監査を実施し、潜在的な弱点を特定・修正します。
7.  **公開鍵の管理**: 組織内で使用されるSSH公開鍵を厳格に管理し、不正な鍵の挿入がないか定期的に監査します。
