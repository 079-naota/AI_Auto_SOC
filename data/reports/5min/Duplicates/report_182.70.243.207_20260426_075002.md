### 1. 攻撃の概要と目的

攻撃元IPアドレス `182.70.243.207` から、ターゲットシステム（Cowrieハニーポット）のSSHサービスに対し、複数の段階にわたる攻撃が試みられました。

**攻撃の概要**:
攻撃者は、まず多種多様なユーザー名とパスワードの組み合わせを用いてSSHログインを試行するブルートフォース/辞書攻撃を実施しました。特に`root`ユーザーに対する複数のパスワードが成功しています（ハニーポット上での擬似的な成功）。ログインに成功すると、攻撃者は`~/.ssh/authorized_keys`ファイルに自身のSSH公開鍵を書き込むことで、永続的なバックドアを確立しようとしました。また、その前段として`.ssh`ディレクトリのファイル属性を変更し、鍵の書き込みを確実に行おうとする試みも確認されています。

**攻撃の目的**:
SSHサービスへの不正アクセスを確立し、パスワード認証をバイパスできるSSH鍵ベースの永続的なバックドアを仕掛けることで、将来にわたって自由にシステムにアクセスできる環境を構築すること。これにより、システムのリソースを悪用したり、さらなる攻撃の足がかりとすることを目的としていると考えられます。

### 2. 推測される手法・使用ツール

*   **ブルートフォース/辞書攻撃**: `steam`, `tunnel`, `cloud`, `gitlab`などの一般的なユーザー名やサービス名を試行し、特に`root`ユーザーに対して「QAZwsx」「3245gs5662d34」「Root1234567@@」「1q2w3e..」「root123123!!」「@a123456」「Root11111111@#」といった複数のパスワードを繰り返し試しています。これは自動化されたパスワード推測ツール（例: Hydra、NmapのSSHスクリプトなど）を使用している可能性が高いです。
*   **永続化メカニズム (Persistence)**: ログイン成功後、以下のコマンドを実行しようとしています。
    *   `cd ~ && rm -rf .ssh && mkdir .ssh && echo "ssh-rsa AAAAB3NzaC1yc2EAAAA.... mdrfckr">>.ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~`
    このコマンドは、ユーザーのホームディレクトリにある既存の`.ssh`ディレクトリを削除し、新しい`.ssh`ディレクトリを作成し、攻撃者のSSH公開鍵を`authorized_keys`ファイルに書き込むことで、パスワードなしでSSHアクセスできるバックドアを確立しようとするものです。
*   **防御回避 (Defense Evasion)**:
    *   `cd ~; chattr -ia .ssh; lockr -ia .ssh`
    このコマンドは、`.ssh`ディレクトリの不変属性（immutable attribute）を解除しようとするものです。`chattr -ia`はLinuxの拡張属性を操作するコマンドであり、これによりファイルやディレクトリの変更保護を無効化し、後続の`rm -rf .ssh`が確実に実行されるようにする意図があったと考えられます。`lockr`は標準的なLinuxコマンドではないため`cowrie.command.failed`となっていますが、同様の保護解除を意図していると推測されます。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由**:
*   **積極的な不正アクセス試行**: 攻撃者は広範なユーザー名とパスワードの組み合わせを試しており、明確な不正アクセス意図が認められます。
*   **複数のログイン成功**: ハニーポット環境ではありますが、`root`ユーザーに対して複数のパスワードでログインに「成功」している記録が多数あります。これは、実環境で脆弱なパスワードが設定されている場合、実際にシステムが侵害されるリスクが極めて高いことを示唆します。
*   **永続化の確立試行**: ログイン後の活動として、SSH公開鍵を`authorized_keys`に書き込む試みは、将来的な検出回避と永続的なアクセス経路の確保を狙うものであり、深刻な脅威となります。
*   **システム改変の意図**: `chattr`コマンドの利用は、システムファイルの保護属性を変更し、バックドアの設置や痕跡の隠蔽、あるいはマルウェアの配置を容易にするための準備行動であると推測され、攻撃の深刻度を高めます。
*   **自動化された攻撃**: 短時間での連続的なログイン試行とコマンド実行は、自動化されたスクリプトやボットネットによる攻撃を示唆しており、広範な標的を狙っている可能性があります。

### 4. 推奨アクション

**緊急アクション**:

1.  **パスワード監査とリセット**: ログに記録されたログイン成功パスワード（`QAZwsx`, `3245gs5662d34`, `Root1234567@@`, `1q2w3e..`, `root123123!!`, `@a123456`, `Root11111111@#`）およびブルートフォースで試行されたパスワードが、社内のどのシステムにおいても使用されていないか緊急で確認してください。もし使用されている場合は、即座に該当アカウントのパスワードをより強力なものにリセットし、多要素認証(MFA)を強制してください。特に`root`アカウントや特権アカウントのパスワードは最優先で確認・強化が必要です。
2.  **SSH公開鍵の確認と削除**: 全てのSSHサーバーにおいて、全ユーザーの`~/.ssh/authorized_keys`ファイル内に、攻撃者が挿入しようとした公開鍵（`ssh-rsa AAAAB3NzaC1yc2EAAAA.... mdrfckr`）が登録されていないか確認してください。発見された場合は、直ちに削除し、ファイルとディレクトリのパーミッションを適切に設定し直してください（例: `chmod 700 ~/.ssh`, `chmod 600 ~/.ssh/authorized_keys`）。
3.  **攻撃元IPアドレスのブロック**: ファイアウォール、IDS/IPS、またはWAFにて、攻撃元IPアドレス `182.70.243.207` からのSSHポート(TCP/22)へのアクセスを即座にブロックしてください。ただし、これはボットネットの一部である可能性もあるため、状況に応じて地理的ブロックや評判ベースのIPブロックリストの適用も検討してください。

**中長期アクション**:

1.  **SSH設定の hardening (強化)**:
    *   **パスワード認証の無効化**: 可能であれば、パスワード認証を無効にし、公開鍵認証のみを許可する設定（`PasswordAuthentication no`）に移行してください。
    *   **多要素認証 (MFA) の導入**: SSH接続に多要素認証を義務付けてください。
    *   **rootログインの禁止**: SSH経由での`root`ユーザーによる直接ログインを禁止してください（`PermitRootLogin no`）。必要な場合は、一般ユーザーでログイン後、`sudo`コマンドを使用することを徹底させてください。
    *   **SSHポートの変更**: 標準の22番ポートから非標準のポートに変更することを検討してください（セキュリティバイオブスキュリティであり、根本的な解決ではないが、自動化されたスキャンからのヒット数を減らす効果はある）。
2.  **不正ログイン対策ツールの導入**: `Fail2ban`などのツールを導入し、繰り返しログインに失敗するIPアドレスを自動的にブロックする仕組みを構築してください。
3.  **アカウント管理の徹底**:
    *   不要なユーザーアカウントを定期的に棚卸し、削除してください。
    *   強力なパスワードポリシーを強制し、定期的なパスワード変更を推奨してください。
4.  **ログ監視の強化とアラート設定**:
    *   SSHログイン成功/失敗、特に`root`ユーザーへのログイン試行、または`authorized_keys`ファイルへの変更試行を監視し、異常を検知した際にアラートが上がるようにSIEMシステムやログ管理ツールを設定してください。
    *   `chattr`コマンドなどのファイル属性変更コマンドの実行も監視対象に含めてください。
5.  **定期的なセキュリティ脆弱性診断とパッチ適用**: OSやアプリケーションの脆弱性スキャンを定期的に実施し、常に最新のセキュリティパッチを適用して既知の脆弱性を解消してください。
6.  **セキュリティ意識向上トレーニング**: 組織内の全ユーザーに対し、強力なパスワードの重要性やフィッシング詐欺への注意喚起など、基本的なセキュリティ意識向上トレーニングを定期的に実施してください。
