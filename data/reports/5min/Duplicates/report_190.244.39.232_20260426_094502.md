# SOC自動分析レポート: 190.244.39.232

**生成日時:** 2026-04-26 09:47:47

## SOCアナリストレポート

### 1. 攻撃の概要と目的

攻撃元IPアドレス 190.244.39.232 から、SSHサービスを標的とした集中的なブルートフォース攻撃が確認されました。この攻撃の明確な目的は、システムの最高権限である `root` ユーザーアカウントの乗っ取りと、それに続く永続的なアクセス経路の確立です。

攻撃者は、約40分間にわたり断続的にSSHログインを試行しました。初期の試行では `odoo` などの一般的なユーザー名や、不明な文字列のユーザー名 (`345gs5662d34`) が使用されましたが、最終的には `root` ユーザーに対して異なる複数のパスワードで合計6回ものログイン成功を収めています。

ログイン成功後、攻撃者は即座に以下の悪意のあるコマンドを実行しています。
1.  既存の `.ssh` ディレクトリを削除し、再作成する。
2.  新しく作成した `.ssh` ディレクトリ内に `authorized_keys` ファイルを作成し、攻撃者自身のSSH公開鍵を書き込む。この公開鍵には `mdrfckr` というコメントが含まれています。
3.  `authorized_keys` ファイルおよび `.ssh` ディレクトリのパーミッションを適切に設定する。

この一連の操作により、攻撃者はパスワード認証なしで、自身の秘密鍵を使っていつでもシステムに再接続できる永続的なバックドアを確立しようとしています。また、`.ssh` ディレクトリの属性を `chattr` コマンドや `lockr` コマンド（このコマンドは失敗している）で変更し、自身の確立したアクセス経路が変更または削除されるのを防ごうとする試みも確認されました。

**攻撃のフェーズ (MITRE ATT&CK マッピング)**:
*   **初期アクセス (Initial Access)**: T1078 (Valid Accounts) - SSHサービスへのブルートフォース攻撃による正規アカウントの悪用。
*   **永続化 (Persistence)**: T1098 (Account Manipulation: SSH Authorized Keys) - `root` ユーザーの `authorized_keys` ファイルに公開鍵を追加し、将来的なアクセス経路を確保。
*   **防御回避 (Defense Evasion)**: T1070 (Indicator Removal: File Deletion, Command-Line Interface) - `.ssh` ディレクトリの削除・再作成。T1222 (File and Directory Permissions Modification) - `.ssh` ディレクトリの属性変更 (`chattr`, `lockr`) 試行。

### 2. 推測される手法・使用ツール

*   **自動化されたブルートフォース/辞書攻撃**:
    *   攻撃者は、`odoo`, `root`, `user0`, `shiv`, `oracle`, `ovpn`, `windows` など、様々なユーザー名と、広範なパスワードリストを組み合わせて試行しています。
    *   特に `root` ユーザーに対しては、`135792468`, `abcdefgh`, `Aa112211.`, `Gp123456`, `nPSpP4PBW0`, `3245gs5662d34` といった異なるパスワードでログインに成功しており、これは自動化されたパスワードクラッキングツール（例: Hydra, Medusa, Nmapのssh-bruteスクリプトなど）を使用している可能性が高いことを示唆します。
*   **SSH公開鍵バックドアの確立**:
    *   ログイン成功後、攻撃者は統一されたシェルコマンドを実行し、`~/.ssh/authorized_keys` に自身のSSH公開鍵を書き込んでいます。この手法は、システムへの永続的なアクセスを確立するための一般的な方法です。

### 3. 脅威レベルとその理由

**脅威レベル: 緊急 (CRITICAL)**

**理由**:
1.  **`root` ユーザーでの複数回のログイン成功**: システムの最高権限を持つ `root` ユーザーとして繰り返しログインに成功していることは、最も深刻な脅威です。これにより、攻撃者はシステムに対する完全な制御権を獲得し、データの窃取、改ざん、破壊、マルウェアの展開、他のシステムへの横展開、サービス停止など、あらゆる悪意のある活動を実行できる状態にあります。
2.  **永続的なアクセスの確立**: SSH公開鍵認証によるバックドアの設置は、攻撃者が一度侵入した後もパスワード変更などの対策を講じても、継続的にシステムにアクセスできる状態を作り出すため、システムのセキュリティに対する長期的な脅威となります。
3.  **攻撃の自動化と執拗さ**: 短時間で様々なユーザー名とパスワードを試行し、複数の異なる `root` パスワードでログイン成功を繰り返していることから、攻撃が自動化されたツールによって実行されており、非常に活発かつ執拗な攻撃者であることを示しています。複数の `root` パスワードが脆弱であったか、情報漏洩によって攻撃者に知られていた可能性があります。
4.  **明確な悪意のある行動**: ログイン後のコマンドは、情報収集や偵察ではなく、直接的にシステムへの不正アクセス経路を確立し、その保護を図る悪意に満ちた活動です。

### 4. 推奨アクション

このログはハニーポットの記録であると仮定した場合でも、実際のシステムで同様の攻撃が発生した場合に備えて、以下の緊急対応と予防策を速やかに実行する必要があります。

**緊急対応（侵害を前提とした措置）**:
1.  **緊急遮断**: 攻撃元IPアドレス 190.244.39.232 からのSSH接続を、ファイアウォールまたはIPS/IDSで即座にブロックしてください。
2.  **侵害システムのネットワーク隔離**: この攻撃を受けた（可能性のある）システムを、他のネットワークセグメントから物理的または論理的に隔離し、さらなる被害拡大を防いでください。
3.  **`root` パスワードのリセット**: ログイン成功に使用された全ての `root` ユーザーパスワード (`135792468`, `abcdefgh`, `Aa112211.`, `Gp123456`, `nPSpP4PBW0`, `3245gs5662d34`) を直ちに、強力かつユニークなものに変更してください。また、その他の重要なシステムアカウントのパスワードも変更を検討してください。
4.  **`authorized_keys` ファイルの確認と削除**: `/root/.ssh/authorized_keys` ファイルを確認し、攻撃者によって追加されたSSH公開鍵（コメント `mdrfckr` のもの）を削除してください。また、他のユーザーの `~/.ssh/authorized_keys` ファイルも同様に確認し、不正な鍵がないことを確かめてください。
5.  **システム全体の整合性チェックとマルウェアスキャン**: 侵害された可能性のあるシステム全体に対して、最新の定義ファイルで包括的なマルウェアスキャンを実行し、ルートキットやその他の悪意のあるソフトウェアがインストールされていないか確認してください。ファイル整合性監視ツール（例: AIDE, Tripwire）を使用している場合は、チェックを実行してください。
6.  **ログの詳細調査**: 攻撃者がログイン成功後に実行した可能性のある他の操作（データ窃取、追加ツールのダウンロード、他のホストへの接続試行など）を特定するため、システム認証ログ、履歴ログ、ネットワークログなどを詳細に調査してください。

**予防策と強化対策**:
1.  **SSHパスワード認証の無効化**: 可能な限り、SSHパスワード認証を無効にし、厳格な公開鍵認証のみを許可するように設定を変更してください。これはSSHブルートフォース攻撃に対する最も効果的な対策の一つです。
2.  **`root` ユーザーの直接ログイン禁止**: SSH経由での `root` ユーザーの直接ログインを禁止し、一般ユーザーでログイン後に `sudo` を使用する運用に切り替えてください。
3.  **多要素認証 (MFA) の導入**: SSHアクセスにMFAを導入し、認証セキュリティを強化してください。
4.  **強力なパスワードポリシーの徹底**: 全てのシステムアカウントに対し、十分な長さと複雑さを持つパスワードの使用を義務付け、定期的な変更を推奨してください。また、一般的なパスワードや辞書攻撃で推測されやすいパスワードの使用を禁止するポリシーを適用してください。
5.  **ブルートフォース対策ツールの導入**: Fail2Banなどのツールを導入し、一定回数以上のログイン失敗があったIPアドレスを自動的にブロックするように設定してください。
6.  **システムのパッチ適用**: SSHサーバーソフトウェアを含む全てのシステムソフトウェアを最新の状態に保ち、既知の脆弱性を悪用されるリスクを低減してください。
7.  **ネットワークセグメンテーションとアクセス制御**: SSHポート (デフォルト22) へのアクセスを、信頼できる特定のIPアドレス範囲に限定するネットワークベースのアクセス制御を導入してください。
8.  **ログ監視とアラートの強化**: SSHログイン試行、特に `root` ユーザーへの試行や、認証成功後の不審なコマンド実行を検知できるよう、ログ監視システムとアラート設定を強化してください。SIEMソリューションを活用し、異常なイベントをリアルタイムで検知・通知できるようにしてください。
9.  **SSH公開鍵の厳格な管理**: システムに存在するすべてのSSH公開鍵が正当なものであることを定期的に確認し、不要な鍵や不明な鍵は削除してください。
