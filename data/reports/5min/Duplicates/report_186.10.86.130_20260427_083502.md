### 1. 攻撃の概要と目的

提供されたCowrieハニーポットのログによると、攻撃元IPアドレス `186.10.86.130` から、SSHサービスに対する組織的なブルートフォース攻撃が複数回観測されました。攻撃者は主に `root` ユーザーの認証情報を推測しようとしており、複数のパスワードの組み合わせを試行しています。

この攻撃の主な目的は以下の通りと推測されます。

1.  **システムへの不正アクセス:** `root` ユーザーの認証情報を取得し、対象システムに管理者権限でログインすること。
2.  **永続的なアクセス経路の確立（Persistence）:** ログイン成功後、SSH公開鍵をユーザーの `.ssh/authorized_keys` ファイルに追加することで、将来的にパスワードなしでシステムにアクセスできるバックドアを設置すること。これにより、初期侵入が発覚・対処された後も、攻撃者がシステムへのアクセスを維持できるようになります。
3.  **システム制御の奪取:** `root` ユーザーでのアクセス権を得ることで、システムの完全な制御権を掌握し、マルウェアの展開、データ窃取、あるいはシステムの破壊活動などに繋がる可能性があります。

ログには複数のログイン成功が記録されており、攻撃者は異なるパスワードリストを試していることが分かります。ログイン成功後には、必ずと言ってよいほど `.ssh/authorized_keys` の改ざんを試みています。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**

*   **`root` ユーザーでのログイン成功:** 複数のパスワードで `root` ユーザーとしてログインに成功している記録があります。これは、もしこれが実システムであった場合、攻撃者が管理者権限を奪取したことを意味し、極めて深刻な事態です。
*   **永続化メカニズムの確立:** ログイン成功後に SSH 公開鍵を `authorized_keys` に追加しようとしていることは、攻撃者が一度侵入した後も継続的にアクセスを維持しようとする明確な意図を示しています。これにより、初期対応でパスワードを変更したとしても、攻撃者は鍵認証で再侵入できる可能性があります。
*   **自動化された広範な攻撃:** 攻撃が自動化されていることから、特定のターゲットを狙ったものではなく、インターネット上に公開されている脆弱な SSH サービスを無差別に探索・攻撃している可能性が高いです。このような攻撃は、他のシステムにも同様の脅威をもたらす可能性があります。
*   **繰り返される試行:** 時間を置いて複数の認証情報の組み合わせを試み続けており、執拗にシステムへのアクセスを試みていることが伺えます。

### 4. 推奨アクション

このログはハニーポットのものであるため、実際のシステム侵害は発生していませんが、もし同様の攻撃が実システムに対して行われたと仮定した場合の推奨アクションを以下に示します。

#### 緊急対応 (Immediate Actions):

1.  **対象システムの緊急遮断/隔離:** もし、このログが実システムのものであった場合、当該システムは既に侵害されている可能性が高いため、直ちにネットワークから隔離し、被害拡大を防止してください。
2.  **SSH サービスのアクセス制限強化:** 不正なログイン試行が継続しているため、インターネットからの SSH アクセスを一時的に遮断するか、信頼できる特定の IP アドレスからのアクセスのみに限定してください（IPホワイトリスト方式）。
3.  **不正な SSH 公開鍵の特定と削除:**
    *   `root` ユーザーのホームディレクトリ (`/root/.ssh/authorized_keys`)、および他のユーザーのホームディレクトリ配下 (`~/.ssh/authorized_keys`) を確認し、身に覚えのない SSH 公開鍵（特にコメントが `mdrfckr` のもの）が追加されていないか確認してください。
    *   発見された場合は、直ちにその公開鍵を `authorized_keys` ファイルから削除し、その鍵認証でのアクセスを無効化してください。
4.  **侵害の痕跡調査 (Incident Response - IR):**
    *   実際に侵害された可能性のあるシステムについて、より詳細なフォレンジック調査を実施してください。
    *   SSH 認証ログ、システムログ、ネットワークトラフィックログなどを分析し、攻撃者がログイン後にどのような活動を行ったか、他のシステムへの横展開がないかなどを調査してください。
    *   システム内に他のバックドア、マルウェア、設定変更などが残されていないか確認してください。
5.  **パスワードのリセット:** ログに記録された `root` ユーザーのログインに成功したパスワード（例: `p2puser@123`, `3245gs5662d34`, `g`, `root1234567`, `uway2026`, `c*o5^V:^[^^`, `cnp200@HW`, `root1.2026`）が、他のシステムやサービスで使い回されていないか確認し、直ちに全てのリセットを実施してください。

#### 恒久対策 (Long-term Measures):

1.  **パスワードポリシーの強化:**
    *   すべてのユーザーに対し、複雑性要件（大文字、小文字、数字、記号の組み合わせ）、最小長、有効期限などを定めた厳格なパスワードポリシーを適用してください。
    *   一般的な辞書ワードや推測されやすいパスワードの使用を禁止してください。
2.  **多要素認証 (MFA) の導入:**
    *   SSH ログインを含むすべての重要なシステムアクセスに多要素認証を必須化してください。
3.  **SSH 鍵認証の推奨とパスワード認証の無効化:**
    *   可能であれば、SSH パスワード認証を無効にし、よりセキュアな SSH 鍵認証のみを許可するように設定してください。その際、鍵にはパスフレーズを設定し、定期的に鍵をローテーションする運用を推奨します。
4.  **SSH アクセス制限の強化:**
    *   ファイアウォールまたは SSH デーモン (sshd_config) の設定により、SSH 接続を許可する送信元 IP アドレスを信頼できるものに限定してください（IPホワイトリスト）。
    *   デフォルトの SSH ポート (22番) を変更することも、単純なスキャン攻撃に対する効果は限定的ですが、ノイズを減らす対策としては有効です。
5.  **ブルートフォース対策ツールの導入:**
    *   Fail2ban などのツールを導入し、一定回数以上のログイン失敗を検知した場合に、その送信元 IP アドレスを自動的にブロックする仕組みを構築してください。
6.  **ログ監視とアラート体制の強化:**
    *   SSH 認証ログを含むシステムログを集中管理し、不審なログイン試行（特に `root` ユーザーに対するもの）、ログイン成功、設定変更などの異常をリアルタイムで検知し、アラートを発するシステムを導入してください。
7.  **外部公開サービスの監査:**
    *   インターネットに公開されているすべてのサービス（特に SSH）の必要性を定期的に見直し、不要なサービスは停止してください。また、公開する必要があるサービスについても、その設定が最新かつセキュアであることを確認してください。
8.  **最新のセキュリティパッチ適用:**
    *   OS や SSH デーモンを含むすべてのソフトウェアを最新の状態に保ち、既知の脆弱性が悪用されないようにしてください。

上記の推奨事項を実施することで、同様の攻撃に対する耐性を向上させることができます。
