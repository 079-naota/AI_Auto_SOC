## SOCアナリストレポート

### 1. 攻撃の概要と目的

攻撃元IP `102.219.126.124` から、SSHサービスに対する継続的な攻撃が観測されました。
攻撃は、まず不正なログイン試行（ブルートフォース攻撃）から開始され、複数の異なるユーザー名とパスワードの組み合わせが試されています。特に`root`ユーザーを標的とし、複数回にわたり認証に成功しています。
認証成功後、攻撃者は直ちに `~/.ssh/authorized_keys` ファイルに自身のSSH公開鍵を登録するコマンドを実行しています。これにより、パスワード認証を迂回し、将来的に永続的なアクセス経路を確立しようと試みています。これは典型的なバックドア設置の手口です。

### 2. 推測される手法・使用ツール

*   **攻撃手法:**
    *   **ブルートフォース攻撃:** 複数の異なるパスワードリストを用いて`root`アカウントを含む様々なユーザーアカウントに対するログイン試行が繰り返されており、自動化されたブルートフォースツールが使用されていると推測されます。
    *   **永続化 (Persistence):** 認証成功後、`rm -rf .ssh && mkdir .ssh && echo "..." >> .ssh/authorized_keys && chmod -R go= ~/.ssh` という一連のコマンドを実行しています。これは、既存のSSH設定を削除し、攻撃者自身のSSH公開鍵（コメント `mdrfckr`）を `authorized_keys` に追加することで、パスワードなしでシステムへ再接続するためのバックドアを設置する手法です。
    *   **防御回避 (Defense Evasion):** `chattr -ia .ssh` や `lockr -ia .ssh` といったコマンドを試行していることから、`.ssh`ディレクトリに対する不変属性（immutable attribute）の解除や、何らかのロック機構の無効化を試み、永続化の実行を確実なものにしようとする意図が見られます。
*   **使用ツール:**
    *   **自動化されたSSHブルートフォースツール:** `Hydra`、`Medusa`、またはカスタムスクリプトなどが考えられます。
    *   **標準SSHクライアント:** 認証成功後のコマンド実行や、その後の再接続試行から、標準的なSSHクライアントが使用されていると推測されます。

### 3. 脅威レベルとその理由

*   **脅威レベル: 高 (High)**
*   **理由:**
    1.  **特権ユーザーへの認証成功:** 攻撃者は`root`ユーザーで複数回認証に成功しており、これは実環境であればシステムに対する完全な制御権を獲得したことを意味します。成功したパスワードには `1qaz@WSX3edc$RFV!@`, `ali123456`, `Root8`, `Aa112211.`, `Config123`, `3245gs5662d34` など、ある程度の複雑性を持つものが含まれており、攻撃者が洗練されたパスワードリストや既存の漏洩情報を使用している可能性を示唆しています。
    2.  **明確な永続化の意図:** 認証成功直後にバックドアとしてSSH公開鍵を設置しようとする行動は、一時的なアクセスではなく、システムへの継続的かつステルスなアクセスを確保する明確な悪意を示しています。
    3.  **防御回避の試み:** `chattr`コマンドなどを用いて`.ssh`ディレクトリの保護を解除しようとする試みは、攻撃者がより強固な永続化を目指していることを示します。
    4.  **攻撃の継続性:** ログイン失敗後も繰り返し接続を試み、異なる認証情報で成功していることから、非常に執拗な攻撃であり、今後も同様の攻撃が続く可能性が高いです。

このログはハニーポットの記録であるため、実際のシステム侵害には至っていませんが、もしこれが実運用中のサーバーであれば、既にシステムは完全に掌握され、深刻な被害が発生している状況と判断されます。

### 4. 推奨アクション

この攻撃を受けて、以下の推奨アクションを直ちに実施してください。

1.  **攻撃元IPのブロック:**
    *   ファイアウォール（FW）や侵入防止システム（IPS）において、攻撃元IPアドレス `102.219.126.124` からのすべての通信を直ちにブロックリストに追加してください。
    *   当該IPアドレスの所属する地理的情報やASN（Autonomous System Number）を調査し、必要に応じてより広範なIPレンジをブロック対象に含めることを検討してください。
2.  **SSH認証情報のリセットと強化:**
    *   システム内のすべてのSSHユーザー、特に`root`ユーザーおよび一般的なユーザー（`drcom`, `ubuntu`, `user`など、今回のログで試行されたユーザー名）のパスワードを直ちに、より強力で複雑なものに変更してください。
    *   今回の攻撃で成功したパスワード（`1qaz@WSX3edc$RFV!@`, `ali123456`, `Root8`, `Aa112211.`, `Config123`, `3245gs5662d34`）が他のシステムやサービスで使い回されていないか緊急で確認し、該当する場合は直ちに変更してください。
3.  **SSH公開鍵認証設定の監査と無効化/削除:**
    *   すべてのSSHユーザーの `~/.ssh/authorized_keys` ファイルを監査し、攻撃者が登録しようとした公開鍵（コメント `mdrfckr`、鍵の内容 `ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`）が存在しないか確認し、もし存在すれば直ちに削除してください。
    *   不明な公開鍵や不要な公開鍵も削除し、適切な権限（`chmod 700 ~/.ssh`, `chmod 600 ~/.ssh/authorized_keys`）が設定されていることを確認してください。
4.  **SSHセキュリティ設定の強化:**
    *   **多要素認証（MFA）の導入:** 可能であれば、SSHログインにパスワードと合わせてMFAを必須としてください。
    *   **パスワード認証の無効化（推奨）:** 可能な限り、SSHサービスではパスワード認証を無効化し、より安全な公開鍵認証のみを許可する設定に変更することを強く推奨します。
    *   **rootユーザーの直接ログイン禁止:** `/etc/ssh/sshd_config` で `PermitRootLogin no` を設定し、`root`ユーザーの直接SSHログインを禁止してください。一般ユーザーでログイン後、`su` または `sudo` を使用して`root`権限を取得する運用に切り替えてください。
    *   **SSHポートの変更:** 標準の22番ポートから別のポートに変更することを検討してください（これは「セキュリティの深層防御」の一部であり、根本的な対策ではありませんが、自動化されたスキャンからのヒット数を減らす効果があります）。
5.  **ブルートフォース対策の導入/強化:**
    *   `fail2ban` や `denyhosts` といったツールを導入し、SSHへの不正なログイン試行が一定回数以上発生した場合に、自動的に攻撃元IPアドレスをブロックする設定を適用してください。
    *   SSHサービスのレート制限を設定し、短時間でのログイン試行回数を制限してください。
6.  **ログ監視の強化:**
    *   SSHのログイン試行ログ、認証成功/失敗ログ、実行されたコマンドログなどをリアルタイムで監視し、異常を早期に検知できるシステム（SIEMなど）を構築・強化してください。
    *   今回の攻撃ログで使用されたパスワードや公開鍵情報を脅威インテリジェンスとして登録し、今後の監視に活用してください。
7.  **システム脆弱性スキャンとパッチ適用:**
    *   システムに対して定期的な脆弱性スキャンを実施し、SSHサービスを含むOSやアプリケーションの既知の脆弱性を特定・修正してください。
    *   OSやソフトウェアのセキュリティパッチを常に最新の状態に保ってください。
8.  **フォレンジック調査（実環境の場合）:**
    *   もしこれが実環境であれば、認証に成功したアカウントの活動ログ、作成・変更されたファイル、ネットワーク通信履歴などを詳細に調査し、他の侵害（マルウェア感染、データ窃取など）がないか徹底的なフォレンジック調査を実施する必要があります。特に、`chattr`や`lockr`コマンドが実際に実行された場合のシステム監査ログを確認し、不正な変更が試みられていないか確認してください。
