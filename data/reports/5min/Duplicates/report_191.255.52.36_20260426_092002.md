## SOCアナリストレポート

### 1. 攻撃の概要と目的

攻撃元IPアドレス `191.255.52.36` から、当社のSSHサービスに対して一連のブルートフォース攻撃と不正なログイン試行が行われました。ログはCowrie（SSHハニーポット）上で記録されており、実際のシステム侵害は発生していませんが、攻撃者の意図と手法を明確に示しています。

攻撃者は、一般的なユーザー名（`ubuntu`、`wang`）や総当たり的なユーザー名（`345gs5662d34`）の試行に加えて、`root`ユーザーに対する複数のパスワード試行を繰り返しました。最終的に、複数の異なるパスワードで`root`ユーザーとしてログインに成功（ハニーポット上での成功）し、以下の明確な目的を持って一連のコマンドを実行しています。

1.  **永続性の確保 (Persistence):**
    *   `root`ユーザーのホームディレクトリ（`~/.ssh/`）に攻撃者のSSH公開鍵を登録し、パスワード認証を必要としないバックドアアクセス経路を確立しようとしました。これにより、初期のログインパスワードが変更されても、攻撃者は継続してシステムにアクセスできる状態を作り出すことを意図しています。
2.  **認証情報の変更と掌握 (Credential Access / Impact):**
    *   `echo "root:V7x7C26rMbb4"|chpasswd|bash` コマンドにより、`root`ユーザーのパスワードを攻撃者自身の設定した値に変更しようとしました。これは、システムへの完全な制御を掌握し、正規の管理者のアクセスを妨害する目的があります。
3.  **偵察活動 (Discovery):**
    *   `cat /proc/cpuinfo`、`free -m`、`crontab -l`、`w`、`uname -a`、`whoami`、`lscpu`、`top`などの多数のシステム情報取得コマンドを実行し、システムの構成、リソース使用状況、稼働中のプロセス、既存のcronジョブ、ログインユーザーなどを広範に調査しました。これは、後のマルウェア展開やさらなる悪用（例: コインマイナーのインストール、ボットネットへの追加、脆弱性探索、横展開）のための環境分析を目的としています。
4.  **競合排除 (Defense Evasion / Impact):**
    *   `/tmp/secure.sh`や`/tmp/auth.sh`といった不審なスクリプトの削除、関連プロセスの強制終了（`pkill -9 secure.sh; pkill -9 auth.sh; pkill -9 sleep;`）、および`/etc/hosts.deny`ファイルのクリアを試みました。これは、他の攻撃者によって既にシステムに設置されている可能性のあるマルウェア（特にコインマイナーなど、リソースを消費するもの）を排除し、自身のマルウェアや活動がシステムリソースを独占できるようにする目的があると推測されます。

### 2. 推測される手法・使用ツール

*   **ブルートフォース攻撃 / クレデンシャルスタッフィング:** 複数の異なるユーザー名とパスワードを短時間に試行していることから、HydraやNmapの`ssh-brute`スクリプトのような自動化されたブルートフォースツール、または盗難された認証情報リストを使用したクレデンシャルスタッフィングが行われたと推測されます。
*   **認証情報の操作 / 永続性の確保:** `~/.ssh/authorized_keys`への公開鍵追加、`chpasswd`コマンドによるパスワード変更は、SSHサービスにおける永続的なアクセス確保と認証情報操作の典型的な手法です。
*   **Linuxコマンドラインツール:** 偵察、競合排除、ファイル操作のために、`cd`, `rm`, `mkdir`, `echo`, `chattr` (失敗), `chpasswd`, `cat`, `grep`, `wc`, `head`, `awk`, `free`, `which`, `ls`, `crontab`, `w`, `uname`, `top`, `whoami`, `lscpu`, `pkill`など、標準的なLinuxコマンドが幅広く使用されています。
*   **シェルスクリプトの利用:** ログイン成功後のコマンド実行は非常に迅速かつパターン化されており、攻撃者が手動で操作するのではなく、自動化されたシェルスクリプト（またはツール）がこれらのコマンドを順次実行した可能性が高いです。特に`cd ~ && rm -rf .ssh && mkdir .ssh && echo "..." >> .ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~`のような一連のコマンドは、スクリプトによるものです。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**

1.  **Root権限でのログイン成功:** 攻撃者が`root`ユーザーとして複数回ログインに成功しているという事実は、もしこれが本番環境であればシステムへの完全な侵害に直結します。
2.  **永続性メカニズムの確立試行:** `authorized_keys`への公開鍵登録は、最も一般的なバックドア設置方法の一つであり、システムへの持続的なアクセスを可能にします。これは、検出後も攻撃者がアクセスを維持する高いリスクを意味します。
3.  **認証情報の変更試行:** `root`パスワードの変更は、正当なシステム管理者のアクセスを妨害し、システムを乗っ取ることを目的とした極めて悪質な行為です。
4.  **広範な偵察活動:** システムの内部情報を広範囲に収集する行動は、攻撃者が次の攻撃フェーズ（例：より高度なマルウェアの展開、脆弱性の悪用、横展開など）の準備をしていたことを強く示唆しており、単なるいたずらでは終わりません。
5.  **競合排除の明確な意図:** 他のマルウェアを削除しようとする行動は、攻撃者がシステムリソースを独占し、自身の悪意ある活動を確立しようとする強い意思を示しています。これは通常、コインマイナーやボットネットマルウェアなどのデプロイの前兆です。
6.  **自動化された攻撃:** 一連の行動が自動化されていることから、これは特定のターゲットを狙ったものではなく、インターネットに公開された脆弱なSSHサービスを探索し、広く侵害を試みる大規模なスキャン・攻撃キャンペーンの一部である可能性が高いです。これにより、今後も継続的な攻撃が予想されます。

### 4. 推奨アクション

このログはハニーポットの記録ですが、実際のシステムで同様の事態が発生した場合を想定し、以下の推奨アクションを提案します。

#### 4.1 緊急対応 (Immediate Response)

1.  **攻撃元IPの即時ブロック:** 攻撃元IPアドレス `191.255.52.36` からのすべての通信を、境界ファイアウォール、IDS/IPS、WAFなどで即座にブロックします。
2.  **SSH認証情報の変更と無効化:**
    *   ログに記録された全てのパスワード（`Hello@123`, `3245gs5662d34`, `123123Aa`, `odoo123`, `Aa111111!`, `V7x7C26rMbb4`など）が他のシステムで使用されていないか確認し、使用されている場合は即座に強力でユニークなパスワードに変更します。
    *   全ての`root`ユーザーおよび特権ユーザーのパスワードを、複雑性の高いものに強制的に変更します。
    *   `/etc/ssh/sshd_config`設定を確認し、`PermitRootLogin no`を設定して`root`ユーザーの直接SSHログインを禁止します。
3.  **バックドアの確認と削除:**
    *   全てのユーザーのホームディレクトリ（特に`/root/.ssh/`）の`.ssh`ディレクトリ配下にある`authorized_keys`ファイルを確認し、身に覚えのない公開鍵（特にログに記録された`ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`）が存在しないか確認し、発見した場合は削除します。
    *   `~/.ssh/`ディレクトリのパーミッションが`700`、`authorized_keys`ファイルが`600`であることを確認します。
4.  **システム侵害の痕跡調査:**
    *   **プロセス:** `ps aux`、`top`コマンドで不審なプロセスが実行されていないか確認します。特に高リソースを消費するプロセスや見慣れないプロセスに注意します。
    *   **ファイルシステム:** `/tmp`、`/var/tmp`、`/dev/shm`などの一時ディレクトリや、ユーザーのホームディレクトリ、`/usr/local/bin`などに不審なファイルやスクリプト（特にログに登場した`secure.sh`、`auth.sh`など）が存在しないか確認します。
    *   **ネットワーク接続:** `netstat -tulnp`コマンドで、不審なリスニングポートや確立された外部への接続がないか確認します。
    *   **cronジョブ:** `crontab -l`、`/etc/cron.*`、`/etc/crontab`をチェックし、攻撃者によって追加された可能性のある定期実行タスクがないか確認します。
    *   **システムログ:** `/var/log/auth.log`、`/var/log/secure`、`/var/log/syslog`などを詳細に分析し、不審なログイン、コマンド実行、ユーザー追加、権限変更などのログがないか確認します。
    *   **ホストベースのファイアウォール:** `/etc/hosts.deny`がクリアされていないか確認し、必要に応じて復元します。また、`iptables`や`firewalld`の設定に不正な変更がないか確認します。
5.  **イメージからの復旧検討:** もし本番環境で実際に侵害が確認された場合、既知のクリーンなバックアップからの復旧を最優先に検討します。

#### 4.2 長期的な対策 (Long-term Measures)

1.  **SSH認証の厳格化:**
    *   **公開鍵認証の強制:** パスワード認証を無効化し、公開鍵認証のみを許可するように設定します。
    *   **MFA (多要素認証) の導入:** 可能であれば、SSHログインに多要素認証（例: TOTP、FIDO2/U2F）を導入し、セキュリティを強化します。
2.  **パスワードポリシーの強化:**
    *   強力で複雑なパスワードを強制するポリシーを導入し、定期的なパスワード変更を義務付けます。
    *   よく使われるパスワードやデフォルトの認証情報の使用を禁止します。
3.  **アクセス制限の強化:**
    *   **IPアドレス制限:** SSHへのアクセス元IPアドレスを、信頼できるネットワーク（VPN、オフィスIPなど）に厳しく制限します。
    *   **ポート変更:** SSHのデフォルトポート22番を、標準的でない高位ポート（例: 49152～65535）に変更することで、自動化されたスキャンの対象になりにくくします（ただし、これはセキュリティ対策ではなく、単なるノイズリダクションです）。
4.  **侵入検知・防止システム (IDPS) の導入と設定:**
    *   `Fail2ban`や`CrowdSec`などのツールを導入し、短期間でのログイン失敗回数が多いIPアドレスを自動的にブロックするように設定します。
    *   IDS/IPSを導入し、SSHブルートフォースや異常なコマンド実行パターンを検知・ブロックできるようにします。
5.  **ログ監視とSIEMの導入:**
    *   SSHログイン、ログイン失敗、rootユーザーによる操作、重要なファイル変更（特に`.ssh/authorized_keys`、`/etc/passwd`など）を監視し、リアルタイムでアラートを発するSIEM（Security Information and Event Management）システムを導入します。
6.  **システムの定期的な監査とアップデート:**
    *   OS、SSHサーバソフトウェアを含む全てのシステムとアプリケーションを常に最新の状態に保ち、既知の脆弱性を修正します。
    *   定期的にセキュリティ設定のレビュー、脆弱性スキャン、ペネトレーションテストを実施し、潜在的な弱点を特定・改善します。
7.  **インシデントレスポンス計画の見直し:** 今回の攻撃で明らかになった知見を基に、インシデントレスポンス計画を更新し、SSH関連の侵害に対応するための手順と責任を明確化します。
