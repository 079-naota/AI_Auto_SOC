# SOC自動分析レポート: 116.99.174.17

**生成日時:** 2026-04-27 05:57:18

---

## SOC分析レポート

### 1. 攻撃の概要と目的

*   **攻撃元IPアドレス**: 116.99.174.17
*   **攻撃期間**: 2026年4月27日 02:13:34 UTC から 02:31:53 UTC (約18分間)

今回の事象は、攻撃元IPアドレス 116.99.174.17 から、弊社ハニーポット（Cowrie）のSSHサービスに対し、継続的なブルートフォース/辞書攻撃が行われたことを示しています。

攻撃者は複数の一般的なユーザー名（admin, user, root, ubnt, config, squid, support, guest, 1234, nikita, ftpuser）と、辞書攻撃に用いられるような単純なパスワードの組み合わせを用いて、繰り返しログインを試みました。特に`root`ユーザーは集中的に狙われ、以下の7回のログイン成功が記録されています。

1.  root / admin
2.  root / @
3.  root / root123
4.  root / 12345
5.  root / abcd1234
6.  root / ipscan
7.  root / alpine

これらのログイン成功後、攻撃者はSSHトンネリング（`direct-tcpip`リクエスト）を複数回試行しています。これは、ハニーポットを経由して、さらに別の外部または内部のシステムへの接続、マルウェアのダウンロード、あるいは偵察活動（ポートスキャンなど）を試みたものと推測されます。

攻撃の主な目的は、SSHサービスへの不正アクセスを確立し、初期侵入ポイントを確保することであると考えられます。その後、獲得したアクセス権を利用して、さらなる悪意のある活動（例: マルウェアの展開、データ窃取、コマンド＆コントロール（C2）通信、内部ネットワークへの横展開）を試みた可能性が高いです。

### 2. 推測される手法・使用ツール

*   **攻撃手法**:
    *   **SSHブルートフォース/辞書攻撃**: 短期間に様々なユーザー名とパスワードの組み合わせを繰り返し試行していることから、自動化されたブルートフォース攻撃または辞書攻撃が実行されたと判断されます。ターゲットとなるユーザー名として`root`や`admin`など、一般的なアカウント名が頻繁に狙われています。
    *   **SSHトンネリングの試行**: ログイン成功後に`direct-tcpip`リクエストが発生していることから、攻撃者はSSHのポートフォワーディング機能を利用して、他のサービスへの接続や、自身のC2サーバーとの通信チャネルを確立しようと試みたと考えられます。これは、初期アクセス後の内部偵察やマルウェア展開の一般的な手法です。

*   **使用ツール**:
    *   具体的なツール名はログから特定できませんが、このような自動化されたブルートフォース攻撃には、`Hydra`、`Medusa`、`Nmap`のSSH関連スクリプト（例: `ssh-brute`）、あるいはカスタムスクリプトなどが一般的に使用されます。
    *   `direct-tcpip`リクエストは、ログイン後に自動的に何らかのペイロード（スクリプト、バイナリなど）を実行し、外部への接続を試みる機能を持つマルウェア（例: Miraiボットネットの亜種）や、攻撃者が手動でポートフォワーディングを設定した場合に発生します。短時間で複数のログイン試行とそれに続く`direct-tcpip`リクエストが発生していることから、マルウェアによる自動化された活動である可能性が高いです。

### 3. 脅威レベルとその理由

*   **脅威レベル**: **高 (High)**

*   **理由**:
    *   **複数回のログイン成功**: ハニーポット環境ではありますが、`root`アカウントに対する7回ものログイン成功が確認されており、これは実際のシステムであれば極めて深刻な侵害に直結します。
    *   **`direct-tcpip`リクエストの実行**: ログイン成功後にSSHトンネリング（`direct-tcpip`）が複数回試みられていることは、攻撃者が単なるアクセス確立に留まらず、さらなるシステム侵害や内部ネットワークへの侵入を意図している明確な兆候です。これは、マルウェアのダウンロード、C2通信、内部ネットワークスキャンなど、より高度な攻撃段階への移行を示唆します。
    *   **継続的かつ自動化された攻撃**: 短期間に繰り返しログイン試行が行われ、特定のユーザー名と弱いパスワードを効率的に試す挙動から、自動化された攻撃スクリプトやボットネットの一部である可能性が高いです。このような攻撃は、脆弱なシステムを無差別に探し出し、大規模な侵害活動の一環として実行されます。
    *   **広範囲な影響の可能性**: 攻撃元IPアドレスが1つに限定されているものの、使用されたクレデンシャルのパターンから、特定の企業を狙ったものではなく、インターネット上に公開された脆弱なSSHサーバー全般をターゲットにした無差別攻撃であると考えられます。この攻撃パターンは、他の公開サービスにも同様の脅威をもたらす可能性があります。

### 4. 推奨アクション

#### 4.1. 緊急対応 (Immediate Actions)

*   **攻撃元IPのブロック**: 攻撃元IPアドレス `116.99.174.17` からのSSHアクセスを、ファイアウォール、ACL、IPS/IDSなどを用いて即座にブロックリストに追加してください。
*   **関連システムのログ調査**: 全ての公開されているSSHサーバーおよび関連システムの認証ログ（/var/log/auth.logなど）を緊急で確認し、同様のログイン試行や成功事例がないか調査してください。
*   **クレデンシャル調査**: ログに記録されたログイン成功パスワード（`admin`, `@`, `root123`, `12345`, `abcd1234`, `ipscan`, `alpine`）が、実運用中のシステムで現在も使用されていないか緊急で確認し、使用されている場合は即座に変更してください。

#### 4.2. 予防策 (Preventative Measures)

*   **SSHポートの変更**: デフォルトのSSHポート（22番）を非標準のポートに変更し、無差別なポートスキャンからの可視性を低減します。
*   **鍵認証の強制**: パスワード認証を無効化し、公開鍵認証のみを許可します。これにより、パスワードブルートフォース攻撃を根本的に排除できます。
*   **多要素認証 (MFA) の導入**: SSHへのアクセスに、鍵認証に加えてMFAの導入を検討し、セキュリティを強化します。
*   **強力なパスワードポリシーの適用**: パスワードの複雑性、長さ、有効期限に関する厳格なポリシーを強制し、デフォルトパスワードや容易に推測可能なパスワードの使用を禁止します。
*   **アクセス元IP制限**: 信頼できるIPアドレス範囲からのみSSHアクセスを許可するよう、ファイアウォールやセキュリティグループで制限を設定します。
*   **侵入検知/防御システム (IDS/IPS) の導入**: Fail2Banなどのツールを導入し、一定回数ログイン失敗を繰り返すIPアドレスを自動的にブロックするように設定します。
*   **最新のソフトウェア維持**: SSHデーモンを含む全てのシステムソフトウェアを常に最新の状態に保ち、既知の脆弱性を悪用されるリスクを低減します。
*   **不要なアカウントの削除**: デフォルトで存在するが使用されていないアカウント（例: guest, admin, testなど）は削除または無効化します。

#### 4.3. 検出・監視 (Detection & Monitoring)

*   **SSH認証ログの継続的な監視**: SSH認証ログに対して、ログイン失敗回数の閾値超過、特定のユーザー名（例: `root`, `admin`）への不正なログイン試行、または成功に対してアラートを発する監視ルールを設定します。
*   **SSHトンネリング活動の監視**: `direct-tcpip`のようなSSHトンネル要求や、不正なポートフォワーディング活動を検知するルールをSIEM/ログ管理システムに設定し、異常な通信を即座にアラート化します。
*   **ネットワークトラフィック監視**: SSHトラフィックの異常なパターン（例: 異常に長いセッション、通常とは異なる量のデータ転送、未知のポートへの接続試行）を監視し、潜在的なC2通信やデータ窃取を検知します。
*   **ハニーポットの活用**: 引き続きハニーポットを運用し、新たな攻撃手法やクレデンシャルを収集・分析することで、脅威インテリジェンスを向上させます。

#### 4.4. 事後対応 (Post-Incident Response)

*   **フォレンジック調査**: もし実際のシステムでログイン成功が確認された場合、徹底的なフォレンジック調査を実施し、侵入経路、侵害された範囲、実行されたコマンド、データの流出有無、および残留マルウェアの有無を詳細に特定します。
*   **クレデンシャルのローテーション**: 侵害された可能性のある全てのシステムにおけるアカウントのパスワードやSSH鍵をローテーションします。
*   **インシデントレポートの作成**: 調査結果に基づき、詳細なインシデントレポートを作成し、経営層および関係者へ報告します。
