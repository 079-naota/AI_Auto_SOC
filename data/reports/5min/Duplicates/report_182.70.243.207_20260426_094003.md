# SOC自動分析レポート: 182.70.243.207

## SOCアナリストレポート：SSHサービスに対する不正アクセス試行と永続化の試み

### 1. 攻撃の概要と目的

攻撃元IPアドレス `182.70.243.207` から、当組織のSSHサービスに対して広範なブルートフォース/辞書攻撃が実施されました。この攻撃者は、複数のユーザー名とパスワードの組み合わせを試行した後、最終的に`root`ユーザーでのログインに複数回成功しました。

ログイン成功後、攻撃者は以下の明確な目的を持って活動しています。
1.  **永続的なアクセス経路の確立:** 自身のSSH公開鍵を`root`ユーザーの`authorized_keys`ファイルに追加することで、将来的にパスワードなしでシステムに再接続できるバックドアを設置しようとしています。
2.  **システム制御の維持:** `.ssh`ディレクトリの属性を変更するコマンド（`chattr`）を試すことで、自身の設置した鍵が削除されるのを防ぐ、またはシステム上の既存の保護を解除しようとしていると推測されます。

この一連の行動は、単なる偵察ではなく、明確なシステム掌握と永続化を意図した悪意のある活動です。

### 2. 推測される手法・使用ツール

*   **手法:**
    *   **ブルートフォース/辞書攻撃:** `steam`, `tunnel`, `cloud`, `gitlab` などの一般的に使用されるユーザー名や、`root`ユーザーに対して複数の異なるパスワードを短時間に連続して試行していることから、自動化されたブルートフォースまたは辞書攻撃が実行されたと推測されます。
    *   **永続化 (Persistence) 機構の設置:** ログイン成功後、速やかに`~/.ssh/authorized_keys`ファイルに攻撃者の公開鍵を追加し、将来のパスワードレスSSHログインを可能にしようとしています。これはMITRE ATT&CKフレームワークの「永続化（T1098 - Account Manipulation: SSH Authorized Keys）」に該当します。
    *   **防御回避 (Defense Evasion):** `chattr -ia .ssh`コマンドは、Linuxのファイル属性を変更し、重要なファイルをImmutable（変更不可）またはAppend-only（追記のみ可）にする属性を解除しようとするものです。これにより、`authorized_keys`ファイルの改ざんを防ぐ保護が設定されていた場合に、それを無効化しようとしたと考えられます。`lockr`コマンドは標準的なLinuxコマンドではなく、ハニーポットでは失敗していますが、攻撃者が期待していた特定のツールやマルウェアの一部である可能性があります。
*   **使用ツール:**
    *   **SSHクライアント:** SSHプロトコルを介した接続に使用。
    *   **自動化されたスクリプト/ツール:** 複数のログイン試行と、ログイン後の特定のコマンド（`rm -rf .ssh && mkdir .ssh && echo "..." >> .ssh/authorized_keys && chmod -R go= ~/.ssh`）の一貫した実行パターンから、Hydra、Medusa、NmapのSSHスクリプトなどのパスワードクラッキングツール、またはカスタムスクリプトが使用されている可能性が高いです。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**
1.  **root権限でのログイン成功:** ログには`root`ユーザーでのログイン成功が複数回記録されており、これはシステムに対する最高の権限を持つアカウントが侵害される可能性があったことを示します。実際の環境であれば、これにより攻撃者はシステムの完全な制御権を掌握できたでしょう。
2.  **永続的なバックドアの設置試行:** 攻撃者は自身のSSH公開鍵を設置することで、システムへの永続的なアクセスを確保しようとしました。これにより、初期のログインクレデンシャルが変更されたとしても、攻撃者は依然としてシステムにアクセスできる状態を作り出すことを意図しています。
3.  **組織的な攻撃の兆候:** 連続的かつ体系的なログイン試行と、ログイン成功後の明確な手順（`.ssh`ディレクトリの操作と鍵の設置）は、単一の偶発的な試行ではなく、自動化されたツールや攻撃フレームワークを利用した組織的な攻撃の兆候を示しています。
4.  **ハニーポット環境での検知:** 今回のログはハニーポット (Cowrie) 上で収集されたものですが、もしこれが実際のプロダクション環境であった場合、上記の攻撃はシステムに深刻な侵害をもたらしていたと考えられます。

### 4. 推奨アクション

以下の対策を早急に実施し、類似の攻撃に対する防御を強化することを強く推奨します。

1.  **攻撃元IPアドレスのブロック:**
    *   攻撃元IPアドレス `182.70.243.207` からの全ての通信を、組織の境界ファイアウォールおよびホストベースのファイアウォール（iptables/firewalldなど）で即座にブロックしてください。
2.  **SSHサービス設定の強化:**
    *   **`root`ユーザーSSHログインの禁止:** `/etc/ssh/sshd_config` にて `PermitRootLogin no` を設定し、`sshd`サービスを再起動してください。
    *   **パスワード認証の無効化:** 可能であれば、パスワード認証を無効にし、公開鍵認証のみを許可する (`PasswordAuthentication no`、`PubkeyAuthentication yes`) 設定を検討してください。
    *   **多要素認証 (MFA) の導入:** SSHログインに多要素認証を導入し、セキュリティを強化してください。
    *   **SSHポートの変更:** 標準の22番ポートではなく、使用頻度の低い高位ポートに変更することを検討してください（ただし、これは攻撃の量を減らす効果はありますが、根本的なセキュリティ対策ではありません）。
3.  **アカウントおよびパスワードの監査と強化:**
    *   ログに記録されたすべての成功パスワード (`QAZwsx`, `3245gs5662d34`, `Root1234567@@`, `1q2w3e..`, `root123123!!`, `@a123456`, `Root11111111@#`) が、実システムで使用されていないことを確認してください。もし使用されている場合は、即座に複雑で推測されにくいパスワードに変更してください。
    *   `steam`, `tunnel`, `cloud`, `gitlab`, `345gs5662d34` など、ログに記録されたユーザー名がシステムに存在しないか確認してください。存在する場合は、パスワードの強化またはアカウントの削除を検討してください。
    *   すべてのユーザーアカウントに対して、強力なパスワードポリシーを適用し、定期的なパスワード変更を義務付けてください。
4.  **侵入検知/防御システム (IDS/IPS) の導入・強化:**
    *   Fail2Banのようなツールを導入し、SSHのログイン失敗回数が多いIPアドレスを自動的にブロックする設定を行ってください。
    *   SSHログイン成功後の予期せぬコマンド実行（特にファイルシステム操作やユーザー・グループ操作など）を検知・アラートするIDS/IPSルールを設定してください。
    *   重要な設定ファイル（例: `~/.ssh/authorized_keys`, `/etc/passwd`, `/etc/shadow`）への変更を監視するファイル整合性監視 (FIM) ツールを導入し、不正な変更を検知できるようにしてください。
5.  **システムログの継続的な監視:**
    *   SSH関連のログ（`/var/log/auth.log` や `journalctl` など）を継続的に監視し、不正なログイン試行、成功、不明なコマンド実行などの異常な活動がないか定期的にレビューする体制を確立してください。SIEM（Security Information and Event Management）システムへのログ連携を推奨します。
6.  **インシデントレスポンス計画の見直し:**
    *   このような侵害が発生した場合のインシデントレスポンス計画が適切に機能するかを確認し、必要に応じて訓練を実施してください。
