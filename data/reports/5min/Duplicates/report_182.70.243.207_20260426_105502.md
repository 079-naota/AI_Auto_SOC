# SOC自動分析レポート: 182.70.243.207

**生成日時:** 2026-04-26 10:56:44

## SOCアナリストレポート

### 1. 攻撃の概要と目的

攻撃元IPアドレス `182.70.243.207` から、ターゲットシステム（Cowrieホニーポット）のSSHサービスに対し、複数の段階にわたる攻撃が観測されました。

**攻撃の概要:**
1.  攻撃者はまず `steam` ユーザー名でパスワード認証を試行し、失敗しました。
2.  その後、`root` ユーザー名に対するブルートフォース/辞書攻撃を繰り返し実行し、複数回にわたって異なるパスワードでのログインに成功しています。
3.  ログイン成功後、攻撃者は `~/.ssh/authorized_keys` ファイルに自身の公開鍵を追加しようとしました。これにより、将来的にパスワードなしでSSH接続が可能となる永続的なアクセス経路を確立しようとしています。
4.  また、`chattr` や `lockr` コマンドを使用して `.ssh` ディレクトリの属性を変更しようとしていますが、`lockr` コマンドは失敗しています。
5.  永続化の試行後、攻撃者は `345gs5662d34` という不明なユーザー名とパスワードで接続を試みていますが、これは失敗しています。
6.  一連の攻撃は継続的に行われ、異なる `root` パスワードでのログイン成功と、同じSSH公開鍵の追加試行が複数回観測されました。

**攻撃の目的:**
攻撃者の主な目的は、`root` ユーザー権限でのターゲットシステムへの不正アクセスと、その後の永続的なバックドアの確立です。これにより、システムを完全に制御下に置き、データの窃取、マルウェアの配布、他のシステムへの横展開（ラテラルムーブメント）などを実行する基盤を構築しようとしていたと考えられます。

### 2. 推測される手法・使用ツール

*   **認証情報窃取 (Credential Access):**
    *   **ブルートフォース攻撃/辞書攻撃:** `steam` や `root` などの一般的なユーザー名に対して、既知のパスワードリストや生成されたパスワードを繰り返し試行し、有効な認証情報を特定しようとしました。ログには複数の異なる `root` パスワードでログインに成功した記録があります。
*   **永続化 (Persistence):**
    *   **SSH公開鍵認証の悪用:** `root` ユーザーとしてログイン後、自身のSSH公開鍵を `~/.ssh/authorized_keys` ファイルに追加しようとしました。これにより、パスワード認証を回避し、鍵認証のみで将来的にシステムへアクセス可能なバックドアを確立することを試みました。

        *   この公開鍵には `mdrfckr` というコメントが付与されており、これは特定の攻撃者グループやマルウェアに関連する特徴である可能性があります。
    *   **ファイル属性操作の試行:** `chattr -ia .ssh` や `lockr -ia .ssh` といったコマンドで `.ssh` ディレクトリの属性を変更しようとしました。これは、`.ssh` ディレクトリの内容が変更できないように保護されている場合に、その保護を解除して公開鍵を追加するための準備動作と考えられます（`lockr` は一般的なコマンドではなく、ホニーポット上で失敗しています）。
*   **使用ツール:**
    *   ログに特定のツール名は明示されていませんが、繰り返し行われるログイン試行と定型的なコマンド実行パターンから、HydraやMedusaのようなSSHパスワードクラッカーと、それに続くペイロード展開のための自動化されたスクリプトやツールが使用されている可能性が高いと推測されます。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**

1.  **`root` ユーザー権限でのログイン成功:** 攻撃者はシステム上で最も高い権限を持つ `root` ユーザーとして複数回ログインに成功しています。これは、実際のシステムであればシステムへの完全な制御を意味し、あらゆる悪意のある活動が可能となります。
2.  **永続化メカニズムの確立試行:** 攻撃者が自身の公開鍵を `authorized_keys` に追加しようとしたことは、バックドアを確立し、将来にわたって継続的にシステムにアクセスしようとする明確な意図を示しています。これはシステムのセキュリティ体制を根本から破壊する行為です。
3.  **複数の認証情報の窃取:** 攻撃者が複数の異なる `root` パスワードを特定していることは、その攻撃能力と試行の広範さを示唆しています。
4.  **自動化された攻撃:** ログのパターンから、攻撃は自動化されたスクリプトまたはツールによって実行されている可能性が高いです。これは、組織に対する広範なスキャンや攻撃キャンペーンの一部である可能性があり、今後も同様の攻撃が継続するリスクが高いことを意味します。
5.  **既知のIoC (`mdrfckr` 公開鍵):** 特定のコメント付きの公開鍵が使われていることから、この攻撃が既知のマルウェアキャンペーンや攻撃者グループと関連している可能性があります。

これらの活動は、実際のシステムであれば深刻な侵害（Compromise）に繋がるものであり、情報漏洩、システム破壊、他のシステムへの感染拡大など、重大な結果を引き起こす可能性が高いため、「高」と判断します。

### 4. 推奨アクション

このログはホニーポットで観測されたものですが、もしこれが本番環境のシステムであったと仮定し、以下の推奨アクションを実施すべきです。

#### 4.1. 緊急対応 (Immediate Actions)

1.  **攻撃元IPのブロック:** ファイアウォール、IDS/IPS、またはACLで、攻撃元IPアドレス `182.70.243.207` からのSSHポート（および他の公開サービスポート）への通信を即座にブロックする。
2.  **侵害された認証情報の無効化/変更:**
    *   ログに記録されたすべての成功した`root`パスワード (`QAZwsx`, `3245gs5662d34`, `Root1234567@@`, `1q2w3e..`, `root123123!!`, `@a123456`, `Root11111111@#`) を直ちに、より複雑で推測されにくいものに変更する。
    *   システムに存在する他のすべての特権アカウント（`admin`、サービスアカウントなど）のパスワードも念のため変更を推奨する。
3.  **SSH公開鍵の確認と削除:**
    *   すべてのサーバーの `root` ユーザーのホームディレクトリ（`~/.ssh/`）内の `authorized_keys` ファイルをレビューする。
    *   攻撃者が追加しようとした公開鍵（コメントが `mdrfckr` のもの）が存在しないか確認し、存在する場合は直ちに削除する。
    *   また、その他に身に覚えのない公開鍵がないかも確認し、必要に応じて削除する。
4.  **不審なアカウントの確認:** ログに登場する `345gs5662d34` など、攻撃者がログインを試みた不審なユーザーアカウントがシステム内に作成されていないか確認し、存在する場合は直ちにロックまたは削除する。
5.  **侵害調査 (Compromise Assessment):**
    *   **ログの収集と分析:** SSHログ、認証ログ、システムログ（`auth.log`, `secure`など）、アプリケーションログなどを広範囲に収集し、攻撃者が他のシステムにアクセスしたり、マルウェアをインストールしたり、データを窃取したり、特権昇格を行った痕跡がないか徹底的に調査する。
    *   **不審なアクティビティの特定:** 不審なプロセス、不審なネットワーク接続、異常なファイル作成/変更、不審なcronジョブやスケジュールされたタスクがないかを確認する。
    *   **マルウェアスキャン:** 信頼できるアンチウイルス/EDRソリューションを使用して、マルウェアやルートキットのスキャンを実行する。
    *   **システム整合性の確認:** 重要なシステムファイルや設定ファイルが改ざんされていないか、ファイルのハッシュ値などを比較して確認する。

#### 4.2. 中期対応 (Mid-term Actions)

1.  **SSH設定の強化:**
    *   **パスワード認証の無効化:** 可能であれば、SSHデーモンでパスワード認証を無効にし、公開鍵認証のみを強制する (`PasswordAuthentication no`)。
    *   **`root` ログインの禁止:** SSH経由での `root` ユーザーの直接ログインを禁止する (`PermitRootLogin no`)。代わりに一般ユーザーでログイン後、`sudo` を利用する運用にする。
    *   **ポート変更:** SSHポートを標準の22番以外に変更する (セキュリティバイアリティの一環として)。
    *   **レートリミット/ブルートフォース対策:** `Fail2ban` などのツールを導入し、不正なログイン試行回数を制限し、繰り返し試行するIPアドレスを自動的にブロックする。
2.  **パスワードポリシーの強化:** すべてのシステムで強力なパスワードポリシーを適用し、定期的なパスワード変更を義務付ける。多要素認証 (MFA) の導入を検討する。
3.  **監視体制の強化:**
    *   SSHログインの成功/失敗ログ、`authorized_keys` ファイルへの変更、`root` ユーザーの特権コマンド実行などをリアルタイムで監視し、異常を検知した際にアラートを生成する体制を確立する。
    *   SIEM (Security Information and Event Management) を活用し、ログの一元管理と相関分析を行う。

#### 4.3. 長期対応 (Long-term Actions)

1.  **脆弱性管理プログラムの確立:** 定期的な脆弱性スキャン、ペネトレーションテスト、パッチ適用管理を実施し、システムの脆弱性を継続的に低減する。
2.  **セキュリティ意識向上トレーニング:** 従業員に対し、強力なパスワードの使用、不審なメールやリンクの識別、ソーシャルエンジニアリングに対する注意喚起など、定期的なセキュリティ意識向上トレーニングを実施する。
3.  **脅威インテリジェンスの活用:** `mdrfckr` のような既知の攻撃者やマルウェアに関連する IoC を脅威インテリジェンスフィードに登録し、今後の攻撃検知に役立てる。
4.  **ホニーポットの運用継続:** ホニーポットの監視を継続し、新たな攻撃パターンや脅威情報を収集することで、防御戦略の改善に役立てる。
