# SOC自動分析レポート: 102.219.126.124

## SOC分析レポート

### 1. 攻撃の概要と目的

提供されたログは、IPアドレス `102.219.126.124` からのSSHサービスに対するブルートフォース攻撃、およびその後の認証成功後の侵入試行を示しています。この攻撃者は、複数の異なるパスワードを使用して `root` ユーザーの認証に繰り返し成功し、その後、自身のSSH公開鍵を被害システムの `~/.ssh/authorized_keys` に追加することで、永続的なアクセス経路を確立しようとしています。

攻撃の目的は、標的システムに対する `root` 権限での不正アクセスを獲得し、その後、パスワードなしでのSSH接続を可能にするバックドア（SSH公開鍵）を設置することで、長期的なシステム制御を確保することであると強く推測されます。

### 2. 推測される手法・使用ツール

*   **手法:**
    *   **SSHブルートフォース攻撃 / パスワードリスト攻撃:** ログには、複数の異なるユーザー名（`drcom`, `ubuntu`, `user` など）と、特に `root` ユーザーに対して多数の異なるパスワード（`drcom123`, `1qaz@WSX3edc$RFV!@`, `ali123456`, `Root8`, `Aa112211.`, `Config123`, `Aa123321` など、さらには `3245gs5662d34` が複数回利用されている）を試行している記録があります。これは、一般的な辞書攻撃やパスワードリストを用いた攻撃パターンに合致します。
    *   **永続化 (Persistence) の試み:** 認証成功後、攻撃者は `~/.ssh` ディレクトリの属性を変更しようとし（`chattr -ia .ssh; lockr -ia .ssh`）、その後、既存の `.ssh` ディレクトリを削除し、新たに作成した `.ssh/authorized_keys` ファイルに自身のSSH公開鍵（`ssh-rsa AAAAB3NzaC1yc2E... mdrfckr`）を書き込み、適切なパーミッション（`chmod -R go= ~/.ssh`）を設定しようとしています。これは、次回以降のアクセスにパスワードを必要とせず、公開鍵認証で容易に接続するための典型的な手法です。

*   **使用ツール:**
    *   **自動化されたSSHクライアント:** 多数のパスワード試行と、認証成功後にほぼ同一のコマンドを短い間隔で実行していることから、手動操作ではなく、自動化されたスクリプトやツール（例：Hydra, Medusaなどのブルートフォースツール、またはカスタムスクリプト）が使用されている可能性が高いです。
    *   **標準的なLinuxコマンド:** 認証成功後に実行しようとしたコマンドは、`cd`, `rm`, `mkdir`, `echo`, `chmod`, `chattr` など、Linuxシステムで一般的に利用されるコマンドです。`lockr` は標準的なLinuxコマンドではないため、ハニーポットがエミュレートしない未知のコマンドであるか、タイプミスである可能性があります。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**

*   **`root` ユーザーの認証成功:** ログには `root` ユーザーでの認証が複数回成功した記録があり、これはシステムに対する最高権限が奪取されたことを示唆します。実際のシステムで発生していれば、システム全体の完全な制御が可能となり、極めて重大なインシデントです。
*   **永続化の試み:** 認証成功後のコマンド実行により、攻撃者はSSH公開鍵認証を利用して恒久的なアクセス経路を確立しようとしています。これにより、一度パスワードが変更されても、攻撃者は継続してシステムにアクセスできる状態となり、検知と対応が困難になります。
*   **自動化された攻撃:** 攻撃が自動化されていることから、広範なネットワークスキャンと脆弱性探索の一環として行われている可能性があり、他のシステムも同様の攻撃に晒されている可能性があります。
*   **ハニーポットログであること:** 今回のログはCowrieハニーポットのものであるため、実際のシステムへの直接的な被害は発生していませんが、これらの試行が本番環境で成功した場合の潜在的な影響は非常に大きいです。

### 4. 推奨アクション

#### 4.1. 緊急対応 (Immediate Actions)

1.  **攻撃元IPのブロック:** 攻撃元IPアドレス `102.219.126.124` からの全てのインバウンド通信（特にSSHポート22）をファイアウォールまたはIPS/IDSで即座にブロックしてください。
2.  **SSHパスワードの変更:** ログに記録された、`root` ユーザーで認証成功した全てのパスワード（例: `1qaz@WSX3edc$RFV!@`, `3245gs5662d34`, `ali123456`, `Root8`, `Aa112211.`, `Config123`, `Aa123321`）について、本番環境でこれらのパスワードが使用されていないか確認し、もし使用されている場合は、直ちに強力でユニークなパスワードに変更してください。
3.  **不正なSSH鍵の確認と削除:** 全てのLinuxサーバーにおいて、`~/.ssh/authorized_keys` ファイルを緊急でレビューし、ログに示された公開鍵（`ssh-rsa AAAAB3NzaC1yc2E... mdrfckr`）またはその他の不正な公開鍵が追加されていないか確認し、発見した場合は直ちに削除してください。
4.  **`root` ユーザーのSSHログインの無効化:** `sshd_config` ファイルで `PermitRootLogin no` を設定し、`root` ユーザーがSSHで直接ログインできないように変更してください。代替として、一般ユーザーでログイン後、`su` または `sudo` を使用するようにします。

#### 4.2. フォレンジック調査 (Forensic Investigation)

1.  **ログの深掘り調査:**
    *   攻撃元IP `102.219.126.124` の詳細情報（Whois情報、過去の悪性活動レポートなど）を調査し、脅威インテリジェンス情報を収集します。
    *   本番環境の認証ログ（`/var/log/auth.log` や `/var/log/secure` など）を詳細に調査し、`102.219.126.124` からのSSH接続試行や、他のIPアドレスからの類似の不正ログイン試行がないかを確認します。
    *   Cowrieハニーポットの完全なログやセッション記録を収集し、より詳細な攻撃者の行動パターンや使用ツールを分析します。
2.  **システム侵害の確認:** 認証に成功したパスワードが本番環境で利用されていた場合、そのシステムが侵害された可能性が高いため、以下の調査を実施します。
    *   不審なプロセスやファイルがないか確認。
    *   ネットワーク接続履歴や通信内容を調査。
    *   システム設定の変更履歴を調査。

#### 4.3. 予防的措置 (Preventive Measures)

1.  **多要素認証 (MFA) の導入:** SSH接続にMFAを必須とすることで、パスワードが漏洩した場合でも不正ログインを防ぐことができます。
2.  **パスワードポリシーの強化:**
    *   パスワードの複雑性要件を厳格化し、定期的なパスワード変更を強制します。
    *   一般的なパスワードや辞書攻撃で推測されやすいパスワードの使用を禁止します。
3.  **SSH設定の強化:**
    *   SSHパスワード認証を無効化し、公開鍵認証のみを許可することを強く推奨します。
    *   `sshd_config` で `MaxAuthTries` を低く設定し、ログイン試行回数を制限します。
    *   `AllowUsers` や `AllowGroups` を利用し、SSHログイン可能なユーザーを制限します。
    *   SSHサービスのポートをデフォルトの22番から変更することも、攻撃スキャンを避ける上で有効な場合があります（ただし、根本的な解決策ではありません）。
4.  **侵入検知・防止システム (IDS/IPS) の導入と強化:**
    *   ブルートフォース攻撃を検知し、自動的に攻撃元IPをブロックするIDS/IPSソリューションまたはFail2banのようなツールを導入・設定します。
5.  **定期的なセキュリティ監査と脆弱性診断:** 定期的にシステムのセキュリティ設定を確認し、脆弱性がないか診断を実施します。
6.  **インシデントレスポンス計画の見直し:** 今回の攻撃パターンを踏まえ、SSHサービスに対する攻撃が発生した場合のインシデントレスポンス計画を見直し、対応手順を明確化します。
