### 1. 攻撃の概要と目的

攻撃元IPアドレス 152.32.238.146 から、当社のSSHサービスに対して約16分間にわたり、断続的な認証情報推測攻撃（ブルートフォース/辞書攻撃）が実施されました。この攻撃では、多数の一般的なユーザー名（例: `testuser`, `admin`, `oracle`, `root`など）と推測されやすいパスワードを組み合わせたログイン試行が繰り返されました。

ログは、攻撃者が`root`ユーザーとして複数回（少なくとも4回）ログインに成功したと記録しています。ログイン成功後、攻撃者はシステムの最高権限を奪取し、将来的なアクセスを確保するための永続化メカニズムを確立しようとしました。具体的には、自身のSSH公開鍵を`root`ユーザーの`.ssh/authorized_keys`ファイルに追加するコマンドを実行し、パスワードなしでの再接続経路を確保しようと試みました。

この攻撃の主な目的は、当社のシステムへの不正アクセスを確立し、そのアクセスを永続化させることで、さらなる悪意ある活動（マルウェアの展開、データの窃取、他のシステムへの横展開など）の足がかりを築くことであると推測されます。

### 2. 推測される手法・使用ツール

*   **認証情報推測攻撃 (Credential Guessing / Brute-force / Dictionary Attack):**
    短時間で異なるユーザー名とパスワードの組み合わせを多数試行していることから、自動化された認証情報推測ツール（ブルートフォースまたは辞書攻撃ツール）が使用された可能性が高いです。使用されたユーザー名やパスワードの傾向（`root`、`admin`、`Pa$$w0rd`など）も、一般的な辞書攻撃ツールでよく見られるものです。
*   **永続化 (Persistence):**
    ログイン成功後に実行されたコマンド`cd ~ && rm -rf .ssh && mkdir .ssh && echo "ssh-rsa ..." >> .ssh/authorized_keys && chmod -R go= ~/.ssh && cd ~`は、攻撃者のSSH公開鍵をターゲットシステムの`authorized_keys`に追加する典型的な手法です。これにより、攻撃者はパスワードを必要とせず、自身の秘密鍵を用いてSSH接続を再確立できるようになります。これは、MITRE ATT&CKフレームワークにおける「Persistence (T1098: Account Manipulation: SSH Authorized Keys)」に該当します。
*   **使用ツール (推定):**
    攻撃の自動性、試行されたログイン情報の多様性、およびログイン後の定型的なコマンド実行パターンから、Hydra、Medusa、Nmapの`ssh-brute`スクリプト、または同様の機能を持つカスタムスクリプトなどのSSHブルートフォースツールが使用されたと推測されます。

### 3. 脅威レベルとその理由

**脅威レベル: 高 (High)**

**理由:**
1.  **ログイン成功:** 攻撃ログに`root`ユーザーでのログイン成功が複数回記録されています。これは、現実のシステムであれば、攻撃者が最高権限でのアクセスを一時的にでも獲得したことを意味し、極めて深刻な事態です。
2.  **最高権限 (Root) での活動:** `root`ユーザーとしてログイン後、自身の公開鍵を`authorized_keys`に書き込むことで、永続的なアクセス経路を確立しようと試みています。これは、システムの完全な制御を奪取し、将来にわたってアクセスを維持する意図があることを明確に示しています。
3.  **永続化の試み:** `authorized_keys`への公開鍵追加は、パスワード変更などの対策を行っても、攻撃者が鍵認証を用いてシステムに再アクセスできる状態を作り出すため、非常に危険な行為です。
4.  **継続的な攻撃:** ログイン成功後も、異なるユーザー名とパスワードでのログイン試行が継続しており、攻撃者が他の脆弱なアカウントを探しているか、別の攻撃ツールが並行して動作している可能性があります。

### 4. 推奨アクション

#### 緊急対応 (Immediate Actions):

1.  **攻撃元IPのブロック:** 攻撃元IPアドレス `152.32.238.146` からのすべての通信を、エッジファイアウォール、IPS/IDS、またはホストベースのファイアウォールで即座にブロックしてください。
2.  **rootアカウントのパスワード変更:** 攻撃により認証情報が特定された可能性のあるすべてのシステムで、`root`アカウントおよび他の特権アカウントのパスワードを、複雑で推測困難なものに直ちに変更してください。
3.  **authorized_keysの確認と削除:** すべてのシステム（特にSSH公開サービスを提供するサーバー）において、`root`ユーザーおよび他のすべてのユーザーの`.ssh/authorized_keys`ファイルの内容を精査し、不正なSSH公開鍵（特にログに記載されている`ssh-rsa AAAAB3NzaC1yc2EAAAABJQAAAQEArDp4cun2lhr4KUhBGE7VvAcwdli2a8dbnrTOrbMz1+5O73fcBOx8NVbUT0bUanUV9tJ2/9p7+vD0EpZ3Tz/+0kX34uAx1RV/75GVOmNx+9EuWOnvNoaJe0QXxziIg9eLBHpgLMuakb5+BgTFB+rKJAw9u9FSTDengvS8hX1kNFS4Mjux0hJOK8rvcEmPecjdySYMb66nylAKGwCEE6WEQHmd1mUPgHwGQ0hWCwsQk13yCGPK5w6hYp5zYkFnvlC8hGmd4Ww+u97k6pfTGTUbJk14ujvcD9iUKQTTWYYjIIu5PmUux5bsZ0R4WFwdIe6+i6rBLAsPKgAySVKPRK+oRw== mdrfckr`）が追加されていないか確認し、発見した場合は直ちに削除してください。
4.  **SSHサービス設定の強化:**
    *   `PermitRootLogin no` を設定し、`root`ユーザーの直接ログインを禁止してください。
    *   `PasswordAuthentication no` を設定し、鍵認証のみに限定してください。
    *   `MaxAuthTries` を低めに設定し、試行回数を制限してください。
    *   `AllowUsers` や `AllowGroups` を用いて、SSHログインを許可するユーザーを厳格に制限してください。
    *   可能であれば、SSHポートを標準の22番以外に変更することを検討してください（これは完全な対策ではありませんが、自動化されたスキャンからの露出を減らします）。
5.  **フォレンジック調査:** もしこのログが実システムで発生したものであれば、侵害された可能性のある全てのシステムにおいて、詳細なフォレンジック調査を実施してください。目的は、他のバックドアの有無、マルウェアの展開、データ漏洩、横展開の形跡などを特定することです。

#### 再発防止策 (Preventive Measures):

1.  **強力なパスワードポリシーの導入:** すべてのシステムユーザーに対し、複雑性、長さ、定期的な変更を強制する強力なパスワードポリシーを適用してください。
2.  **アカウントロックアウトポリシーの導入:** 認証失敗回数を制限し、一定回数失敗したアカウントを一時的または永続的にロックアウトするポリシーを設定してください。
3.  **多要素認証 (MFA) の導入:** SSHを含む重要なサービスにおいて、MFAの導入を強く推奨します。
4.  **ログ監視とアラート体制の強化:** SSHログインイベント（成功および失敗）、特に特権アカウントへのログイン試行や、`.ssh`ディレクトリへの変更を検知した場合に、リアルタイムでアラートを生成し、SOCチームに通知する監視体制を強化してください。
5.  **脅威インテリジェンスの活用:** 攻撃元IPアドレスを脅威インテリジェンスフィードと照合し、既知の悪意あるエンティティやC2サーバーと関連がないか確認してください。
6.  **脆弱性管理プログラムの実施:** 定期的な脆弱性スキャンとパッチ適用を通じて、システムのセキュリティホールを最小限に抑えてください。
7.  **ハニーポットの活用:** Cowrieのようなハニーポットを引き続き運用し、攻撃者の手法や使用ツールに関する知見を蓄積してください。
